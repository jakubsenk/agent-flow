Sekce 2

Co je ceos-agents · architektura 12 min

Co řeknete

Začnu od základu, protože budu navazovat. Co tedy ceos-agents přesně je:

  • Claude Code plugin — běží v rámci platformy Anthropic Claude Code, kterou znáte.
  • Plugin distribuujeme přes marketplace, instalace je jeden příkaz.
  • Architektura je dvouvrstvá: skills jsou orchestrátory (vědí CO udělat), agenti jsou specialisté (vědí JAK).
  • 18 specializovaných agentů, 28 skills.

Markdown je strategická volba:

  • Žádný runtime kód — vše je markdown, který Claude Code interpretuje.
  • 100 % top-15 Claude Code pluginů používá markdown.
  • BMAD-METHOD (45 700 hvězd) v březnu 2026 dokonce odstranili YAML workflow engine ve verzi 6.1.0, package -91 % (z ~3 MB na ~270 kB).
  • Proč BMAD odešel od YAML: YAML workflow engine se ukázal jako režie navíc — uživatelé strávili víc času laděním YAML schémat než vlastní logikou agentů, LLM si s markdown poradí lépe (čte ho jako přirozený text, YAML musí parsovat strukturně), a runtime engine přidával dependency, která se rozbíjela při upgradech. BMAD si přiznal, že "YAML jako primární orchestrace = anti-pattern pro LLM-driven systémy" a vrátil se k čistému markdown s persona handoffs.
  • My jsme markdown nikdy neopustili — z přesně stejných důvodů, které BMAD musel objevit cestou.

Pipeline je sekvenční uvnitř ticketu, paralelní mezi tickety:

  • Uvnitř jednoho ticketu sekvenční: triage → analýza dopadu → fix ↔ review → test → publish. Agenti jsou na sebe závislí — paralelizace by přidala koordinační režii (Free-MAD: nezkoordinovaná paralelizace = 17.2× error amplification).
  • Mezi více tickety paralelní: /fix-bugs + worktrees umožňují řešit X ticketů zároveň v izolovaných git worktree (každý nezávislý). Konfigurovatelné Batch size per projekt.
  • Cursor 3.2 má naopak 8 paralelních agentů uvnitř jednoho úkolu (worktrees), Anthropic Team Lead má peer-to-peer messaging mezi agenty. Pro náš task type by to přidávalo komplexitu bez zisku.
  • Sekvence + 18 specializovaných agentů — víc než kdokoli na trhu. BMAD má 6, Devin 4, Cursor specializované módy.

Engineering zajímavosti · ceos-agents (méně zřejmé, ale důležité)

Sledovatelnost akceptačních kritérií (maps_to)

Architect při decomposition feature emituje maps_to: AC-{N}: "{text}" u každého subtasku. Acceptance-gate na konci pipeline ověří per-AC fulfillment s code + test evidence (FULFILLED / PARTIALLY / NOT ADDRESSED). Spec-reviewer má --verify mode pro audit implementace proti spec. Tržně unikátní — Copilot Coding Agent má jen partial.

Když si agent není jistý, zeptá se uživatele (NEEDS_CLARIFICATION)

Když fixer/triage narazí na nejednoznačnost v zadání (např. "issue říká 'oprav přihlášení', ale jsou tam 3 různé login flows — který z nich?"), neimprovizuje. Emituje fenced block s konkrétní otázkou, pipeline přejde do paused stavu. User odpoví přes /resume-ticket --clarification "...odpověď...", pipeline pokračuje s odpovědí jako kontextem.

Ochrana proti zneužití: max 3 clarifications per pipeline + per-iter cap (agent nemůže donekonečna zdržovat). Pause-timeout config (default 30 dní, min 1h, max 365d) — po vypršení se pipeline auto-aborte. Tracker-comment producer/receiver protokol s EXTERNAL INPUT marker wrap (proti prompt injection z odpovědí).

Historie pipeline jako kontext pro další běhy

Každý dokončený run zapíše metadata do .ceos-agents/pipeline-history.md (append-only, 50-run retention): issue_id, použití agenti, výsledek (success/blocked), block reason (sanitized), timestamps.

Není to ML learning — LLM se nemění. Je to context-aware připomenutí: fixer při Step 1 dostane posledních 5 runů jako součást kontextu, reviewer posledních 10. V praxi to vypadá: "v posledních 5 runech jsi v souboru auth.ts nedokázal opravit edge case X — pozor na něj" nebo "reviewer 3× po sobě blokoval kvůli chybějící null check". Agent to vidí před tím, než začne pracovat — zvyšuje to šanci, že se vyhne opakování stejných chyb v sérii podobných ticketů.

Sanitized: 14 credential patterns (Bearer, AWS AKIDs, Slack/GitHub tokens, JWT, SSH/PGP keys atd.) automaticky stripped. Žádné PII, žádné sekrety v historii.

Webhooky (notifikace ven) — ochrana proti rozbité URL

K čemu jsou webhooky: ceos-agents umí posílat HTTP notifikace o událostech (PR vytvořen, ticket zablokován, pipeline doběhl) na URL, kterou si operátor nakonfiguruje v Automation Config — typicky napojení na interní Slack, Teams nebo monitoring (Grafana, Datadog).

Problém, který řešíme: co když je ta URL pomalá nebo rozbitá? Bez ochrany by pipeline u každé události čekala na timeout (typicky 5-30 sekund) → fix by se prodloužil o minuty zbytečným čekáním.

Naše řešení (3 vrstvy):

  • Po 3 selháních za sebou se webhook pro zbytek pipeline runu vypne — žádné další požadavky tam neletí. Po dalším runu se začíná čistě.
  • Selhání webhook nikdy nezablokuje fix — pipeline si jen zaloguje [WARN] Webhook delivery failed a pokračuje. Notifikace je advisory (informativní), ne kritická závislost.
  • Bezpečnost: curl povoluje jen http:// nebo https:// — operátor nemůže (omylem ani záměrně) poslat na file://, gopher:// a další schémata, která by mohla zneužít interní zdroje (SSRF útok).

Proč filip-superpowers a forge 12 min

Co řeknete

Když ceos-agents začínalo, dělal jsem ho ad-hoc — psal jsem agenty rukou, testoval, opravoval, iteroval.

Brzo jsem narazil na strop kvality:

  • LLM přijde s něčím, co vypadá dobře.
  • Já to zkontroluju, něco opravím.
  • Tři týdny později zjistím, že verze 5.x má 41 testů, které jsou jen doc-string grepy, ne skutečné testy.
  • To je přesně to, před čím varuje Karpathy v eseji "Vibe Coding" — bez evaluace nemáte pipeline, máte jen iluzi.

filip-superpowers — odkud to vychází:

  • Osobní motivace: nesnáším se bavit s Claudem bez research. Bez něj mi vrací staré věci (knowledge cutoff několik měsíců zpátky), nebo věci špatně (halucinuje API, deprecated patterns, fake citace). Research-first pipeline je pro mě nepostradatelná disciplína, ne nice-to-have.
  • Můj soukromý plugin, fork komunitního superpowers Jesseho Vincenta (GitHub: obra/superpowers).
  • Vincent dnes vede #2 nejpoužívanější Claude Code framework — 121 tisíc hvězd.
  • Po lednu 2026, kdy ho Anthropic přijal do oficiálního marketplace, akcentoval: "don't trust LLM output without process".
  • Z toho vychází i moje filip-superpowers.

Forge — core orchestrace v té paměti:

  • Multi-fázová pipeline pro vývoj komplexních věcí.
  • 10 fází: research → brainstorm → spec → TDD → plan → execute → verify.
  • Pro kontext: tahle prezentace zrovna teď jede v jedné z těch fází (brainstorming).
  • v6.9.0 ceos-agents byla forged, v6.10.0 forged, v7.0.0 právě běží paralelně přes forge, v8.0.0 specs jsou hotové.
  • Nezapisujeme features, forge-ujeme je.

Jak forge tvoří agenty a dělá zadání

Tvoří agenty

  • Každá fáze má TEMPLATE prompt (markdown soubor) s placeholders
  • Substituce: {task}, {research_findings}, {spec_path}, {file_list}
  • Multi-agent dispatch: parent skill volá Task() N-krát s jiným zadáním
  • Heterogenita = záměrně různé úhly pohledu ("lenses"). Konkrétně 5 perspektiv:
    • academic — peer-review studie, arXiv
    • production — case studies firem, post-mortems
    • OSS code — čtení skutečných implementací na GitHubu
    • community — HN, Reddit, Discord diskuze
    • vendor docs — oficiální dokumentace Anthropic, OpenAI atd.
    Každý research agent dostane jinou perspektivu — vyhýbáme se group-think (viz Free-MAD: heterogenní paralelní agenti +12-16 % kvality vs uniformní).

Dělá zadání

  • Phase 0 routing analyzuje user vstup, vybírá fáze (bug-fix neskáče celé 10 fází)
  • Phase 1 LEAD agent rozloží task na ≤25 zkoumatelných otázek (decomposition)
  • Spec phase generuje EARS-format requirements (machine-checkable: WHEN/WHILE/IF)
  • Plan phase generuje task graph s blokádami a paralelismem
Jak forge měří kvalitu — commander-verdict skóre

Phase 8 verification spustí 5 paralelních verifikátorů, každý hodnotí jinou dimenzi (skóre 0-1):

  • security — vulnerabilities, prompt-injection, credentials handling
  • correctness — splňuje implementace EARS requirements?
  • spec-alignment — neodchýlil se kód od spec phase 4?
  • robustness — error handling, edge cases, retry logic
  • devils-advocate — co může selhat v produkci, na co jsme zapomněli

Šestý agent — COMMANDER VERDICT (opus) — agreguje 5 dimenzí do jednoho skóre + verdiktu (FULL_PASS / PARTIAL_PASS / FAIL).

Další engineering zajímavosti · forge

Mutační testování · brána 70 % zachycených mutací

Phase 5 TDD generuje testy ze spec, ne z kódu (testy nesmí vědět, jak je implementace napsaná). Pak forge automaticky mutuje produkční kód (změní operátory, hodnoty, podmínky) a měří, kolik mutací testy zachytí.

Pod 70 % kill rate = testy jsou slabé, zachytávají jen happy path → forge přidá generování dalších testů. AdverTest research: 70 % gate dává 8.56 % quality improvement vs no-gate.

Skrytá rezerva testů (20 %) jako slepý benchmark

Phase 5 vygeneruje sadu testů, ale 20 % drží jako "hidden reserve" — agent v phase 7 (execute) je nikdy neuvidí. Když phase 8 verifikuje, hidden testy fungují jako blind benchmark: kolik scénářů agent vyřešil bez toho, aby věděl, že existují?

Tohle je obrana proti "test-driven memorization" — agent by mohl optimalizovat pro viditelné testy a fail na unseen edge cases. Hidden reserve to ošetřuje.

Strojově ověřitelné požadavky ve formátu EARS

Phase 4 spec generuje requirements v EARS formátu (Easy Approach to Requirements Syntax) — WHEN {trigger} THE SYSTEM SHALL {response}, WHILE {state} ..., IF {condition} THEN ....

Strojově parsovatelné — phase 5 z toho generuje testy automaticky, phase 8 reviewer ověřuje per-REQ pokrytí. Žádné natural-language ambiguity ("user-friendly", "fast"). v6.9.0 spec: 90 EARS REQs / 118 ACs.

3 alternativní návrhy → soudce vybere (proti uniformnímu uvažování)

  • Conservative — minimální risk, proven patterns
  • Innovative — best-in-class, nejmodernější přístupy
  • Skeptical — co může selhat, devil's advocate

Pak JUDGE agent (opus) rozhodne mezi nimi (nebo synthesize). Free-MAD research: heterogenní agenti +12-16 % vs uniformní; nezkoordinovaná paralelizace 17.2× error amplification — proto judge layer.

Co řeknete o autonomii

Forge NENÍ "klikni a hotovo". Realita:

  • Počáteční zadání (1-3 odstavce) píšu já. Forge to nedělá za mě.

Co je "pivot" (krátké vysvětlení)
"Pivot = strategická změna směru na základě dat. Když mám původní vizi, zkoumám trh a výzkum, a oboje řekne 'tohle nikde neprodukuje produkční systémy', neignoruju to a trvám na své vizi — pivotuju. Změním směr."
Co je TOML (krátké vysvětlení)
"TOML je config formát, něco jako .ini, ale s vnořenými strukturami. Pravděpodobně ho znáte z pyproject.toml, Cargo.toml, next.config.toml. Pro nás to znamená: jednoduchý textový soubor, kde uživatel napíše projekt-specifická pravidla, která se mergnou nad generic agent prompt."
Co řeknete

Tento blok je důležitý, protože vám ukáže, jak se moje myšlení vyvíjelo na základě dat. Nebyl to lineární plán → execution. Byla to vize → research → pivot.

Původní vize, kterou jsem měl ještě před vznikem forge:

  • Meta-agent generuje všechny agenty per-project.
  • Tedy ne 18 generických agentů, ale 18 projekt-specific agentů, vyladěných podle konkrétního kódu, technologie a konvencí daného projektu.
  • Validace by probíhala nezávisle — tady jste poprvé viděl claude-grade.

Smyčka byla: meta-agent vygeneruje agenty per-project → claude-grade (audit nezávislé instance LLM) zhodnotí kvalitu (skóre A-F + Top 3 to fix) → meta-agent opraví → ship.

Krásný princip. Self-improving system, žádné generic kompromisy. Ale data ukázala něco jiného.
