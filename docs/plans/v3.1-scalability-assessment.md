## 2. Maintainability Assessment

### Current Scale
- 22 commands in pure markdown — manageable but approaching the upper comfort zone
- 13 agents — well within sustainable range (each is self-contained, ~3-4k bytes)
- Agent files are clean: consistent structure (Goal/Expertise/Process/Constraints), no cross-references
- Commands vary widely in complexity: from 429 bytes (create-pr) to 15,333 bytes (fix-bugs)

### Duplicity Analysis

The three pipeline commands (fix-bugs, fix-ticket, implement-feature) share significant logic:

| Shared Section | fix-bugs | fix-ticket | implement-feature | Status |
|----------------|----------|------------|-------------------|--------|
| Configuration parsing | ~16 lines | ~14 lines | ~16 lines | Near-identical |
| Pipeline profile parsing | ~20 lines | ~20 lines | ~15 lines | Near-identical |
| Decompose flag parsing | ~6 lines | ~6 lines | ~6 lines | Identical |
| Decomposition decision | ~26 lines | ~26 lines | ~26 lines | Near-identical |
| Subtask execution | ~23 lines | ~22 lines | ~20 lines | Near-identical |
| Hook execution (pre/post) | ~17 lines | ~17 lines | ~12 lines | Near-identical |
| Block handler | ~16 lines | ~14 lines | ~18 lines | Structurally identical |
| Fix verification | ~15 lines | ~21 lines | ~16 lines | Structurally identical |

**Estimated duplicated content:** ~270 lines across the 3 pipeline commands (out of 948 total lines combined). That is ~28% duplication.

### Adding New Components

- Effort to add a new command: copy existing pattern, customize orchestration steps (~15-30 minutes for simple commands, ~1-2 hours for pipeline commands)
- Effort to add a new agent: copy agent template, define Goal/Expertise/Process/Constraints (~15 minutes)
- Effort to change shared logic (e.g., block handler format, hook execution pattern): must update 3 pipeline command files manually

### Pain Points
- Shared logic (profile parsing, verification, hooks, block handler) is duplicated across pipeline commands
- No inheritance or composition between command files — each is a standalone document
- Config contract changes require updating CLAUDE.md + potentially all commands that read the affected config section
- The largest command (fix-bugs.md at 15,333 bytes / 380 lines) is reaching the limit of comfortable markdown readability

### Mitigations Available
- CLAUDE.md serves as single source of truth for conventions and config contract
- Test harness validates structural consistency across agents
- `/migrate-config` handles config evolution for consuming projects
- Agent files are well-isolated — changes to one agent do not cascade

## 3. Break-Even Analysis

### What Pure Markdown Handles Well
- **Agent definitions** (role, expertise, constraints) — natural fit; markdown is ideal for persona definitions
- **Simple command orchestration** (step-by-step instructions) — adequate for linear flows
- **Config contract documentation** — excellent; tables + prose are the right format
- **Example templates** — excellent; markdown is the native format for examples
- **Cross-references** (e.g., "see implement-feature step 5") — workable at current scale

### Where Markdown Starts to Strain
- **State management:** Pipeline state is implicit (no persistent tracking between runs). Resume depends on parsing git state and issue tracker comments.
- **Shared logic:** Duplicated sections across pipeline commands (profile parsing, verification, hooks, block handler). Changing the block handler format requires editing 3 files.
- **Conditional logic:** Complex branching (decompose/no-decompose + profile + dry-run) produces deeply nested instruction trees. fix-bugs.md has 4 conditional dimensions interacting.
- **Error coordination:** Cross-agent error handling is described but not enforced programmatically. An LLM could misinterpret the block handler instructions.
- **Validation:** Config contract validation (check-setup) is a 5,897-byte markdown file that essentially describes a program. This is at the boundary of what markdown should express.

### Features That Would Require Code Orchestration
- Real-time dashboard with live updates (requires runtime)
- Webhook-triggered pipeline execution (requires runtime)
- Cross-run state persistence and learning (requires runtime)
- Fleet orchestration of parallel pipeline runs (requires runtime)
- Metrics time-series storage and querying (requires runtime)

## 4. Recommendation

### Verdict: Stay on Pure Markdown for Plugin v3.x

The current scale (13 agents, 22 commands) is sustainable in pure markdown. The ~33k token total overhead is manageable within Claude's 200k context window, and the per-interaction overhead of ~3k-6k tokens (1.5%-3%) is negligible. Adding a few more commands/agents (up to ~30 commands, ~15 agents) would not fundamentally change this picture.

The primary risk is not token overhead but **maintainability**: the ~270 lines of duplicated logic across pipeline commands will become increasingly painful to keep in sync as complexity grows.

### When to Transition

Consider transitioning to hybrid (markdown + code orchestration) when:
1. **Command count exceeds ~30** — shared logic duplication becomes unmanageable (currently 22)
2. **Pipeline state needs persistence** — between-run learning, metrics history, cross-run correlation
3. **Real-time features needed** — live dashboard, webhook triggers, fleet management
4. **Multi-repo coordination** — cross-repository pipeline dependencies
5. **Shared logic changes become error-prone** — if a block handler change is missed in 1 of N commands

### Next Steps

1. **Short term (v3.x):** Continue with pure markdown. Use test harness to catch structural drift. Accept the ~270-line duplication as manageable technical debt.
2. **Medium term (v4.0):** Extract shared command logic into a "pipeline-base" template that pipeline commands reference. Consider a thin code layer for state persistence. This could reduce the 3x duplication to near zero.
3. **Long term (runtime):** Implement `claude-agents-runtime` as a separate project for features that fundamentally require code execution (webhooks, fleet, persistence, live dashboard).
