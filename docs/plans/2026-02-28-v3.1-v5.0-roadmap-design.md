## Motivace

CLAUDE-agents v3.0 pokrývá: bug-fix pipeline, feature pipeline, scaffold, decomposition engine, dashboard. Plugin je v ekosystému Claude Code pluginů unikátní — žádný jiný nástroj nekombinuje issue-tracker-to-PR pipeline se specializovanými agenty, rollback, hooks a zero dependencies.

Tento dokument definuje strategickou roadmapu pro převálcování konkurence a postupnou eskalaci od pure-markdown pluginu k runtime platformě.

### Kde CLAUDE-agents vede (a nikdo jiný to nemá v kombinaci)

1. Issue tracker jako kanonický vstup (YouTrack+, rozšiřitelné přes MCP)
2. 6-stage sequential pipeline (triage → analysis → fix → review → test → publish)
3. 12 specializovaných agentů s model selection per stage
4. Rollback agent jako first-class participant
5. Block comment protokol + `/resume-ticket` (checkpoint-based recovery)
6. 4 hook injection points + custom agents
7. Scaffold pipeline (stack-selector → scaffolder → validate → git init)
8. Decomposition s configurable fail/commit strategy
9. Pure markdown, zero dependencies, installable jako plugin
10. Generic Automation Config contract — 100% project-agnostic

## Strategický přístup: Platform Play

**Fáze 1 (v3.1–v3.3):** Vyčerpat potenciál pure-markdown pluginu — každá feature, která JDE udělat bez runtime.
**Fáze 2 (Runtime):** Companion projekt `claude-agents-runtime` (Node.js/Python) pro features, které vyžadují persistent state, web server, paralelizaci na OS úrovni.

### Architektonické pravidlo

Nový agent jen pokud má UNIKÁTNÍ ROLI, kterou existující agent nemůže převzít rozšířením promptu. Jinak: rozšířit prompt existujícího agenta.

### Versioning pravidlo (z CLAUDE.md)

| Level | Trigger |
|-------|---------|
| MAJOR | Breaking change v Automation Config — nový required key, přejmenovaná sekce |
| MINOR | Nová backward-compatible feature — nový optional key, nový command/agent |
| PATCH | Behavior fix bez contract change |

## v3.1 — "Intelligence Layer"

**Zaměření:** Parallel execution + analytics + cost awareness + historický kontext

### 3.1.1 — Parallel Batch Execution

**Co:** Worktree-based paralelní zpracování ticketů v `/fix-bugs`.

**Jak:**
- `/fix-bugs N` spustí N ticketů, každý ve vlastním git worktree
- Pipeline per ticket je nezávislá (triage → ... → publish)
- Orchestrace: command vytvoří worktrees, spustí Agent tool paralelně, sbírá výsledky
- Merge strategy: každý ticket = vlastní branch → vlastní PR (žádné cross-ticket dependencies)

**Config (existující sekce Worktrees):**
```markdown
### Worktrees (optional)
| Klíč | Hodnota |
|------|---------|
| Batch size | 3 |
| Base path | `.worktrees/` |
| Cleanup | auto |
```

**Aktuální stav:** `fix-bugs` command UŽ obsahuje worktree orchestrační logiku a Worktrees config sekci. Zbývá: ověřit reálné paralelní spuštění Agent tool, result collection, error coordination across worktrees, cleanup.

**Impact:** HIGH — 3-5x rychlejší batch processing.
**Effort:** LOW-MEDIUM — základ existuje, potřeba end-to-end ověření a edge case handling.

### 3.1.2 — Pipeline Metrics Report

**Co:** Command `/metrics` — sbírá data z issue trackeru + git, generuje markdown report.

**Metriky:**
- Success rate (issues fixed / issues attempted)
- Průměrný čas na fix (od start po merged PR)
- Breakdown by stage (kolik issues blocked v triage vs. fixer vs. test)
- Top blocked reasons (z `[CLAUDE-agents]` komentářů)
- Token cost estimate per run (z modelu a počtu agentů)

**Jak:**
- Nový command `/metrics` — query issue tracker přes MCP, parsuje `[CLAUDE-agents]` komentáře, čte git log
- Výstup: markdown report (`metrics-report.md`) nebo stdout
- Žádný nový agent — command přímo používá MCP tools

**Config rozšíření:**
```markdown
### Metrics (optional)
| Klíč | Hodnota |
|------|---------|
| Output | `./reports/metrics.md` |
| Period | last 30 days |
```

**Vztah k `/dashboard`:** Dashboard (v3.0) vizualizuje aktuální stav pipeline (issues, blocked, PR). Metrics analyzuje historická data (success rate, trendy, failure patterns). Data source se částečně překrývá (oba čtou issue tracker + `[CLAUDE-agents]` komentáře). Long-term: dashboard by měl zobrazovat metrics data jako sekci.

**Impact:** HIGH — první plugin s pipeline analytics. Žádný konkurent to nemá.
**Effort:** LOW — jen command, žádný agent.

### 3.1.4 — Agent Effectiveness Tracking

**Co:** Rozšíření `/metrics` o per-agent success rate a failure analysis.

**Metriky per agent:**
- Success rate per stage (kolikrát stage prošel vs. blokoval)
- Průměrný počet iterací fixer↔reviewer
- Top důvody selhání per agent (z `[CLAUDE-agents]` blocked komentářů)
- Failure pattern detection: "65% bloků v test-engineer stage → top důvod: chybějící test fixtures"

**Proč je to kritické:**
- V pure markdown pluginu jsou agent prompty CELÝ produkt. Kvalita promptů = kvalita výstupů.
- Bez měření effectiveness nemůžeš cíleně zlepšovat prompty.
- Failure analysis feeduje zpátky do prompt improvement — skutečný competitive advantage.

**Impact:** HIGH — základ pro continuous prompt improvement.
**Effort:** LOW — rozšíření `/metrics` output, žádný nový command.

### 3.1.5 — Fix Verification Stage

**Co:** Volitelný post-publish verifikační krok — ověření, že fix skutečně opravil bug.

**Jak:**
- Po merge PR + deploy zkontroluje, jestli se bug reprodukuje
- Implementace přes existující hook systém: `Post-publish` hook nebo `Post-publish agent`
- Alternativně: nový optional config klíč `Verify command` v Build & Test sekci
- Pokud verifikace selže → issue se znovu otevře s komentářem

**Proč je to důležité:**
- Aktuální pipeline: triage → ... → publish → ???
- Nikdo neověřuje, jestli fix SKUTEČNĚ opravil bug v produkci/staging
- Closed-loop verification je unikátní differentiator — žádný konkurent to nemá

**Config rozšíření:**
```markdown
### Build & Test
| Klíč | Hodnota |
|------|---------|
| Build | `dotnet build` |
| Test | `dotnet test` |
| Verify | `npm run verify-fix` |
```

**Impact:** HIGH — closed-loop feedback. Buduje trust v pipeline.
**Effort:** LOW — rozšíření existujícího hook mechanismu.

### 3.1.6 — Pipeline Profiles

**Co:** Konfigurovatelné profily pro různé typy úkolů.

**Proč:**
- Typo fix nepotřebuje triage + code analysis
- Critical production bug potřebuje full pipeline + E2E testy
- Refactoring nepotřebuje triage, ale potřebuje enhanced review
- One-size-fits-all pipeline je neefektivní

**Config rozšíření:**
```markdown
### Pipeline Profiles (optional)
| Profil | Skip stages | Extra stages |
|--------|-------------|-------------|
| fast | triage, code-analyst | — |
| strict | — | e2e-test-engineer |
| minimal | triage, code-analyst, test-engineer | — |
```

**Použití:**
- `/fix-ticket ISSUE-123 --profile fast`
- `/implement-feature ISSUE-456 --profile strict`

**Poznámka:** Profily se aplikují na oba pipeline typy (fix i feature). Stage-skip list může být odlišný per pipeline — např. "fast" pro feature pipeline skipuje spec-analyst (pokud je issue dobře popsaný), zatímco pro bug pipeline skipuje triage.

**Impact:** HIGH — flexibilita pro existující uživatele. Šetří čas a tokeny.
**Effort:** MEDIUM — rozšíření command parsing + conditional stage execution.

### 3.1.7 — Cost Estimation (Range-Based)

**Co:** Command `/estimate ISSUE-ID` — predikuje token usage PŘED spuštěním pipeline.

**Jak:**
- Přečte issue z trackeru → odhadne velikost (LOC affected, počet souborů)
- Na základě historických dat (z `/metrics`) nebo heuristiky spočítá expected tokens
- **Range estimate** (ne point estimate — přesnost point estimate je nerealistická):
  - Best case: 1 fixer iterace, žádná decomposition
  - Typical: 2-3 fixer iterace
  - Worst case: max iterations + decomposition
- Výstup: tabulka s odhadem tokenů a ceny per stage (best / typical / worst)

**Proč range místo point estimate:**
- Token usage závisí na: složitost issue, velikost codebase, počet fixer iterací (1-5), zda se spustí decomposition
- Point estimate s ±30% accuracy poškodí trust. Range je upřímný.

**Impact:** MEDIUM — unikátní feature. Pomáhá s budget planning.
**Effort:** LOW — jen command s heuristikou.

### 3.1.8 — Historical Context pro Code Analyst

**Co:** Rozšíření `code-analyst` agenta o sekci "Relevant Past Fixes".

**Jak:**
- Code analyst přečte posledních 10 commitů v dotčených souborech
- Parsuje `[CLAUDE-agents]` blocked komentáře pro stejný modul/area
- Předá context fixeru: "tento soubor měl 3 bugy v posledním měsíci, pattern: off-by-one errors v pagination"

**Proč NE nový agent:**
- Code analyst UŽ čte codebase a mapuje impact zone
- Přidání historického kontextu je rozšíření jeho expertízy, ne nová role
- Nový agent by přidal zbytečný context window overhead

**Impact:** MEDIUM — vyšší success rate fixeru díky kontextu z minulých fixů.
**Effort:** LOW — rozšíření promptu existujícího agenta.

### Continuous: Prompt Quality Iteration

Průběžný proces (ne jednorázová feature):
- Po každém release analyzovat per-agent effectiveness data z `/metrics`
- Identifikovat nejslabší agenty a cíleně zlepšovat prompty
- Failure analysis → prompt improvement → měření → repeat
- Toto je ongoing disciplína, ne roadmap item s deadlinem

**Mechanismus:** Měsíční review `/metrics` outputu. Manuální analýza per-agent failure patterns. Cílené prompt edity v `agents/*.md`. Měření efektu v dalším měsíci. Zodpovědnost: autor pluginu.

### 3.2.1 — Multi-Tracker Support (dokončení)

**Co:** End-to-end podpora GitHub Issues + Jira + Linear jako first-class trackery vedle YouTrack.

**Aktuální stav:** Config klíč `Type` v Issue Tracker sekci existuje od v1.6 a publisher agent ho podporuje. ALE: reálně bylo testováno jen s YouTrack přes MCP. Pro ostatní trackery chybí:
- End-to-end ověření celé pipeline (triage → ... → publish) s non-YouTrack trackerem
- `check-setup` validace per tracker type (správné MCP tool names, state transition syntax)
- MCP configuration guide per tracker (jaký MCP server, jaké env vars)
- Example `.mcp.json` pro každý tracker

**Co se změní:**
- Agents: žádná změna (čtou data z issue, nezáleží odkud přijdou)
- Commands: rozšíření `check-setup` o validaci per tracker type
- Docs: MCP configuration guide per tracker
- Skill routing: beze změny

**Config příklad (GitHub):**
```markdown
### Issue Tracker
| Klíč | Hodnota |
|------|---------|
| Type | github |
| Instance | `github.com` |
| Project | `owner/repo` |
| Bug query | `is:issue is:open label:bug` |
| State transitions | In Progress: `add label:in-progress`, Blocked: `add label:blocked` |
```

**Impact:** HIGH pro community adoption — většina potenciálních uživatelů je na GitHub.
**Effort:** MEDIUM — hlavně dokumentace, check-setup validace, end-to-end testování. Agenti se nemění.

## v3.3 — "Autonomous Developer"

**Zaměření:** AI prioritizace + config management + template library

**Poznámka k verzi:** Žádná feature v této fázi nezavádí breaking change v Automation Config (žádný nový required key). Proto MINOR verze v3.3, ne v4.0. MAJOR verze bude vyhrazena pro budoucí breaking changes.

### 3.3.1 — AI Backlog Prioritization

**Co:** Command `/prioritize` — analyzuje backlog a navrhne pořadí fixů.

**Jak:**
- Přečte otevřené issues z trackeru
- Pro každý issue odhadne: impact (kolik uživatelů/modulů ovlivňuje), risk (jak kritická oblast kódu), effort (velikost change), dependencies (co musí být fixnuto první)
- Výstup: ranked list s doporučeným pořadím a zdůvodněním
- Nový agent: `priority-engine` (opus) — potřebuje opus pro komplexní reasoning přes celý backlog

**Proč nový agent (výjimka z pravidla):**
- Žádný existující agent nedělá cross-issue analýzu. Všechny jsou per-issue.
- Priority engine potřebuje vidět CELÝ backlog najednou — jiná role než triage-analyst (který vidí jeden issue).

**Impact:** HIGH — unikátní feature. Žádný konkurent to v pluginu nemá.
**Effort:** MEDIUM — nový agent + command.

### 3.3.2 — Config Migration

**Co:** Command `/migrate-config` — upgraduje Automation Config z starší verze na novou.

**Jak:**
- Detekuje aktuální verzi configu (z přítomnosti/nepřítomnosti sekcí)
- Navrhne změny pro upgrade (nové optional sekce, přejmenované klíče)
- Uživatel schválí → command aplikuje změny

**Proč je to důležité:**
- Při upgrade z v3 na v4 — co se stane s user's Automation Config?
- Bez migration path = frustrovaní uživatelé = ztráta community trust

**Impact:** MEDIUM — insurance pro budoucí breaking changes.
**Effort:** LOW — command s pattern matching.

### 3.3.3 — Template Library Command

**Co:** Command `/template STACK` — vygeneruje Automation Config pro daný tech stack.

**Jak:**
- Databáze templates v `examples/configs/` (z v3.2)
- `/template nextjs` → vygeneruje kompletní Automation Config pro Next.js + GitHub
- `/template list` → seznam dostupných templates
- Rozšíření `/onboard` — nabídne template jako starting point

**Impact:** MEDIUM — snižuje setup time z 15 minut na 1 minutu.
**Effort:** LOW — command + templates (které už existují z v3.2).

### 3.3.4 — Plugin Composability

**Co:** Dokumentace + guardrails pro koexistenci s jinými pluginy.

**Jak:**
- Namespace prefix `CLAUDE-agents:` na všech commands (už máme)
- Dokumentace: "jak používat CLAUDE-agents vedle jiných pluginů"
- Conflict detection v `/check-setup` — varování pokud jiný plugin registruje stejné command names
- Doporučení pro custom agents: namespace konvence

**Impact:** LOW-MEDIUM — prevence problémů, ne nová funkce.
**Effort:** LOW.

### 3.3.5 — Pure Markdown Scalability Assessment

**Co:** Architektonické posouzení, zda pure markdown orchestrace ještě škáluje pro v4.0+ features.

**Proč:**
- 22 commands = 22 markdown souborů s komplexní flow control logikou v přirozeném jazyce
- Žádné sdílení logiky mezi commands (DRY princip nelze uplatnit)
- `/prioritize` potřebuje iterovat přes N issues, scorovat, sortovat — hodně logiky pro prompt
- Bug v command logic = úprava promptu, doufej že to funguje

**Posouzení by mělo odpovědět:**
- Je 22+ commands v pure markdown ještě udržitelné?
- Kde je break-even point pro přechod na hybrid (markdown prompty + code orchestrace)?
- Měl by runtime převzít orchestraci, zatímco markdown zůstane jen pro agent prompty?
- Jaký je context window overhead pluginu? (13 agentů + 22 commands + routing skill = kolik tokenů se loaduje do Claude Code session?) Je to praktický scaling constraint?

**Impact:** Strategické rozhodnutí pro long-term architekuru.
**Effort:** LOW — research, žádná implementace.

## Runtime Platform (Companion Project: `claude-agents-runtime`)

**Zaměření:** Companion runtime pro features, které pure markdown nezvládne.

**DŮLEŽITÉ:** Runtime NENÍ verze pluginu. Je to samostatný projekt (`claude-agents-runtime`) s vlastním repem a semver od v1.0. Volat ho "v5.0" v kontextu pluginu je jen pozice v roadmapě, ne verze pluginu. Plugin zůstává na svém semver (v3.x / v4.x).

**Architektura:**
```
┌─────────────────────────────────────────┐
│  CLAUDE-agents plugin (pure markdown)    │
│  Commands + Agents + Skills              │
│  = jádro pipeline orchestrace            │
└──────────────┬──────────────────────────┘
               │ events (JSONL)
               │
┌──────────────▼──────────────────────────┐
│  claude-agents-runtime (Node.js/Python)  │
│                                          │
│  ├─ Live Dashboard (SSE/WebSocket)       │
│  ├─ Webhook Server (příjem triggerů)     │
│  ├─ Fleet Orchestrator (parallel VMs)    │
│  ├─ Web Approval UI (diff viewer)        │
│  ├─ Persistent Store (SQLite)            │
│  ├─ CLI Companion (claude-agents run)    │
│  └─ Multi-Repo Coordinator              │
└─────────────────────────────────────────┘
```

**Klíčový princip:** Pure markdown plugin zůstává jádrem. Runtime je OPTIONAL companion. Plugin funguje i bez runtime — runtime jen přidává capabilities.

### R.1 — Event Protocol

**Co:** Commands emitují strukturované eventy (JSONL) pro konzumaci runtime.

**Formát:**
```json
{"timestamp": "2026-03-15T10:30:00Z", "event": "stage_start", "pipeline": "fix-ticket", "issue": "PROJECT-123", "agent": "triage-analyst", "model": "sonnet"}
{"timestamp": "2026-03-15T10:30:45Z", "event": "stage_complete", "pipeline": "fix-ticket", "issue": "PROJECT-123", "agent": "triage-analyst", "result": "pass", "tokens": 2340}
{"timestamp": "2026-03-15T10:30:46Z", "event": "stage_start", "pipeline": "fix-ticket", "issue": "PROJECT-123", "agent": "code-analyst", "model": "sonnet"}
```

**Dopad na plugin:** Commands zapisují events do souboru/stdout. Minimální change.

### R.2 — Live Dashboard

- Node.js server čte JSONL events přes SSE
- Frontend: Gantt-style timeline s barevnými stavy per issue per stage
- Real-time progress bars, live logs

### R.3 — Webhook Server

- Express/Fastify server příjímá webhooky z issue trackerů
- Auto-trigger pipeline: issue created → spustí `/fix-ticket`
- Event-driven: push to branch → spustí `/scaffold-validate`

### R.4 — Fleet Orchestration

- Process manager + worktrees/Docker containers
- Stovky paralelních pipeline runů
- Auto-scaling: dynamický batch size podle zatížení a token budget
- VM/container isolation per pipeline run

### R.5 — Web Approval UI

- React/Svelte SPA s Monaco diff viewer
- Pipeline se zastaví na checkpoint, čeká na human approval v browseru
- Bidirectional WebSocket komunikace
- Komentáře k reviewer findings → fixer iteruje

### R.6 — Persistent Store

- SQLite databáze: historie všech pipeline runů
- Metrics time series: success rate, avg fix time, token cost — vše s trendy
- Learning: "tento typ bugu se historicky fixuje za 2 iterace v tomhle modulu"
- API pro query: `/api/metrics?period=30d&project=BIFITO`

### R.7 — Multi-Repo Coordinator

- Cross-repo dependencies: PR v repo A závisí na PR v repo B
- Coordinated releases: version bump v repo A + B + C současně
- Mono-repo awareness: affected packages detection

### R.8 — CLI Companion

```bash
# Spuštění pipeline z terminálu (bez Claude Code session)
claude-agents run fix-ticket PROJECT-123

# Status běžících pipeline
claude-agents status

# Metriky
claude-agents metrics --period 30d

# Fleet mode
claude-agents fleet fix-bugs --count 20 --parallel 5
```

### Runtime Summary

**PROVISIONAL:** Všechny detaily v této sekci (architektura, JSONL formát, sub-features) jsou aspirační vize. Finální design závisí na výsledku 3.3.5 Scalability Assessment, který posoudí, zda a jak přejít na runtime.

**Technologie:** Node.js (TypeScript) nebo Python — rozhodne se v design phase po 3.3.5.
**Distribuce:** npm package / pip package + Docker image.
**Vztah k pluginu:** Plugin je source of truth pro pipeline logic. Runtime je execution layer.
