## 1. Vize & cíl

### Co v3.0 přináší

CLAUDE-agents v2.0 pokrývá kompletní bug-fix pipeline. v3.0 rozšiřuje plugin na **univerzální vývojový nástroj** — od vytvoření projektu přes implementaci features až po bug fixy:

```
SCAFFOLD (nový projekt od nuly)
    → FEATURE PIPELINE (implementace features)
        → BUG-FIX PIPELINE (oprava bugů)
            → FEATURE PIPELINE (nové features)
                → ...
```

Jeden plugin, jeden lifecycle, jeden install.

### 4 propojené features

| Feature | Účel | Nové agenty | Nové commands |
|---------|------|------------|---------------|
| **Feature Pipeline** | Automatizace implementace features | spec-analyst, architect | implement-feature |
| **Scaffold** | Vytvoření projektu od nuly | stack-selector, scaffolder | scaffold, scaffold-add, scaffold-validate |
| **Decomposition** | Rozklad složitých úkolů na subtasky | (žádné — reuse architect) | (rozšíření fix-ticket, implement-feature) |
| **Dashboard** | HTML přehled stavu pipeline pro tým | (žádné) | dashboard |

### Proč jeden release

1. **Scaffold potřebuje Feature Pipeline** — scaffold vytvoří skeleton, Feature Pipeline implementuje features.
2. **Decomposition vylepšuje oba pipelines** — bug-fix i feature pipeline benefitují z rozkladu na subtasky.
3. **Dashboard zobrazuje všechny pipelines** — unified view přes bug-fix, feature i scaffold.
4. **Sdílený Config Contract** — Feature Workflow sekce v Automation Config je shared dependency.

|-----------|------|-----------|-------------|

## 2. Feature Pipeline

### Proč

Bug-fixy jsou jen zlomek vývojářské práce — většina času jde do implementace nových features. Feature pipeline znovupoužívá 60%+ existující infrastruktury (fixer, reviewer, test-engineer, publisher, rollback-agent) a přidává pouze vstupní analýzu.

### Paralelismus s bug-fix pipeline

| Bug-fix pipeline | Feature pipeline | Sdílené |
|------------------|------------------|---------|
| Bug report → triage | Feature request → specifikace | Analogický krok |
| Root cause analýza | Architektonický návrh | Analogický krok |
| Oprava (patch) | Implementace (nový kód) | **Stejný agent (fixer)** |
| Code review | Code review | **Identický (reviewer)** |
| Testy | Testy | **Identický (test-engineer)** |
| Publish | Publish | **Identický (publisher)** |

### Nové agenty

#### spec-analyst (sonnet)

Analyzuje feature request a extrahuje strukturovanou specifikaci.

```yaml
name: spec-analyst
description: Analyzes feature requests and extracts structured specifications with acceptance criteria
model: sonnet
```

- **Read-only** — nikdy nemodifikuje kód
- Vstup: feature request z issue trackeru
- Výstup: strukturovaná specifikace (popis, acceptance criteria, scope, omezení)
- Hybridní interakce: pokud request je vague, doptá se (max 3 otázky) — pokud je jasný, jde rovnou do specifikace
- Analogický k triage-analyst (ale pro features místo bugů)

#### architect (opus)

Navrhuje architekturu a generuje task tree pro implementaci.

```yaml
name: architect
description: Designs architecture and generates task trees for feature implementation and complex bug decomposition
model: opus
```

- **Read-only** — nikdy nemodifikuje kód
- Vstup: specifikace od spec-analyst (feature) nebo impact report od code-analyst (bug)
- Výstup: architektonický návrh + task tree (YAML format pro decomposition engine)
- Identifikuje: affected files, estimated diff lines, dependency order, acceptance criteria per subtask
- Opus model protože architektonická rozhodnutí vyžadují hloubkovou analýzu

### Command: `/implement-feature`

```
/CLAUDE-agents:implement-feature PROJ-100 [--decompose|--no-decompose] [--dry-run]
```

**Flow:**
```
1. Načti Automation Config
2. Nastav issue tracker state (On start set)
3. Vytvoř branch (Branch naming pattern)
4. spec-analyst — extrakce requirements, acceptance criteria
5. architect — design + task tree
6. DECOMPOSITION DECISION (auto-detect nebo flag)
   - Pokud decompose: task tree → subtasky → každý přes pipeline
   - Pokud single-pass: rovnou fixer → reviewer → test
7. [Pre-fix hook]
8. fixer → reviewer loop (max Fixer iterations)
9. [Post-fix hook + custom agent]
10. test-engineer (max Test attempts)
11. [Pre-publish hook + custom agent]
12. publisher → PR
13. [Post-publish hook + webhook]
```

### Config Contract rozšíření

Nová **volitelná** sekce v Automation Config:

```markdown
### Feature Workflow

| Key | Value |
|-----|-------|
| Feature query | state: Open type: Feature |
| On start set | In Progress |
```

Pokud sekce chybí, `/implement-feature` použije ruční zadání issue ID (bez automatického query).

## 3. Scaffold

### Design rozhodnutí

| Rozhodnutí | Volba | Důvod |
|------------|-------|-------|
| Separátní plugin? | **Ne — součást CLAUDE-agents** | Sdílený Config Contract, stejný lifecycle, zero runtime cost (markdown files), jeden install |
| Template engine? | **Ne — pure LLM generace** | Zero maintenance, libovolný stack, vždy aktuální, validator je quality gate |
| Kolik agentů? | **2 (stack-selector, scaffolder)** | Minimální set s jasnou separation of concerns |
| Interaktivita? | **Hybridní** | Pokud user dal dost info → one-shot. Pokud ne → stack-selector se doptá. |

### Pure LLM generace — jak to funguje

Scaffold nepoužívá templates. Scaffolder agent generuje projekt z training knowledge LLM modelu. Kvalitu zajišťuje validation loop:

```
scaffolder generuje skeleton → scaffold-validate ověří
     ↑                                    │
     └──── retry (max 3×) s chybou ───────┘
```

**Proč ne templates:**
- LLM zná aktuální best practices pro libovolný stack — Python, .NET, Angular, Go, Rust, cokoliv
- Templates stárnou (FastAPI 0.x → 1.0, Next.js app router vs pages router)
- Zero maintenance burden
- Validator (build + test + lint) je silnější quality gate než template correctness

### Nové agenty

#### stack-selector (sonnet)

Analyzuje scaffold zadání a rozhodne tech stack.

```yaml
name: stack-selector
description: Analyzes project requirements and selects optimal technology stack for scaffolding
model: sonnet
```

- **Read-only** — nikdy nemodifikuje kód
- Hybridní interakce: pokud user specifikoval stack → jde rovnou do rozhodnutí. Pokud ne → doptá se (max 3 otázky, přednostně multiple choice).
- Rozhoduje: jazyk + verze, framework, databáze, CI/CD provider, testing framework, linter
- Výstup: strukturované rozhodnutí s rationale

#### scaffolder (sonnet)

Generuje minimální projekt skeleton.

```yaml
name: scaffolder
description: Generates minimal buildable project skeleton with tests, CI/CD, Docker, and CLAUDE.md
model: sonnet
```

- **Execution agent** — generuje soubory
- Generuje **minimální skeleton** (~10-15 souborů), ne kompletní aplikaci
- Skeleton musí: buildovat, projít alespoň 1 testem, projít linterem
- Generuje CLAUDE.md s Automation Config (zná Config Contract z promptu)
- Batched generace: Core → Config → Quality → Ops → Docs
- Pinned dependency verze (ne latest, ne ^)

**Co skeleton obsahuje:**
- Build config (pyproject.toml / package.json / go.mod / ...)
- Entry point (src/main.py / src/index.ts / ...)
- 1 smoke test ("app starts" / "health check responds")
- Dockerfile + .dockerignore
- .gitignore
- CI config (.gitea/workflows/ci.yml nebo .github/workflows/ci.yml dle CI provideru)
- README.md
- CLAUDE.md s Automation Config (required + optional sekce, manual input označen HTML komentáři)

**Co skeleton NEOBSAHUJE:**
- Business logiku (endpointy, modely, controllery) — to implementuje Feature Pipeline
- Komplexní testy — jen 1 smoke test
- Frontend build pipeline (pokud jde o backend API)

### Commands

#### `/scaffold`

Hlavní scaffold command. Vytvoří nový projekt od nuly.

```
/CLAUDE-agents:scaffold [--lang <language>] [--framework <framework>] [--db <database>] [--ci <provider>]
```

**Flow:**
```
1. Detekce stavu cílového adresáře:
   - Prázdný adresář → plný scaffold
   - Existující projekt bez CLAUDE.md → nabídni scaffold-add claude-md
   - Existující projekt s CLAUDE.md → upozorni a nabídni implement-feature
2. stack-selector — analyzuje zadání + flags
   - Pokud flags poskytují dostatek info → přeskoč otázky
   - Pokud ne → doptej se (max 3 otázky)
3. scaffolder — generuje skeleton do temp adresáře
4. scaffold-validate — ověř skeleton v temp adresáři
   - Build OK? Test OK? Lint OK? CLAUDE.md valid?
   - Pokud fail → scaffolder retry (max 3×) s error outputem
   - Pokud 3× fail → rollback (smaž temp), report chyby
5. Přesuň z temp do cílového adresáře (atomická operace)
6. git init + initial commit
7. Report: "Skeleton ready. Seznam souborů. Další kroky."
8. Nabídni: "Chcete implementovat features přes /implement-feature?"
```

**Rollback strategie:** Scaffold generuje do temp adresáře. Teprve po úspěšné validaci se soubory přesunou do cílového adresáře. Pokud validace selže 3×, temp se smaže a uživatel dostane error report.

#### `/scaffold-add`

Přidání komponenty do existujícího projektu.

```
/CLAUDE-agents:scaffold-add <component>
```

Podporované komponenty:
- `claude-md` — vygeneruje CLAUDE.md s Automation Config (auto-detect stacku z existujících souborů)
- `ci` — vygeneruje CI/CD config
- `docker` — vygeneruje Dockerfile + docker-compose.yml
- `tests` — vygeneruje testovací setup

**Flow:**
```
1. Auto-detect existujícího tech stacku (package manifest, framework files)
2. Požádej o potvrzení: "Detekován {stack}. Generuji {component}. OK?"
3. scaffolder — generuje jen požadovanou komponentu
4. Validace
```

#### `/scaffold-validate`

Validace projektu.

```
/CLAUDE-agents:scaffold-validate
```

**Kontroly:**
- Build: detekuje build systém z package manifestu, spustí build command
- Test: spustí test command
- Lint: spustí linter (pokud nakonfigurován)
- CLAUDE.md: ověří přítomnost required sekcí Automation Config
- Docker: pokud Dockerfile existuje, ověří `docker build` (optional)

**Report format:**
```
[OK]   Build — pyproject.toml detected, `uv run pytest` passed
[OK]   Tests — 1 test passed
[WARN] Lint — no linter configured
[OK]   CLAUDE.md — all required sections present
[SKIP] Docker — no Dockerfile found

Result: 2 OK, 1 WARN, 1 SKIP
```

### Propojení scaffold → Feature Pipeline

Po scaffold skeleton je projekt buildable ale prázdný. Uživatel má dvě cesty:

1. **Interaktivní:** Napíše feature requesty do issue trackeru → spustí `/implement-feature` na každý
2. **Automatická:** Scaffold nabídne "Chcete implementovat features?" → uživatel popíše features v přirozeném jazyce → scaffold vytvoří issues v trackeru → spustí `/implement-feature` na každý

V MVP implementujeme cestu 1. Cesta 2 je optimalizace pro budoucnost.

## 4. Decomposition Engine

### Kdy dekomponovat

Ne každý ticket potřebuje dekompozici. Engine rozhoduje automaticky na základě analýzy:

**Rozhodovací kritéria:**

| Kritérium | Práh | Zdroj |
|-----------|------|-------|
| Affected files | >= 4 | code-analyst / architect output |
| Odhadovaná velikost diffu | > 60 řádků AND >= 3 soubory | code-analyst / architect |
| Risk level | HIGH | code-analyst |
| Logická nezávislost | >= 2 nezávislé změny | architect / code-analyst |
| Explicitní flag | `--decompose` | uživatel |

**Rozhodovací logika:**

```
if "--no-decompose" → SINGLE_PASS
if "--decompose" → DECOMPOSE
if risk == HIGH → DECOMPOSE
if affected_files >= 4 → DECOMPOSE
if diff_lines > 60 AND affected_files >= 3 → DECOMPOSE
if independent_changes >= 2 → DECOMPOSE
else → SINGLE_PASS
```

### Aplikace na oba pipelines

Decomposition se používá v **bug-fix** i **feature** pipeline:

**Bug-fix:** code-analyst report → decomposition decision → pokud DECOMPOSE: architect generuje task tree
**Feature:** architect output → decomposition decision (feature pipeline vždy prochází architect)

### DAG model

Subtask závislosti tvoří Directed Acyclic Graph (DAG). Podporované vzory:

- **Linear chain** (A → B → C): refaktoring → oprava → test
- **Fan-out** (A → [B, C, D]): společný základ → nezávislé změny
- **Diamond** (A → [B, C] → D): základ → dvě větve → integrace
- **Complex**: kombinace výše

### Task tree format

```yaml
# .claude/decomposition/PROJ-42.yaml
version: "1.0"
issue_id: "PROJ-42"
strategy: sequential | parallel | mixed
fail_strategy: fail-fast
commit_strategy: squash

subtasks:
  - id: "sub-1"
    title: "Popis subtasku"
    scope: "Co přesně udělat"
    files: [src/file1.ts, src/file2.ts]
    estimated_lines: 25
    depends_on: []
    acceptance_criteria:
      - "Kritérium 1"
      - "Kritérium 2"
    status: pending | in_progress | completed | failed | blocked | skipped
    commit_hash: null
    restore_point: null
```

### Execution model

**Sekvenční mod (default):**
```
subtask-1 → [fixer→build→reviewer→test→commit]
    → subtask-2 → [fixer→build→reviewer→test→commit]
        → integration test → squash → PR
```

Kontext mezi subtasky: engine předává fixer agentu summary předchozích subtasků (co se změnilo, proč, jaké soubory).

**Paralelní mod (worktrees, opt-in):**
```
BASE BRANCH → worktree-1: subtask-1 [pipeline]
            → worktree-2: subtask-2 [pipeline]  (paralelně)
            → worktree-3: subtask-3 [pipeline]  (paralelně)
            → MERGE → integration test → PR
```

Podmínky pro paralelní mod: Worktrees config existuje, žádné závislosti mezi subtasky, žádné sdílené soubory.

### Error handling

**Fail-fast (default):**
- Subtask selže → rollback tohoto subtasku → zastavit → report "X/N completed"
- Předchozí completed subtasky zůstanou
- `/resume-ticket` pokračuje od selhavšího subtasku

**Per-subtask rollback:**
```
Pro každý subtask engine ukládá:
  - restore_point: commit hash PŘED subtaskem
  - commit_hash: commit hash PO subtasku

Rollback subtask-N: git reset --hard {restore_point_N}
```

### Resume capability

Nový checkpoint pro `/resume-ticket`:

| Checkpoint | Signal | Přeskočí |
|-----------|--------|---------|
| `DECOMPOSE_PARTIAL` | Task tree existuje + některý subtask completed | Triage + analysis + hotové subtasky |

### Interaktivní schválení

Před exekucí engine zobrazí plán a čeká na potvrzení:

```
## Decomposition Plan — PROJ-42

| # | Subtask | Files | ~Lines | Závisí na |
|---|---------|-------|--------|-----------|
| 1 | Oprava validace | UserService.ts | ~25 | — |
| 2 | Error handler | UserController.ts | ~35 | 1 |
| 3 | Integrační test | validation.test.ts | ~40 | 1, 2 |

Strategie: sekvenční | Celkem: ~100 řádků
Pokračovat? [A/n]
```

### Decomposition Config

Nová **volitelná** sekce v Automation Config:

```markdown
### Decomposition

| Key | Value |
|-----|-------|
| Max subtasks | 7 |
| Fail strategy | fail-fast |
| Commit strategy | squash |
```

Defaults: max_subtasks=7, fail_strategy=fail-fast, commit_strategy=squash.

## 5. Dashboard

### Účel

Statická HTML stránka agregující pipeline data pro týmovou viditelnost. `/status` command zůstává pro rychlý terminálový pohled — dashboard je sdílitelný HTML artefakt.

### Data sources

1. **Issue tracker** (primární): issues, komentáře, stavy — přes MCP server
2. **Source control** (sekundární): branches, PRs — přes MCP server
3. **`[CLAUDE-agents]` komentáře**: pipeline stage, block důvody, triage checkpointy

### Command: `/dashboard`

```
/CLAUDE-agents:dashboard [--days <N>] [--output <path>] [--state <filter>] [--stage <filter>]
```

Flags:
- `--days N` — období (default: 30)
- `--output path` — výstupní soubor (default: `./claude-agents-dashboard.html`)
- `--state filter` — filtr podle issue stavu
- `--stage filter` — filtr podle pipeline stage

### HTML layout

6 sekcí v jednom self-contained HTML souboru (inline CSS + JavaScript):

1. **Header** — projekt, timestamp, verze pluginu
2. **Pipeline Overview** — 3 karty: Active (modrá), Blocked (červená), Completed (zelená)
3. **Issue Table** — sortovatelná tabulka se všemi issues (ID, název, stav, stage, agent, PR)
4. **Blocked Issues Panel** — detail blocked issues s důvodem a doporučením
5. **Recent Activity Timeline** — posledních 20 pipeline eventů
6. **Statistics** — success rate, throughput, avg time per stage, block distribution

### Parsování `[CLAUDE-agents]` komentářů

**Triage checkpoint:**
```
[CLAUDE-agents] Triage dokončen. Severity: {severity}. Area: {area}.
```

**Block comment:**
```
[CLAUDE-agents] 🔴 Pipeline Block
Agent: {agent}
Krok: {krok}
Důvod: {důvod}
Detail: {detail}
Doporučení: {doporučení}
```

Pipeline stage se odvozuje z posledního `[CLAUDE-agents]` komentáře:
- "Triage dokončen" → analysis
- Block od fixer → fix
- Block od reviewer → review
- Block od test-engineer → test
- PR link → publish
- Žádný komentář → triage

### Recommendations

Threshold-based pravidla (ne AI-generated):
- blocked > 0 → "N issues blocked — review them"
- Jeden agent blokuje > 30% → "Agent X blocks Y% — consider retry limits"
- Success rate < 50% → "Review pipeline configuration"

### Known limitations (L1)

- Pipeline stage pro aktivně postupující issues může být nepřesný (chybí intermediate checkpoint komentáře)
- Časy per stage jsou odhadnuté z existujících komentářů
- Token spotřeba není trackovaná per issue

## 6. Unified Pipeline Flow

### Kompletní flow: scaffold → features → bug fixes

```
USER: "Chci REST API pro správu objednávek, Python, PostgreSQL"

SCAFFOLD:
  1. stack-selector → Python 3.12, FastAPI, PostgreSQL, pytest, Gitea Actions
  2. scaffolder → skeleton (10-15 souborů)
  3. scaffold-validate → build OK, test OK
  4. git init + commit
  5. "Skeleton ready. Definujte features v issue trackeru."

FEATURE PIPELINE (pro každou feature):
  6. spec-analyst → strukturovaná specifikace
  7. architect → architektonický návrh + task tree
  8. DECOMPOSITION → subtasky (pokud komplexní)
  9. fixer → reviewer → test-engineer (per subtask)
  10. publisher → PR

BUG-FIX PIPELINE (pro každý bug):
  11. triage-analyst → triage
  12. code-analyst → impact report
  13. DECOMPOSITION → subtasky (pokud komplexní)
  14. fixer → reviewer → test-engineer
  15. publisher → PR

DASHBOARD:
  16. /dashboard → HTML přehled celého pipeline
```

### Agent model matrix (v3.0)

| Agent | Model | Type | Pipeline |
|-------|-------|------|----------|
| triage-analyst | sonnet | read-only | bug-fix |
| code-analyst | sonnet | read-only | bug-fix |
| **spec-analyst** | sonnet | read-only | feature, scaffold |
| **architect** | opus | read-only | feature, bug-fix (decomposition) |
| **stack-selector** | sonnet | read-only | scaffold |
| **scaffolder** | sonnet | execution | scaffold |
| fixer | opus | execution | bug-fix, feature |
| reviewer | opus | read-only | bug-fix, feature |
| test-engineer | sonnet | execution | bug-fix, feature |
| e2e-test-engineer | sonnet | execution | bug-fix, feature |
| publisher | haiku | execution | bug-fix, feature, scaffold |
| rollback-agent | haiku | execution | bug-fix, feature |

### Skill routing rozšíření

Stávající `bug-workflow` skill se rozšíří o nové intenty:

| User Intent | Command |
|-------------|---------|
| "Scaffold nový projekt" / "Vytvoř projekt" | scaffold |
| "Přidej CLAUDE.md" / "Nastav CI" | scaffold-add |
| "Ověř projekt" / "Validuj setup" | scaffold-validate |
| "Implementuj feature" / "Přidej funkcionalitu" | implement-feature |
| "Zobraz dashboard" / "Přehled pipeline" | dashboard |

## 8. Implementační pořadí

### Závislosti

```
Feature Pipeline ← základ (spec-analyst, architect, implement-feature)
    ↓
Scaffold ← potřebuje Feature Pipeline pro "implement features after scaffold"
    ↓
Decomposition ← potřebuje architect z Feature Pipeline

Dashboard ← nezávislé, paralelizovatelné s čímkoliv
```

### Navrhované fáze

**Fáze A: Feature Pipeline (~12h)**
1. Agent: spec-analyst
2. Agent: architect
3. Command: implement-feature
4. Config Contract: Feature Workflow sekce
5. Skill routing rozšíření
6. CLAUDE.md update

**Fáze B: Scaffold (~15-20h)**
7. Agent: stack-selector
8. Agent: scaffolder
9. Command: scaffold
10. Command: scaffold-add
11. Command: scaffold-validate
12. Skill routing rozšíření

**Fáze C: Decomposition (~65-95h)**
13. Decomposition engine logika (v fix-ticket, implement-feature)
14. `--decompose` / `--no-decompose` flag parsing
15. Task tree YAML persistence (.claude/decomposition/)
16. Sekvenční execution model
17. Per-subtask rollback
18. Resume capability (rozšíření resume-ticket)
19. Config Contract: Decomposition sekce
20. Paralelní execution model (worktrees) — po ověření sekvenčního

**Fáze D: Dashboard (~9h)**
21. Command: dashboard
22. HTML generace (inline CSS + JS)
23. Issue tracker + source control data fetching
24. `[CLAUDE-agents]` komentář parsing

**Fáze E: Docs & Release**
25. README update
26. CLAUDE.md update (agent matrix, command list)
27. plugin.json version bump → 3.0.0
28. CHANGELOG

### Celkový odhad: ~100-130h
