# Discussion: Bug-Fix Pipeline Improvements

**Date:** 2026-03-08
**Agents:** triage-analyst, fixer, reviewer
**Status:** PROPOSED

---

## triage-analyst (Analytical, systematic, concise)

The biggest leak in our pipeline is right at the start -- garbage in, garbage out. I see bug reports every day that say "it doesn't work" with zero reproduction steps, no expected vs. actual, no environment info. My current process catches the worst offenders (confidence < 50% gets blocked), but that binary gate is too crude. A ticket can pass triage at 51% confidence and still be woefully underspecified for the fixer.

**Acceptance Criteria extraction is the single highest-leverage change we can make.** Today I output a structured triage analysis with summary, area, severity, and reproduction steps. But I never explicitly extract or synthesize acceptance criteria. The fixer gets a vague "fix the thing described here" contract. If I added an AC extraction step -- pull explicit AC from the ticket, or if missing, propose AC and write them back to the issue tracker -- every downstream agent would have a concrete, testable contract to work against. This is not optional padding; this is the definition of "done" that the entire pipeline lacks.

On input quality: I already do duplicate detection and clarity validation, but I should also assess **domain complexity**. Some bugs touch straightforward CRUD logic; others involve concurrency, caching, or distributed state. My severity assessment (CRITICAL/HIGH/MEDIUM/LOW) measures user impact, not fix difficulty. I should add a **complexity dimension** -- an estimate of how hard this will be to fix correctly. This feeds directly into decomposition decisions and helps the fixer calibrate expectations.

For feedback from failures: I have a unique vantage point. If I start tracking which of my triage outputs led to blocked fixes downstream, I can identify patterns -- was my area identification wrong? Did I miss a related component? Were my reproduction steps incomplete? I propose that when a fix blocks, the block comment gets linked back to the triage analysis so we can do post-mortem analysis. A simple `[ceos-agents] Triage accuracy: {area_correct: yes/no, repro_sufficient: yes/no}` annotation on block comments would give us data to improve triage quality over time.

On manual validation: some bugs genuinely cannot be verified by automated tests alone -- visual regressions, UX flows, race conditions that only manifest under load. I should flag these during triage with a `Manual verification required: yes/no` field and a brief justification. This signal propagates through the pipeline and triggers the manual verification step at the end instead of surprising the developer after the PR is created.

## fixer (Pragmatic, minimal, surgical)

I will be blunt: the current pipeline treats me like a code monkey who receives a bug description and produces a diff. The quality of what I receive directly determines whether my fix succeeds or fails. When I block, 70% of the time it is because the analysis upstream was insufficient -- wrong root cause location, missing context about how the module interacts with others, or no clarity on what "fixed" actually means.

**Acceptance criteria as a contract is non-negotiable for me.** Right now I read the triage analysis and impact report, then I have to infer what "correct behavior" looks like. Sometimes the bug report says "button doesn't work" and the impact report says "file: button-handler.ts, 3 callers." Neither tells me what the button should do. If I had explicit AC -- "when user clicks Submit with valid form data, the form submits and shows success toast within 2 seconds" -- I can write a precise regression test in the RED phase and verify the GREEN phase against concrete criteria. Without AC, my TDD cycle is guessing.

**On the fixer-reviewer loop:** The adversarial reviewer is valuable but sometimes creates friction without value. When the reviewer raises 3 issues because they are required to find at least 3, and two of them are LOW-severity style nitpicks, I waste an iteration addressing cosmetic concerns while the actual risk goes unexamined. I would rather the reviewer focus on **AC fulfillment** -- does this fix satisfy every acceptance criterion? -- than on hunting for architectural violations in a 15-line diff.

**Failure feedback is critical for me.** When I block, the block comment captures what went wrong, but nobody learns from it. I propose a **fix retrospective** mechanism: after a block, automatically analyze the failure chain. Was the root cause location from code-analyst correct? Did I attempt the right approach? Was the problem actually fixable within the 100-line diff constraint? This data should feed into a lightweight knowledge base -- not a complex system, just a `.claude/fix-history/{ISSUE-ID}.md` file with the failure analysis. When I encounter a similar bug in the same module later, having that history is enormously valuable. Code-analyst already checks git history, but it does not check pipeline history in a structured way.

**On manual validation:** I support an optional manual verification mode, but it must be opt-in per ticket, not a blanket requirement. My job is to produce a minimal correct fix with passing tests. If the triage flags `Manual verification required: yes`, then after my fix passes review and tests, the pipeline should generate a **verification checklist** derived from the AC -- specific steps for a human to perform -- and pause. But this should never be the default path; it would kill throughput for bugs that are perfectly verifiable by automated tests.

**The 100-line diff limit is sometimes the enemy.** Complex bugs in tightly coupled code genuinely need larger changes. Rather than blocking and punting to a human, I should have a mechanism to request decomposition mid-fix -- signal to the orchestrator "this needs decomposition" even if code-analyst estimated it as simple. Currently, decomposition is decided before I start. I need an escape hatch.

## reviewer (Adversarial, evidence-driven, thorough)

Let me push back on the fixer's complaint about my 3-issue minimum. That rule exists because fixers routinely submit changes that look correct on the happy path but crumble at boundaries. If I cannot find 3 real issues, I must provide explicit per-checklist justification -- that is the safety valve. The minimum forces me to look harder, and that discipline catches real bugs. I will not apologize for it.

That said, the fixer is right about one thing: **I currently review against an implicit, unstated definition of correctness.** I check root cause, completeness, conventions, regressions, security, performance, and over-engineering. But I have no formal acceptance criteria to review against. I am inferring "correct behavior" from the bug report and my reading of the code. This is a significant gap.

**AC-based review would transform my effectiveness.** If every ticket entering my review loop had explicit acceptance criteria, I would add a new mandatory checklist item: "AC fulfillment -- does the fix demonstrably satisfy each criterion?" This is not the same as checking root cause. A fix can address the root cause technically but fail to meet the user-facing acceptance criteria. For example: fix stops the crash (root cause addressed) but the feature still returns wrong data (AC not met). Today I might catch this, but it is not systematic. With AC, it becomes a deterministic check.

**On validation at the end of the pipeline:** The test-engineer currently writes 1-3 unit tests verifying the fix and preventing regressions. But unit tests verify code behavior, not business requirements. I strongly advocate for an **AC verification step** after testing -- a dedicated pass where someone (a new agent or an enhanced test-engineer) goes through each AC item and maps it to either a passing test, a code-level proof (e.g., "this null check prevents the crash described in AC #2"), or flags it as unverifiable by automation. This is different from my code review. I review code quality and correctness; the AC verifier reviews requirement fulfillment.

**On failure analysis:** I see patterns the fixer does not. When I REQUEST_CHANGES and the fixer comes back with the same fundamental approach in iteration 2, it almost always means the fixer does not understand the domain well enough. The code-analyst's impact report should include more **domain context** -- not just call hierarchies, but business rules that govern the affected module. "This function calculates tax rates and must comply with {regulation}" is the kind of context that prevents the fixer from making a technically clean but logically wrong fix.

**Manual verification is essential for certain bug classes.** UI bugs, visual regressions, accessibility issues, and anything involving user-perceived behavior cannot be fully validated by unit tests or even E2E tests. I propose a ticket-level flag `verification_mode: auto | manual | hybrid` that triage sets. In `manual` mode, the pipeline generates a structured test scenario from the AC and pauses for human confirmation. In `hybrid`, automated tests run first, then a manual checklist covers the aspects automation cannot reach. The key insight: the reviewer (me) should determine whether the fix is ready for manual verification, not just whether the code is correct. I should be the gate that says "code looks good, but AC #3 requires visual confirmation -- routing to manual verification."

## Synthesis

**Key agreements across all three agents:**
1. Acceptance criteria are missing from the pipeline and every agent feels the impact.
2. Failure analysis is currently write-only -- block comments are created but never analyzed for patterns.
3. Manual verification is needed for some bug classes but must be opt-in, not default.
4. The test-engineer validates code behavior but not requirement fulfillment -- there is a gap.

**Key disagreement:**
- Fixer wants the reviewer's 3-issue minimum relaxed; reviewer defends it as essential discipline. Resolution: keep the minimum but add AC fulfillment as a first-class checklist item, which gives the reviewer substantive criteria to check rather than hunting for filler issues.

The first three proposals form a coherent "AC pipeline" that can be shipped together. Manual verification is the most complex and should come last.
