# Documentation Overhaul — Full EN Rewrite

**Date:** 2026-03-02
**Status:** PROPOSED
**Version target:** 3.2.0 (MINOR — no functional Automation Config contract changes)

## Problem Statement

The CLAUDE-agents plugin documentation has grown organically across 94 markdown files and ~25,800 lines (~6,200 active, ~19,600 in archival design docs). Key issues:

1. **Bilingual split** — Czech for commands, setup docs, README.cs.md; English for plugin spec. Internal team reads English fluently; Czech adds maintenance burden and duplication.
2. **Underdocumented** — No architecture deep-dive, no getting-started tutorial, no agent output examples, no troubleshooting guide.
3. **Duplication** — Config contract in 3 places. Pipeline descriptions in 3 places. Czech README (536 lines) duplicates English README (158 lines).
4. **Flat structure** — `docs/setup/` too narrow; no separation between guides, reference, and tutorials.
5. **Stale artifacts** — Old worktree, untracked review report, no design doc index.

## Design Decisions

### Language: Everything EN

All active documentation, commands, agents, examples, and tests translated to English. Historical design docs in `docs/plans/` left as-is (archival).

### Framework: Diataxis

Documentation restructured following [Diataxis](https://diataxis.fr/):

| Type | Purpose | Location |
|------|---------|----------|
| Tutorial | "Follow along and learn" | `docs/getting-started.md` |
| How-to Guide | "Solve a specific problem" | `docs/guides/` |
| Reference | "Look up facts" | `docs/reference/` |
| Explanation | "Understand why" | `docs/architecture.md` |

### Diagrams: Mermaid

Replace ASCII pipeline/architecture diagrams with Mermaid (renders in Gitea/GitHub).

Diagram types:
- Architecture overview: `graph TD` (top-down)
- Pipelines: `flowchart TD` with decision diamonds for block/continue
- Data flow: `graph LR` (left-right, config → command → agent)

### Config Contract: Stays in CLAUDE.md

**Decision:** Do NOT extract Config Contract from CLAUDE.md.

**Rationale:** 13 of 22 commands explicitly instruct to "read from CLAUDE.md" (analyze-bug, changelog, create-pr, dashboard, estimate, fix-bugs, fix-ticket, implement-feature, metrics, prioritize, publish, resume-ticket, status). A further 6 commands reference Automation Config in other ways (check-setup, onboard, migrate-config, scaffold, scaffold-add, scaffold-validate). Only 3 commands (template, version-bump, version-check) don't read project CLAUDE.md. Moving the spec would break these instructions without updating all 19 commands + 13 agents. Risk outweighs benefit.

Instead: `docs/reference/automation-config.md` is an **extended guide** with examples, validation rules, and migration notes — referencing CLAUDE.md as the canonical definition.

### Version: 3.2.0 MINOR

Per versioning policy: MAJOR = breaking change in Automation Config contract (new required key, renamed section). This overhaul changes zero Automation Config keys. Block template fields are internal (generated by agents, parsed by commands). Therefore: MINOR.

User will recreate Automation Config on consuming projects via `/CLAUDE-agents:onboard`, eliminating any backward compatibility concerns with old Czech block comments.

## Version Bump

Single version bump **after all 4 phases complete**: 3.1.1 → 3.2.0

Rationale:
- MINOR: New documentation structure, new docs — backward-compatible feature
- Not MAJOR: No functional changes to Automation Config contract (no new required keys, no renamed sections). The `Profil` → `Profile` column header rename in Pipeline Profiles is cosmetic — commands parse by section name and row position, not by column header text.
- Block template field rename is internal interface (agents generate, commands parse)
