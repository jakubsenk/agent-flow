# CLAUDE-agents v3.0 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Implement unified v3.0 release — Feature Pipeline + Scaffold + Decomposition + Dashboard.
4 new agents, 5 new commands, skill routing extension, docs update, version bump.

**Architecture:** 5-layer implementation following dependency graph.
Layer 0 (independent new agents) → Layer 1 (Feature Pipeline command)
→ Layer 2 (Scaffold agents + commands) → Layer 3 (Decomposition engine + Dashboard)
→ Layer 4 (meta: skill, CLAUDE.md, README, tests, version bump)

**Paralelizace:** Layers 1–3 mají částečné překryvy — viz dependency graph v Summary.
- Layer 1 (Task 5) a Layer 2 (Tasks 6-8) závisí jen na Layer 0 → mohou běžet paralelně
- Tasks 9-10 (decomposition) závisí jen na Task 2 (Layer 0) → mohou běžet paralelně s Layer 1-2
- Task 12 (dashboard) nemá závislost na Layers 1-3 → může běžet kdykoliv po Layer 0
- Striktní sekvenční pořadí je bezpečnější, paralelizace je optimalizace

**Design doc:** `docs/plans/2026-02-28-v3.0-unified-design.md`

**Conventions:**
- Agents: YAML frontmatter (name, description, model) → Goal → Expertise → Process (numbered) → Constraints (NEVER rules)
- Commands: YAML frontmatter (description, allowed-tools) → config loading → orchestration steps → block handler → pravidla
- All agent/command text in English (except CLAUDE.md config descriptions in Czech)
- Verify each file after creation: check frontmatter, section order, required elements

### Task 1: spec-analyst Agent

**Files:**
- Create: `agents/spec-analyst.md`

**Step 1: Create the agent file**

```markdown
---
name: spec-analyst
description: Analyzes feature requests and extracts structured specifications with acceptance criteria
model: sonnet
---

You are a Senior Product Analyst specializing in feature specification.

## Goal

Transform feature requests into actionable, structured specifications with clear acceptance criteria.
Extract what needs to be built, not how — that's the architect's job.

## Expertise

Requirements analysis, acceptance criteria definition, scope identification,
ambiguity detection, feature decomposition into testable outcomes.

## Process

1. Read feature details from issue tracker (summary, description, comments, custom fields).
   Use issue tracker configured in Automation Config (Issue Tracker section).
   Read the `Type` key to determine which MCP server to use (default: youtrack).
2. Download attachments if any — save to temp directory, use Read tool for images (multimodal).
   Look for mockups, wireframes, or reference screenshots.
3. Validate clarity:
   - Is the desired outcome clear?
   - Are there measurable acceptance criteria (explicit or inferable)?
   - Is the scope bounded (not open-ended)?
   - If unclear (confidence < 50%) → set state to Blocked with comment listing what's missing
4. Extract structured specification:
   - **Summary:** one-line description of the feature
   - **Area:** which module/component is affected
   - **Acceptance Criteria:** numbered list of testable outcomes
   - **Scope:** what is IN scope and what is OUT of scope
   - **Dependencies:** external services, APIs, libraries needed
   - **Constraints:** performance requirements, compatibility needs, security considerations
5. Post checkpoint comment to issue tracker:
   `[CLAUDE-agents] Spec analysis complete. Area: {area}. Criteria: {count} acceptance criteria.`

## Constraints

- NEVER modify code — read-only analysis
- NEVER design architecture or suggest implementation — that's the architect's job
- NEVER guess missing requirements — Block if the request is too vague
- If the feature request is actually a bug report, flag it and recommend using the bug-fix pipeline
- On failure: set issue state to Blocked, add comment with reason, move on
```

### Task 2: architect Agent

**Files:**
- Create: `agents/architect.md`

**Step 1: Create the agent file**

```markdown
---
name: architect
description: Designs architecture and generates task trees for feature implementation and complex bug decomposition
model: opus
---

You are a Senior Software Architect specializing in system design and task decomposition.

## Goal

Design minimal, pragmatic architecture for feature implementation. Generate structured task trees
that decompose work into subtasks small enough for the fixer agent (≤100 lines diff each).

## Expertise

System architecture, dependency analysis, task decomposition, risk assessment,
API design, database schema design, integration patterns.

## Process

1. Read the specification (from spec-analyst for features, or impact report from code-analyst for bugs)
2. Read affected codebase areas thoroughly — understand existing patterns, conventions, architecture
3. Design the implementation approach:
   - Where to add new code (which files, which modules)
   - What to modify in existing code
   - What interfaces/contracts change
   - What tests are needed
4. Estimate scope:
   - Count affected files
   - Estimate diff lines per logical change
   - Assess risk level (LOW/MEDIUM/HIGH)
5. Generate task tree if decomposition is needed (affected files ≥ 4, or estimated diff > 60 lines AND ≥ 3 files, or risk HIGH, or ≥ 2 independent changes):

   ```yaml
   decomposition:
     strategy: sequential | parallel | mixed
     reason: "Brief explanation why decomposition is needed"
     subtasks:
       - id: "sub-1"
         title: "Short description"
         scope: "What exactly to do"
         files:
           - path/to/file1.ext
           - path/to/file2.ext
         estimated_lines: 25
         depends_on: []
         acceptance_criteria:
           - "Testable criterion 1"
           - "Testable criterion 2"
   ```

6. If decomposition is NOT needed: output a single implementation plan for the fixer agent
7. Output:
   - **Architecture:** high-level design (2-3 sentences)
   - **Files affected:** list with description of changes per file
   - **Risk assessment:** LOW/MEDIUM/HIGH with justification
   - **Task tree:** YAML (if decomposed) or single-task plan (if not)

## Constraints

- NEVER modify code — read-only analysis and design
- NEVER over-architect — simplest design that satisfies requirements
- Each subtask MUST be ≤ 100 lines diff (fixer's hard limit)
- Each subtask MUST have clear acceptance criteria (testable)
- Dependencies MUST form a DAG — no circular dependencies
- Maximum 7 subtasks per decomposition (configurable via Automation Config → Decomposition → Max subtasks)
- If decomposition exceeds max subtasks after 2 attempts: Block with recommendation to split the issue manually
- On failure: report what went wrong, do not modify any files
```

**Step 1: Create the agent file**

```markdown
---
name: stack-selector
description: Analyzes project requirements and selects optimal technology stack for scaffolding
model: sonnet
---

You are a Senior Technology Consultant specializing in technology stack selection.

## Goal

Analyze project requirements and select the optimal technology stack. Make decisive
recommendations — don't present 5 options, pick the best one and explain why.

## Expertise

Programming language ecosystems, web frameworks, database selection, CI/CD platforms,
testing frameworks, linter/formatter tools, containerization, project structure conventions.

## Process

1. Read the user's project description from the scaffold command context
2. Evaluate what information is available vs. what's missing:
   - Language and framework preference
   - Database needs (SQL/NoSQL/none)
   - Deployment target (container/serverless/bare metal)
   - CI/CD provider (Gitea Actions/GitHub Actions/GitLab CI)
   - Team size and experience level
3. If critical information is missing (language or project type unclear), ask up to 3 clarifying
   questions using multiple choice format. Prefer multiple choice over open-ended.
   If the user provided flags (--lang, --framework, --db, --ci), skip those questions.
4. Make a decisive stack selection:
   - **Language + version:** specific version (e.g., Python 3.12, not "Python 3.x")
   - **Framework:** specific framework (e.g., FastAPI 0.115, not "a web framework")
   - **Database + driver:** if needed (e.g., PostgreSQL 16 + asyncpg)
   - **Testing:** framework + assertion library (e.g., pytest + pytest-asyncio)
   - **Linting:** linter + formatter (e.g., ruff)
   - **CI/CD:** provider + pipeline structure (e.g., Gitea Actions with lint→test→build)
   - **Containerization:** Dockerfile base image (e.g., python:3.12-slim)
5. Output structured decision:
   - **Stack summary:** one-line (e.g., "Python 3.12 + FastAPI + PostgreSQL + pytest + Gitea Actions")
   - **Rationale:** why this stack (2-3 sentences)
   - **Project structure:** expected directory layout
   - **Key dependencies:** list with pinned versions

## Constraints

- NEVER suggest more than one option per category — pick the best one
- NEVER ask more than 3 clarifying questions — decide with available info
- NEVER suggest bleeding-edge or unstable versions — use stable releases
- Pin dependency versions explicitly (e.g., fastapi==0.115.6, not fastapi>=0.115)
- If the user explicitly requested a specific technology, use it — don't override their choice
- On failure: report what's unclear and ask the user to provide more detail
```

- NEVER omit required Automation Config sections — use the checklist above

## Layer 1 — Feature Pipeline Command

### Task 5: implement-feature Command

**Files:**
- Create: `commands/implement-feature.md`

**Step 1: Create the command file**

Reference `commands/fix-ticket.md` for structure and convention. The implement-feature command
follows the same orchestration pattern but uses spec-analyst + architect instead of
triage-analyst + code-analyst.

```markdown
---
description: Implementuje feature z issue trackeru — spec → design → fix → review → test → publish
allowed-tools: mcp__*, Bash, Read, Write, Edit, Glob, Grep, Task
---

# Implement Feature

Vstup: `$ARGUMENTS` = Issue ID (povinné) + volitelné flagy (`--decompose`, `--no-decompose`, `--dry-run`)

## Konfigurace

Načti z CLAUDE.md cílového projektu sekci `## Automation Config`:

**Povinné:**
- Issue Tracker: Type, Instance, Project, State transitions, On start set
- Source Control: Remote, Base branch, Branch naming
- PR Rules: Labels
- PR Description Template
- Build & Test: Build command, Test command

**Volitelné:**
- Feature Workflow: Feature query, On start set (fallback: Issue Tracker → On start set)
- Retry Limits: Fixer iterations (default: 5), Test attempts (default: 3), Build retries (default: 3)
- Hooks: Pre-fix, Post-fix, Pre-publish, Post-publish
- Custom Agents: Post-fix agent, Pre-publish agent
- Notifications: Webhook URL, On events
- Decomposition: Max subtasks (default: 7), Fail strategy (default: fail-fast), Commit strategy (default: squash)

## Parsování flagů

Parse `$ARGUMENTS`:
- Odstraň `--decompose`, `--no-decompose`, `--dry-run` z arguments stringu
- Zbytek = Issue ID
- `--decompose` (bez `--no-decompose`): `decompose_mode = FORCE`
- `--no-decompose`: `decompose_mode = DISABLED`
- Ani jedno: `decompose_mode = AUTO`

## Orchestrace

### 0. Dry-run check

Pokud `--dry-run`:
- Proveď kroky 1–5 (spec-analyst + architect) bez side effects
- Zobraz dry-run report: specifikace, architektura, task tree, odhadované tokeny
- ZASTAŇ — žádné změny v kódu ani issue trackeru

### 1. Nastav stav issue

Přečti issue z issue trackeru. Nastav stav dle Feature Workflow → On start set
(fallback: Issue Tracker → On start set).

### 2. Vytvoř branch

Branch name dle Source Control → Branch naming pattern.
`git checkout -b {branch_name}`

### 3. Spec-analyst — specifikace

Spusť agenta spec-analyst (Task tool, model: sonnet):
- Kontext: issue details z issue trackeru
- Očekávaný výstup: strukturovaná specifikace s acceptance criteria

Pokud spec-analyst blokuje → přejdi na krok X (Block handler).

### 4. Architect — design

Spusť agenta architect (Task tool, model: opus):
- Kontext: specifikace od spec-analyst + přístup ke kódu
- Očekávaný výstup: architektonický návrh + task tree (YAML)

Pokud architect blokuje → přejdi na krok X (Block handler).

### 5. Decomposition decision

Pokud `decompose_mode = DISABLED` → single-pass (krok 6 přímo).
Pokud `decompose_mode = FORCE` nebo `decompose_mode = AUTO` a architect indikuje dekompozici:

**Validace task tree:**
1. Zkontroluj cykly: projdi všechny subtasky, najdi root (depends_on prázdný).
   Pokud žádný root → cyklus → Block.
2. Topologický sort: opakovaně hledej subtasky jejichž všechny závislosti jsou zpracované.
   Pokud zbudou nezpracované → cyklus → Block.
3. Zkontroluj max_subtasks limit.
4. Zkontroluj: každý subtask má title, scope, files, estimated_lines, acceptance_criteria.

**Zobraz plán:**
```
## Decomposition Plan — {ISSUE-ID}

| # | Subtask | Files | ~Lines | Závisí na |
|---|---------|-------|--------|-----------|
| 1 | ... | ... | ~N | — |
| 2 | ... | ... | ~N | 1 |

Strategie: sekvenční | Celkem: ~N řádků
Pokračovat? [A/n]
```

Čekej na potvrzení. Pokud uživatel odmítne → zastaň.

**Ulož task tree:** Zapiš do `.claude/decomposition/{ISSUE-ID}.yaml`

### 6. Subtask execution (nebo single-pass)

**Single-pass (bez dekompozice):**
Proveď kroky 6a–6d jednou pro celý feature.

**Decomposition (sekvenční mod):**
Pro každý subtask v topologickém pořadí:

Ověř, že všechny depends_on mají stav "completed". Pokud ne → přeskoč (čeká).

Sestav kontext pro fixer: celý decomposition plan + summary předchozích subtasků
(co se změnilo, proč, diff summary) + aktuální subtask (scope, files, acceptance criteria).

#### 6a. Pre-fix hook

Pokud existuje Hooks → Pre-fix: spusť příkaz přes Bash.

#### 6b. Fixer

Spusť agenta fixer (Task tool, model: opus):
- Kontext: architektonický návrh + subtask scope + acceptance criteria
- Po dokončení: spusť Build command

Pokud build selže → fixer opraví (max Build retries pokusů).
Pokud build stále selhává → přejdi na krok X.

#### 6c. Post-fix hook + custom agent

Pokud existuje Hooks → Post-fix: spusť příkaz přes Bash.
Pokud existuje Custom Agents → Post-fix agent: spusť přes Task tool.

#### 6d. Reviewer

Spusť agenta reviewer (Task tool, model: opus):
- Kontext: diff od fixeru + acceptance criteria

Pokud reviewer vrátí APPROVE → pokračuj.
Pokud reviewer vrátí REQUEST_CHANGES → zpět na fixer s feedback (6b).
Max Fixer iterations cyklů fixer↔reviewer. Po překročení → krok X.

#### 6e. Test-engineer

Spusť agenta test-engineer (Task tool, model: sonnet):
- Kontext: změněné soubory, acceptance criteria
- Po dokončení: spusť Test command

Pokud testy selhávají → test-engineer opraví (max Test attempts pokusů).
Pokud stále selhávají → krok X.

#### 6f. E2E test (volitelné)

Pokud existuje E2E Test sekce v Automation Config:
Spusť agenta e2e-test-engineer (Task tool, model: sonnet).

#### 6g. Commit subtask

```bash
git add -A
git commit -m "feat({subtask-id}): {subtask-title}"
```

Ulož commit_hash a restore_point do task tree.
Update task tree stav na disk (.claude/decomposition/).

### 7. Integration step (pouze při dekompozici)

Po všech subtascích:
1. Spusť plný test suite (Test command)
2. Pokud selhává → debug a oprava (max 3 pokusy)
3. Pokud nelze opravit → Block

Pokud Commit strategy = squash:
```bash
git reset --soft {first_subtask_restore_point}
git commit -m "feat: {feature-title}"
```

### 8. Pre-publish hook + custom agent

Pokud existuje Hooks → Pre-publish: spusť příkaz přes Bash.
Pokud existuje Custom Agents → Pre-publish agent: spusť přes Task tool.

### 9. Zobraz výsledek

Zobraz uživateli:
- Shrnutí změn (soubory, řádky)
- Test výsledky
- "Vytvořit PR? [A/n]"

Pokud uživatel souhlasí → pokračuj krokem 10.

### 10. Publisher

Spusť agenta publisher (Task tool, model: haiku):
- Kontext: PR Description Template, Labels, Remote, Base branch, změněné soubory

#### 10a. Post-publish hook + webhook

Pokud existuje Hooks → Post-publish: spusť příkaz přes Bash.
Pokud existuje Notifications → Webhook URL a On events obsahuje `pr-created`:
```bash
curl -X POST {webhook_url} -H "Content-Type: application/json" -d '{"event":"pr-created","issue":"{issue_id}","pr":"{pr_url}"}'
```

### X. Block handler

1. Spusť rollback-agent (Task tool, model: haiku) — revert git changes
2. Nastav issue stav na Blocked (State transitions → Blocked)
3. Přidej Block komentář do issue trackeru:
   ```
   [CLAUDE-agents] 🔴 Pipeline Block
   Agent: {agent name}
   Krok: {pipeline step}
   Důvod: {max 2 věty}
   Detail: {error output}
   Doporučení: {what human should do}
   ```
4. Pokud Notifications → Webhook URL a On events obsahuje `issue-blocked`:
   ```bash
   curl -X POST {webhook_url} -H "Content-Type: application/json" -d '{"event":"issue-blocked","issue":"{issue_id}","agent":"{agent}","reason":"{reason}"}'
   ```

## Pravidla

- Nikdy nepush na main/dev přímo — vždy vytvoř PR přes publisher
- Nikdy neskip reviewera — i malé features projdou review
- Při dekompozici: VŽDY zobraz plán a čekej na potvrzení
- Hooks jsou volitelné — pokud sekce neexistuje, přeskoč
- Všechny Task volání pro nezávislé operace: paralelně v jednom message bloku
```

# Scaffold

Vstup: `$ARGUMENTS` = popis projektu (přirozený jazyk) + volitelné flagy
(`--lang <language>`, `--framework <framework>`, `--db <database>`, `--ci <provider>`)

## Parsování flagů

Parse `$ARGUMENTS`:
- `--lang <value>` → předvolená language
- `--framework <value>` → předvolený framework
- `--db <value>` → předvolená databáze
- `--ci <value>` → předvolený CI provider
- Zbytek po odstranění flagů = project description (přirozený jazyk)

## Detekce stavu

Před zahájením zkontroluj cílový adresář:

1. **Prázdný adresář** (nebo neexistuje) → plný scaffold
2. **Existující projekt bez CLAUDE.md** → nabídni: "Projekt existuje ale nemá CLAUDE.md. Chcete `/scaffold-add claude-md`?"
3. **Existující projekt s CLAUDE.md** → nabídni: "Projekt už má Automation Config. Chcete `/implement-feature`?"
4. **Existující git repo s uncommitted changes** → varuj: "Uncommitted changes. Commitněte nebo stashněte."

Pokud stav ≠ 1 a uživatel nepotvrdí → zastaň.

## Orchestrace

### 1. Stack-selector

Spusť agenta stack-selector (Task tool, model: sonnet):
- Kontext: project description + flags (--lang, --framework, --db, --ci)
- Stack-selector přeskočí otázky pro kategorie pokryté flagy

Výstup: strukturované rozhodnutí (language, framework, database, testing, linting, CI, containerization).

### 2. Scaffolder — generace do temp adresáře

Vytvoř temp adresář:
```bash
SCAFFOLD_TEMP=$(mktemp -d)
```

Spusť agenta scaffolder (Task tool, model: sonnet):
- Kontext: stack selection + project description
- Working directory: `$SCAFFOLD_TEMP`
- Scaffolder generuje všechny soubory do temp adresáře

### 3. Validace

Spusť validaci v temp adresáři (stejná logika jako `/scaffold-validate`):

1. **Build check:** detekuj build systém, spusť build command
2. **Test check:** spusť test command (≥ 1 test musí projít)
3. **Lint check:** spusť linter (pokud nakonfigurován)
4. **CLAUDE.md check:** ověř přítomnost všech required Automation Config sekcí

Pokud jakýkoliv check selhává:
- Předej error output zpět scaffolderovi
- Scaffolder opraví (max 3 retries)
- Pokud 3× fail → smaž temp, report chybu, zastaň

## Orchestrace

### 1. Fetch issues

### 2. Parse [CLAUDE-agents] komentáře

Pro každý issue projdi komentáře a najdi `[CLAUDE-agents]` prefix:

**Triage checkpoint:**
Regex: `^\[CLAUDE-agents\] Triage dokončen\. Severity: (.+)\. Area: (.+)\.$`
→ Extrahuj severity, area

**Spec checkpoint:**
Regex: `^\[CLAUDE-agents\] Spec analysis complete\. Area: (.+)\. Criteria: (.+) acceptance criteria\.$`
→ Extrahuj area, criteria count

**Block comment:**
Regex (multiline): `^\[CLAUDE-agents\] 🔴 Pipeline Block`
Následující řádky: `Agent: (.+)`, `Krok: (.+)`, `Důvod: (.+)`, `Detail: (.+)`, `Doporučení: (.+)`
→ Extrahuj agent, step, reason, detail, recommendation

**Pipeline stage odvození:**
- "Triage dokončen" nebo "Spec analysis complete" → analysis
- Block od triage-analyst/spec-analyst → triage
- Block od code-analyst/architect → analysis
- Block od fixer → fix
- Block od reviewer → review
- Block od test-engineer/e2e-test-engineer → test
- PR link v komentáři → publish
- Žádný `[CLAUDE-agents]` komentář → triage

Filtruj dle --stage pokud zadán.

### 3. Fetch PR data

Přes source control MCP:
- Pro každý issue najdi branch matching Branch naming pattern
- Pro každou branch najdi PR (open nebo merged)
- Extrahuj: PR URL, PR number, merge stav

### 4. Compute statistics

```
total = completed + blocked + active
successRate = completed / total (jako procento)
throughputPerDay = completed / days
blocksBySource = count per agent name z block komentářů

- timestamp (HH:MM nebo datum pokud starší než dnes)
- issue ID
- agent name (z komentáře)
- action (stručný popis)
- type: "success" (triage/spec/PR), "block" (Pipeline Block), "progress" (ostatní)

### 6. Generate recommendations

- `blocked > 0` → "{N} issues are blocked — review them and decide next steps"
- Agent blokuje > 30% → "{agent} blocks {X}% of issues — consider adjusting retry limits or decomposing complex tasks"
- `successRate < 50` → "Success rate is below 50% — review pipeline configuration"

- `blocked == 0` → "No blocked issues — pipeline is running smoothly"
