# Design: v1.6.0 — Rozšíření pipeline

**Datum:** 2026-02-25
**Status:** APPROVED (2026-02-25)

---

## Přehled

Tři oblasti rozšíření pipeline:

1. **Paralelní fix-bugs s worktrees** — `fix-bugs.md` má worktree logiku od v1.1.0, ale end-to-end flow nebyl nikdy ověřen. Identifikovány 4 konkrétní mezery.
2. **Multi-tracker support** — plugin je textově generický, ale `triage-analyst` a `publisher` nemají instrukci pro výběr správného MCP serveru dle typu trackeru.
3. **Rollback agent** — nový agent pro automatický revert při selhání buildu.

---

## Navrhovaný přístup

### 1. Paralelní fix-bugs s worktrees — E2E ověření

**Identifikované mezery:**

| # | Problém | Řešení |
|---|---------|--------|
| 1 | Chybí fallback, když Worktrees config není v Automation Config | Default: sekvenční CWD zpracování (explicitní opt-in pro worktrees) |
| 2 | Orphaned worktrees po mid-pipeline failure | Cleanup krok i pro blocked bugy, ne jen úspěšné |
| 3 | Hardcoded `.worktrees/` cesta | Čtení `base path` z Automation Config |
| 4 | Cleanup jen pro úspěšné bugy | Rozšířit na všechny bugy (success i blocked) |

**Změny v `commands/fix-bugs.md`:**
- Přidat explicitní fallback: "Pokud Worktrees sekce v Automation Config neexistuje, zpracuj bugy sekvenčně v CWD"
- Cleanup worktree po každém bugu (ne jen úspěšném)
- Čtení `base path` z config místo hardcoded hodnoty

### 2. Multi-tracker support

**Přístup:** Abstrakční vrstvou je MCP server — různé MCP servery pro různé trackery. Plugin nepotřebuje tracker-specific logiku.

**Nový povinný klíč v Automation Config:**

```markdown
### Issue Tracker
| Klíč | Hodnota |
|------|---------|
| Type | `youtrack` / `jira` / `github-issues` / `linear` |
| Instance | ... |
```

`Type` klíč říká agentům, který MCP tool prefix použít. Default: `youtrack` pro zpětnou kompatibilitu s projekty vytvořenými před v1.6.0.

**Dotčení agenti:**
- `triage-analyst` — čte `Type` pro správný MCP tool prefix
- `publisher` — čte `Type` pro správný MCP tool prefix při state changes

### 3. Rollback agent

**Nový soubor:** `agents/rollback-agent.md`

**Model:** haiku (mechanický task — `git reset --hard` + komentář)

**Kdy se volá:** Commands (`fix-bugs`, `fix-ticket`) ho volají po Block signálu od jakéhokoli agenta.

**Co dělá:**
1. V **worktree**: `git reset --hard {base_branch}` (bezpečné — izolovaný workspace)
2. V **CWD** (bez worktree): `git stash` + `git reset --hard {base_branch}` (stash chrání případnou uživatelskou práci)
3. Postne komentář do issue trackeru s failure reason

**Proč `git reset --hard` a ne `git revert`:** Branch nebyla v momentu selhání ještě pushnutá. Revert by vytvořil zbytečný commit.

**Proč volán commandy, ne agenty:** Orchestrace patří do command vrstvy (single-responsibility).

## Alternativy

### Multi-tracker

**A1: Auto-detekce typu trackeru z Instance URL** — nespolehlivé (custom domény). Zamítnuto.

**A2: Tracker-specific instrukce v agentech** — duplicitní logika per tracker per agent. Zamítnuto.

**A3: Explicitní `Type` klíč (přijato)** — jasné, konfigurovatelné, MCP server dělá abstrakci.

### Rollback

**A1: Rollback model = sonnet** — zbytečně drahé pro `git reset`. Zamítnuto.

**A2: Rollback integrovaný do fixeru** — porušuje single-responsibility. Zamítnuto.

**A3: Žádný rollback, branch se nechá** — dirty branch zůstane, uživatel musí ručně cleanup. Zamítnuto.

## Designová rozhodnutí

**DR-1: `Type` klíč je explicitní config, ne auto-detected.** Default `youtrack` pro backward compatibility.

**DR-2: Rollback model = haiku.** Mechanický: `git reset --hard` + komentář. Žádná analýza kódu.

**DR-3: Rollback volán commandy po Block signálu.** Single-responsibility — agenti neorchestrují.

**DR-4: `fix-bugs` bez Worktrees config = sekvenční CWD.** Bezpečný default, explicitní opt-in pro paralelismus.

**DR-5: `git reset --hard`, ne `git revert`.** Branches nebyly pushnuty v momentu selhání.

---

## Vyřešené otázky (2026-02-25)

**1. `Type` je MINOR, ne MAJOR.** Default `youtrack` zajišťuje backward compatibility. Stávající projekty bez `Type` klíče fungují dál.

**2. Rollback jen lokální state.** Remote cleanup je manuální. Force-delete remote branch by byl příliš destruktivní pro automatiku.

**3. Worktree cleanup při SIGINT/crash:** Dokumentovat ruční cleanup (`git worktree list` + `git worktree remove`). Automatická detekce orphaned worktrees je overkill pro v2.0.
