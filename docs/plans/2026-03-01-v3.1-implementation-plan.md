## Konvence

- **Nové commands** = nové `.md` soubory v `commands/`
- **Nový agent** = nový `.md` v `agents/`
- **Nové examples** = nový adresář `examples/` s podadresáři
- **Config sekce** = vždy tabulkový formát (`| Klíč | Hodnota |`)
- **Command frontmatter:** `description` (česky), `allowed-tools` (dle potřeby)
- **Agent frontmatter:** `name`, `description` (anglicky), `model`
- **Agent sekce:** Goal → Expertise → Process → Constraints (přesně v tomto pořadí)
- **Constraint pravidla:** musí začínat `NEVER` nebo definovat hard limit
- **Block Comment Template:** všechny agenty, které mohou selhat, musí používat standardní formát

### 3.1.4 — Metrics & Effectiveness

**Popis:** Nový command `/metrics` — sbírá data z issue trackeru + git, generuje markdown report s pipeline-level i per-agent analytics.

#### Soubory

| Akce | Cesta | Popis |
|------|-------|-------|
| CREATE | `commands/metrics.md` | Nový command |
| EDIT | `skills/bug-workflow/SKILL.md` | Přidat intent mapping pro `/metrics` |
| EDIT | `CLAUDE.md` | Přidat Metrics do Config Contract (optional sekce), přidat `/metrics` do Command Inventory |

#### Změny v `commands/metrics.md`

Nový command soubor. Struktura:

```yaml
---
description: Generuje pipeline analytics report — success rate, per-agent effectiveness, failure patterns
allowed-tools: mcp__*, Read, Glob, Grep, Bash
---
```

**Obsah:**

1. **Parsování flagů:** `$ARGUMENTS` = `--period <N days>` (default: 30), `--output <path>` (default: stdout), `--format <md|json>` (default: md)
2. **Konfigurace:** Načti z Automation Config: Issue Tracker (Type, Instance, Project, Bug query, State transitions), Source Control (Remote), volitelně Metrics sekce (Output, Period)
3. **Orchestrace:**
   - **Krok 1: Fetch issues** — přes MCP server (dle Type) načti všechny issues matching Bug query + Feature query (pokud existuje). Pro každý issue: ID, title, state, comments, created date.
   - **Krok 2: Parse `[CLAUDE-agents]` komentáře** — pro každý issue projdi komentáře. Extrahuj: triage checkpoints, spec checkpoints, block comments (agent, krok, důvod), PR links. Použij stejné regexy jako `commands/dashboard.md` krok 2.
   - **Krok 3: Fetch git data** — `git log --oneline --since="{period} days ago"` přes Bash. Parsuj commit messages pro issue ID references.
   - **Krok 4: Compute pipeline metrics:**
     - `total_attempted` = issues se `[CLAUDE-agents]` komentářem v daném období
     - `total_fixed` = issues s PR merged (nebo ve stavu z State transitions → For Review/Done)
     - `total_blocked` = issues s `[CLAUDE-agents] 🔴 Pipeline Block` komentářem a stále ve stavu Blocked
     - `success_rate` = total_fixed / total_attempted (procento)
     - `avg_time_to_fix` = průměrný čas od prvního `[CLAUDE-agents]` komentáře po merged PR (pokud data dostupná)
     - `block_by_stage` = count bloků per pipeline stage (triage, analysis, fix, review, test, publish)
     - `top_block_reasons` = top 5 nejčastějších "Důvod:" z block komentářů
   - **Krok 5: Compute per-agent metrics:**
     - Pro každý agent (triage-analyst, code-analyst, fixer, reviewer, test-engineer, e2e-test-engineer, publisher): count blocks, success rate, nejčastější důvod selhání
     - `avg_fixer_iterations` = z reviewer komentářů (kolik REQUEST_CHANGES před APPROVE)
     - `failure_pattern_detection` = pokud agent blokuje > 30% issues → flag pattern s top důvodem
   - **Krok 6: Token cost estimate:**
     - Per-issue estimate: count stages × model tokens (sonnet ~30k, opus ~50k, haiku ~5k per invocation)
     - Total estimate pro období
   - **Krok 7: Generate report:**
     - Pokud `--format md`:
       ```
       ## Pipeline Metrics Report — {project} ({period} days)

       ### Pipeline Overview
       | Metrika | Hodnota |
       |---------|---------|
       | Issues attempted | {N} |
       | Issues fixed | {N} ({rate}%) |
       | Issues blocked | {N} |
       | Avg time to fix | {N} hours |
       | Est. token cost | ~{N}k tokens (~${N} USD) |

       ### Block Analysis
       | Stage | Blocks | % of total |
       |-------|--------|------------|
       | triage | ... | ... |
       | ... | ... | ... |

       Top block reasons:
       1. {reason} ({N} occurrences)
       ...

       ### Per-Agent Effectiveness
       | Agent | Invocations | Blocks | Success Rate | Top Failure |
       |-------|-------------|--------|-------------|-------------|
       | fixer | ... | ... | ... | ... |
       | ... | ... | ... | ... | ... |

       ### Recommendations
       - {threshold-based recommendations}

       ---
       Generated: {timestamp} | CLAUDE-agents v{version}
       ```
     - Pokud `--format json`: strukturovaný JSON se stejnými daty
     - Pokud `--output` specifikován: zapiš do souboru, jinak stdout
4. **Pravidla:**
   - Read-only — žádné změny v issue trackeru ani gitu
   - Data se čtou přes MCP servery
   - Pokud MCP nedostupný → report error s vysvětlením
   - Threshold pro recommendations: block_rate > 30% per agent, success_rate < 50%, single agent > 50% bloků

## Worktree zpracování

### Varianta A: Paralelní (Worktrees config existuje)

Načti z Automation Config → Worktrees:
- `batch_size` (default: 3)
- `base_path` (default: `.worktrees/`)
- `cleanup` (default: `auto`)

**Příprava worktrees:**
1. Pro každý bug v aktuálním batchi (max batch_size):
   ```bash
   git worktree add {base_path}/{issue_id} -b {branch_naming_pattern} {base_branch}
   ```
2. Pokud worktree creation selže → zaznamenej error, zkus další bug

**Paralelní spuštění:**
3. Spusť pipeline (kroky 3–8) pro KAŽDÝ bug v batchi PARALELNĚ:
   - Použij Task tool pro každý bug — všechny Task volání v JEDNOM message bloku
   - Každý Task dostane kontext: worktree path, issue ID, kompletní Automation Config
   - Každý Task pracuje ve svém worktree (cd do worktree path jako první krok)

**Result collection:**
4. Po dokončení všech paralelních Tasks:
   - Sbírej výsledky: pro každý bug status (FIXED/BLOCKED/DUPLICATE), PR URL (pokud FIXED), block reason (pokud BLOCKED)
   - Pokud bug BLOCKED → spusť rollback-agent v jeho worktree

**Cleanup:**
5. Pro KAŽDÝ worktree (úspěšný i neúspěšný):
   ```bash
   git worktree remove {base_path}/{issue_id} --force
   ```
   - Pokud `cleanup = manual` → přeskoč cleanup, zobraz seznam worktrees
   - Pokud `cleanup = auto` (default) → cleanup vždy

**Batching:**
6. Pokud bugů > batch_size:
   - Zpracuj první batch (kroky 1–5)
   - Po dokončení a cleanup spusť další batch
   - Opakuj dokud nejsou všechny bugy zpracovány

### Varianta B: Sekvenční (Worktrees config neexistuje)

Zpracuj bugy jeden po druhém v CWD. Žádné worktrees.
```

2. **V Summary sekci** (krok 9) přidat sloupec "Worktree":

```

1. **Parsování:** `$ARGUMENTS` = Issue ID (povinné) + volitelné `--profile <name>`

     - Per-stage token costs (heuristika):
       ```
       triage-analyst (sonnet):    ~15k input + ~5k output

       publisher (haiku):          ~10k input + ~3k output
       ```
     - Iteration multiplier: fixer↔reviewer (best: 1, typical: 2, worst: max z Retry Limits)
     - Decomposition multiplier: pokud estimated affected files >= 4 → decomposition likely → multiply fixer/reviewer/test by estimated subtask count
   - **Krok 5: Range calculation:**
     - **Best case:** 1 fixer iteration, no decomposition, all stages pass first try
     - **Typical case:** 2 fixer iterations, no decomposition
     - **Worst case:** max fixer iterations + decomposition (max subtasks) + max test retries
   - **Krok 6: Cost calculation:**
     - Sonnet: $3/MTok input, $15/MTok output
     - Opus: $15/MTok input, $75/MTok output
     - Haiku: $0.80/MTok input, $4/MTok output
   - **Krok 7: Check historical data** — pokud `/metrics` report existuje (z 3.1.4), přečti avg_fixer_iterations a success_rate. Upřesni typical estimate.
   - **Krok 8: Output:**
     ```
     ## Cost Estimate — {issue_id}: {issue_title}

     **Estimated complexity:** {XS/S/M/L} ({affected_files} files, ~{loc} LOC)
     **Pipeline:** {full | profile: {name}} ({N} stages)

     | Scenario | Tokens (input) | Tokens (output) | Est. Cost |
     |----------|---------------|-----------------|-----------|
     | Best case | ~{N}k | ~{N}k | ~${min} |
     | Typical | ~{N}k | ~{N}k | ~${mid} |
     | Worst case | ~{N}k | ~{N}k | ~${max} |

     ```
4. **Pravidla:**
   - Read-only — žádné side effects
   - Pokud metrics data dostupná → mention "Calibrated with historical data"

Nový agent soubor:

```yaml
---
name: priority-engine
description: Analyzes backlog and recommends fix order based on impact, risk, effort, and dependencies.
model: opus
---
```

```markdown
You are a Backlog Analyst specializing in cross-issue prioritization.

## Goal

Analyze an entire bug/feature backlog and produce a ranked list with recommended fix order, based on impact, risk, effort, and inter-issue dependencies.

## Expertise

Impact assessment, risk analysis, effort estimation, dependency graph construction, cost-benefit optimization.

## Process

1. Receive the list of open issues (ID, title, description, state, labels, comments)
2. For each issue, assess four dimensions:
   a. **Impact** (1-5): How many users/modules does this affect? Labels like "critical", "blocker" increase score. Issues with many duplicates increase score.
   b. **Risk** (1-5): How critical is the affected code area? Core business logic = 5, cosmetic = 1. If historical data available (from metrics or [CLAUDE-agents] comments), factor in: area with recurring bugs = higher risk.
   c. **Effort** (1-5): Estimated implementation complexity. 1 = trivial fix (typo, config), 5 = multi-file refactoring. Use issue description length, affected area size, and any prior analysis as signals.
   d. **Dependencies** (list): Does this issue block or depend on other issues? Use issue links, mentions, and shared code areas.
3. Calculate priority score: `score = (Impact × 2 + Risk × 1.5) / (Effort × 1) + dependency_bonus`
   - `dependency_bonus` = +2 if issue blocks 2+ other issues, +1 if blocks 1 issue
4. Sort by score descending
5. Group into tiers:
   - **P0 (Fix Now):** score >= 8, or labeled critical/blocker
   - **P1 (Fix Next):** score >= 5
   - **P2 (Backlog):** score < 5
6. Output:

   ```markdown
   ## Backlog Prioritization

   ### P0 — Fix Now ({N} issues)
   | # | Issue | Impact | Risk | Effort | Score | Rationale |
   |---|-------|--------|------|--------|-------|-----------|
   | 1 | {ID}: {title} | {N}/5 | {N}/5 | {N}/5 | {score} | {1 sentence} |

   ### P1 — Fix Next ({N} issues)
   | # | Issue | Impact | Risk | Effort | Score | Rationale |
   ...

   ### P2 — Backlog ({N} issues)
   | # | Issue | Impact | Risk | Effort | Score | Rationale |
   ...

   ### Dependencies
   {issue_A} → blocks → {issue_B}
   ...

   ### Recommendations
   - Suggested batch: {top N issues for next /fix-bugs run}
   - Estimated cost for batch: ~${min}-${max} (if estimate data available)
   ```

## Constraints

- NEVER modify issues — read-only analysis
- Max 50 issues per analysis — if backlog larger, prioritize only the first 50 (sorted by creation date) and note the limitation
- If issue description is too vague to assess → assign Effort = 3 (medium) and note "insufficient data"
- Score formula is fixed and transparent — always show the formula and per-dimension scores so results are auditable. Note: dimension scores (Impact, Risk, Effort) are assessed by reasoning and may vary between runs
- On failure: report what was analyzed so far, use Block Comment Template
```

#### Test Runner (`tests/harness/run-tests.sh`)

```bash
#!/bin/bash
# Run all test scenarios
# Usage: ./tests/harness/run-tests.sh [scenario-name]

SCENARIOS_DIR="$(dirname "$0")/../scenarios"
RESULTS=()

# If specific scenario provided, run only that
if [ -n "$1" ]; then
  bash "$SCENARIOS_DIR/$1.sh"
  exit $?
fi

# Run all scenarios
for scenario in "$SCENARIOS_DIR"/*.sh; do
  echo "Running: $(basename "$scenario")..."
  if bash "$scenario"; then
    RESULTS+=("PASS: $(basename "$scenario")")
  else
    RESULTS+=("FAIL: $(basename "$scenario")")
  fi
done

# Summary
echo ""
echo "=== Test Results ==="
for result in "${RESULTS[@]}"; do
  echo "$result"
done
```
