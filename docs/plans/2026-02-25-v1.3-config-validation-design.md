# Design: v1.3.0 — Config validace

**Datum:** 2026-02-25
**Status:** APPROVED (2026-02-25)
**Roadmap items:**
- Aktualizace Automation Config příkladu podle reálných projektů
- `/check-setup` command — validace Automation Config, MCP serverů a tokenů

---

## Přehled

Onboarding nového projektu na CLAUDE-agents vyžaduje správně vyplněný `## Automation Config` v CLAUDE.md, funkční `.mcp.json` se všemi MCP servery a tokeny. Dnes neexistuje způsob, jak tuto konfiguraci ověřit dříve, než se pokusíme spustit pipeline. Chyba se projeví až uprostřed `/fix-bugs` — typicky po minutách práce agenta.

**Cíl v1.3.0:**

1. **`/check-setup` command** — validuje celou konfiguraci projektu a dá uživateli jednoznačný report: co funguje, co chybí, co selhalo.
2. **Aktualizace Automation Config příkladu v README** — příklad v README je dnes příliš abstraktní (placeholder hodnoty). Nahradíme ho realistickým příkladem inspirovaným reálnými projekty.

---

## Navrhovaný přístup

### `/check-setup` command

Command je markdown soubor `commands/check-setup.md`. Instrukuje Claude, co má zkontrolovat a jak výsledek prezentovat. Neobsahuje žádnou projektově specifickou logiku — veškeré hodnoty čte z Automation Config.

#### Co command validuje

**Blok 1: Automation Config (strukturální kontrola)**

Ověřuje, že `## Automation Config` v CLAUDE.md projektu existuje a obsahuje všechny povinné sekce a klíče.

| Co se kontroluje | Jak |
|-----------------|-----|
| Sekce `## Automation Config` existuje v CLAUDE.md | Read CLAUDE.md, najít sekci |
| `### Issue Tracker` — přítomnost klíčů: Instance, Project, Bug query, State transitions, On start set | Přečíst sekci, ověřit, že žádný klíč není prázdný nebo placeholder (`<...>`) |
| `### Source Control` — Remote, Base branch, Branch naming pattern | Totéž |
| `### PR Rules` — Labels, PR description template | Totéž |
| `### Build & Test` — Build command, Test command | Totéž |

Placeholder hodnoty (`<query pro otevřené bugy>`, `<build příkaz>`) jsou ekvivalent chybějící hodnoty — validace je označí jako FAIL.

**Blok 2: MCP servery (přítomnost a konektivita)**

| Co se kontroluje | Jak |
|-----------------|-----|
| `.mcp.json` existuje v projektu | Read `.mcp.json` |
| MCP server pro issue tracker nakonfigurován | Porovnat server URLs/names v `.mcp.json` s hodnotou Instance z Automation Config |
| MCP server pro source control nakonfigurován | Porovnat s hodnotou Remote z Automation Config |
| Tokeny nastaveny (ne prázdné, ne placeholder) | Zkontrolovat env/token pole v `.mcp.json` |

Konektivita: command se pokusí provést jednoduchý read-only MCP dotaz — spustí Bug query z Automation Config (jeden výsledek stačí). Pokud MCP call selže, zaloguje konkrétní chybu (timeout, auth error, not found).

**Blok 3: Build & Test příkazy (dry-run)**

Command spustí Build command a Test command z Automation Config (přes Bash). Cílem je ověřit, že příkazy existují a jsou spustitelné. Přeskočitelné flagem `--skip-build`.

#### Formát výstupu

```
## Setup report — <název projektu / Remote z Automation Config>

### Automation Config
[OK]   ## Automation Config nalezeno v CLAUDE.md
[OK]   Issue Tracker — všechny klíče vyplněny
[OK]   Source Control — všechny klíče vyplněny
[WARN] PR Rules — chybí PR description template (volitelné)
[FAIL] Build & Test — Test command je prázdný

### MCP servery
[OK]   .mcp.json nalezeno
[OK]   Issue tracker MCP server nakonfigurován (<instance>)
[FAIL] Source control MCP server nenalezen pro remote <owner/repo>

### Konektivita
[OK]   Issue tracker — připojení OK, projekt <PROJECT> nalezen, X bugů nalezeno
[FAIL] Source control — autentizace selhala (401 Unauthorized)

### Build & Test
[SKIP] Build přeskočen (--skip-build)

---
Výsledek: 2 FAIL, 1 WARN — pipeline NELZE spustit.
```

Stavy: `[OK]` / `[WARN]` (volitelné, pipeline funguje) / `[FAIL]` (kritické, pipeline selže) / `[SKIP]` (přeskočeno).

Souhrnný verdict:
- Žádné FAIL → "Konfigurace je kompletní. Pipeline je připravena."
- Aspoň jeden FAIL → "Pipeline NELZE spustit. Oprav výše uvedené chyby."

#### Frontmatter

```
allowed-tools: mcp__*, Read, Glob, Grep, Bash
```

Bash je nutný pro dry-run build/test. `mcp__*` pro read-only konektivitu.

#### Argumenty

`/check-setup` — bez povinných argumentů. Volitelný `--skip-build` přeskočí spuštění build/test příkazů.

### Aktualizace Automation Config příkladu v README

Stávající příklad v README.md obsahuje placeholder hodnoty — nedostatečné pro onboarding. Nahradíme ho kompletním příkladem se všemi sekcemi (povinné i volitelné) a realistickými hodnotami.

Konkrétní zlepšení:

- **Bug query** — ukázat reálný formát filtrační podmínky
- **State transitions** — strukturovat jako přechody (On start, Blocked, For Review)
- **Branch naming** — ukázat placeholder syntax s proměnnými (`{PROJECT}-{issue-id}-{short-description}`)
- **PR description template** — vyčlenit jako samostatnou podsekci
- **Build & Test** — ponechat generické, ale přidat komentář o formátu
- **Worktrees (volitelné)** — přidat příklad sekce
- **E2E Test (volitelné)** — přidat příklad
- **Error Handling (volitelné)** — přidat příklad

## Alternativy

### Alt 1: Validace jako agent místo command

`check-setup` jako agent (validator-agent) volaný z command. Zamítnuto: validace je jednorázová akce bez iterace. Command je jednodušší.

### Alt 2: Pouze strukturální kontrola (bez konektivity)

Jen přítomnost klíčů v CLAUDE.md, bez MCP dotazů a build. Zamítnuto: nejčastější chyby jsou špatné tokeny a neexistující projekty v issue trackeru, ne překlepy v CLAUDE.md. Konektivita je nezbytná část.

### Alt 3: Interaktivní průvodce (wizard)

Command by procházel konfiguraci interaktivně a zapisoval do CLAUDE.md. Toto je cíl pro v1.9.0 `/onboard`. `/check-setup` je read-only — jen reportuje, nezasahuje.

### Alt 4: Inline validace v každém command

Každý command by na začátku ověřil config. Zamítnuto: duplicita across commands, obtížná údržba.

## Designová rozhodnutí

**1. Read-only konektivita.** `/check-setup` provádí pouze čtecí MCP dotazy. Nikdy nezapisuje. Bezpečný pro opakované spouštění.

**2. Checklist formát.** `[OK]`/`[WARN]`/`[FAIL]` je skenovátelný. Uživatel okamžitě vidí, co konkrétně selhalo.

**3. Build dry-run je volitelný.** Build může trvat minuty nebo vyžadovat prostředí. Default je spustit, ale `--skip-build` je legitimní volba.

**4. Detekce placeholder hodnot.** Vzor `<...>` v hodnotách Automation Config = FAIL. Příklad v README používá tento vzor — command ho explicitně detekuje.

**5. Souhrnný verdikt.** Explicitní "pipeline ready" nebo "nelze spustit" na konci.

**6. Kompletní příklad v README.** "Minimální příklad" s placeholdery je matoucí. Nový příklad ukáže všechny sekce s realistickými (ale generickými) hodnotami.

## Vyřešené otázky (2026-02-25)

**1. `--skip-build` flag:** Substring check v `$ARGUMENTS` je dostatečný. Konzistentní s konvencí všech commands.

**2. Hloubka konektivity:** Bug query s limit 1 výsledek. Ověří auth + projekt + query syntax najednou. Optimální poměr spolehlivosti a rychlosti.

**3. Parsování `.mcp.json`:** Command matchuje servery dle URL/env proměnných v `.mcp.json`. Explicitní klíč `MCP server name` v Automation Config není potřeba — jednodušší pro uživatele.

**4. PR description template:** Vyčlenit jako samostatnou podsekci `### PR Description Template` pod `### PR Rules`. Víceřádková šablona v tabulkové buňce je nečitelná.
