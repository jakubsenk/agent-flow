# Design: v1.7.0 — Spolehlivost

**Datum:** 2026-02-25
**Status:** APPROVED (2026-02-25)

---

## Přehled

Tři oblasti pro zvýšení spolehlivosti pipeline:

1. **E2E test suite** — mock projekt pro ověření celé pipeline
2. **Konfigurovatelné retry limity** — aktuálně hardcoded v agentech/commandech
3. **Strukturovaný error reporting** — standardizovaný formát Block komentářů

---

## Audit: Kde jsou retry limity hardcoded

| Soubor | Co | Hodnota |
|--------|-----|---------|
| `agents/fixer.md` | Build retries | max 3 attempts |

## Navrhovaný přístup

### 1. E2E test suite

**Mock projekt:** Minimální repozitář v `tests/mock-project/` obsahující:
- `CLAUDE.md` s `## Automation Config` (mock hodnoty)
- Jednoduchý Python soubor s úmyslným bugem
- `tests/` s testem, který ověří opravu

**Test scénáře (4, manuální):**

| # | Scénář | Ověřuje |
|---|--------|---------|
| 1 | Happy path | Celá pipeline: triage → fix → review → test → publish |
| 2 | Triage block | Triage-analyst detekuje duplicitu → Block |
| 3 | Build-fail block | Fixer produkuje kód, který neprojde buildem po 3 pokusech → Block |
| 4 | Dry-run | Pipeline běží jen přes triage + code-analyst → report |

**Proč manuální, ne automatické:** Automatizace by vyžadovala mock MCP servery (simulovat YouTrack, Gitea) — to je mimo scope. Manuální test s reálnými MCP servery (na test projektu) je dostačující a ověří skutečné chování.

**Nové soubory:**
- `tests/mock-project/CLAUDE.md`
- `tests/mock-project/app.py` (s bugem)
- `tests/mock-project/tests/test_app.py`
- `tests/README.md` (popis test scénářů a jak je spustit)

### 2. Konfigurovatelné retry limity

**Nová volitelná sekce v Automation Config:**

```markdown
### Retry Limits (optional)
| Klíč | Hodnota | Default |
|------|---------|---------|
| Fixer iterations | 5 | 5 |
| Test attempts | 3 | 3 |
| Build retries | 3 | 3 |
```

**Jak to funguje:**
- Commands (`fix-ticket.md`, `fix-bugs.md`) čtou limity z Automation Config
- Pokud sekce neexistuje → použijí se stávající defaulty (plná backward compatibility)
- Commands předají limity agentům jako kontextovou instrukci

**Změny v agentech:**
Agenti si ponechají hardcoded defaulty, ale commands je overridují hodnotou z config:

```
Spusť CLAUDE-agents:fixer. Kontext: Max build retries = {hodnota z config nebo 3}.
```

Agent čte limit z kontextu, ne z vlastní definice. Tím se agenti nemusí měnit — změna je pouze v commandech.

### 3. Strukturovaný error reporting

**Problém:** Každý agent má "On failure: set issue to Blocked with explanation" — free-form, nekonzistentní formát.

**Řešení:** Standardizovaný Block Comment Template:

```
## 🔴 Pipeline Block

**Agent:** {název agenta}
**Krok:** {krok kde selhalo}
**Důvod:** {max 2 věty}
**Detail:** {technický output — chybová hláška, diff, test output}
**Doporučení:** {co by měl člověk udělat}
```

**Jak template dostat ke všem agentům:**
- Definovat template v `CLAUDE.md` pluginu (sekce "Block Comment Template")
- Každý agent odkazuje na template: "Při blocku použij formát z Block Comment Template v CLAUDE.md"
- Agenti neobsahují template inline — single source of truth

**Alternativně:** Template v Automation Config (projekt-specifická customizace). Ale standardní formát je dostatečný — customizace přidává zbytečnou komplexitu.

## Alternativy

### E2E testy

**A1: Automatické testy s mock MCP** — Vyšší spolehlivost, ale enormní scope (mock YouTrack API, mock Gitea API). Zamítnuto pro v1.7.0.

**A2: Testy na reálném projektu** — Funguje, ale riskantní (vytváří skutečné issues, branches, PR). Zamítnuto jako default test metoda.

**A3: Mock projekt s reálnými MCP servery (přijato)** — Kompromis. Test projekt na reálné Gitea instanci s test issues.

### Retry limity

**A1: Limity přímo v agent definicích** — Projekt by musel forkovat agenty. Zamítnuto.

**A2: Limity v Automation Config, agenti čtou přímo** — Agenti by museli sami parsovat Automation Config. Zamítnuto — agenti nemají přístup k celému CLAUDE.md.

**A3: Commands overridují (přijato)** — Commands čtou config, předávají agentům jako context. Agenti nepotřebují změnu.

### Error reporting

**A1: JSON structured output** — Strojově čitelné, ale nepřátelské pro čtení v issue trackeru. Zamítnuto.

**A2: Free-form (status quo)** — Nekonzistentní, chybí klíčové informace. Zamítnuto.

**A3: Template s definovanými poli (přijato)** — Lidsky čitelný, konzistentní, obsahuje vše potřebné.

Agenti se **nemění** — template reference a retry override přicházejí z command context.

---

## Designová rozhodnutí

**1. Manuální testy, ne automatické.** Mock MCP servery jsou mimo scope. Reálné MCP + test projekt = pragmatický kompromis.

**2. Retry override přes command context.** Agenti nemají přístup k Automation Config — commands jim limity předávají. Single source of truth v config.

**3. Block Comment Template v CLAUDE.md pluginu.** Standardní pro všechny projekty. Customizace přes Automation Config je YAGNI.

**4. Template je lidsky čitelný markdown.** Konzumenti jsou vývojáři čtoucí issue komentáře — ne stroje.

---

## Vyřešené otázky (2026-02-25)

**1. Mock projekt — jaký bug?** Implementační detail. Rozhodne se při psaní kódu.

**2. Formát předávání retry limitů:** Jednoduchá forma: `Kontext: Fixer iterations = 5, Test attempts = 3, Build retries = 3.` Jasné, parsovatelné LLM.

**3. Block Comment Template — emoji 🔴:** Ano. YouTrack, Jira, GitHub Issues všechny renderují emoji. Plain-text fallback akceptovatelný.

**4. Cross-platform mock projekt:** Implementační detail. Python je cross-platform, není co řešit.

**Upřesnění scope mock projektu:** Slouží jako smoke test pro pipeline logiku (čtení config, volání agentů ve správném pořadí, retry limity, block template formát) **bez závislosti na reálném YouTrack/Gitea**. Neověřuje MCP konektivitu — ta se testuje na reálném projektu.
