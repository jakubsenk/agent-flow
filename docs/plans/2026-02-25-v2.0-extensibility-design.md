# Design: v2.0.0 — Rozšiřitelnost (Hooks + Custom Agenti)

**Datum:** 2026-02-25
**Status:** APPROVED (2026-02-25)
**Verze pluginu:** 2.0.0

---

## Přehled

Aktuální pipeline je uzavřená: pevně daná sekvence agentů, žádné vstupní body pro projekt-specifickou logiku. Projekty s nestandardními požadavky (lint před fixem, security scan po fixu, specializovaný agent pro migraci databáze) nemohou pipeline rozšiřovat bez forku pluginu.

**Co řešíme:**

1. **Pre/post hooks** — projekty mohou vložit shell příkazy před nebo po vybraných pipeline krocích
2. **Custom agenti** — projekty mohou přidat vlastní agent soubory a zaradit je do pipeline na definovaném místě

**Proč 2.0.0:** Obě funkce rozšiřují Config Contract — major bump kvůli rozšíření, i když je zpětně kompatibilní (obě sekce jsou volitelné).

---

## Navrhovaný přístup

### 1. Pre/post hooks

#### Kde jsou definovány

V `## Automation Config`, nová sekce `Hooks`, klíč-hodnota formát:

```markdown
### Hooks

- Pre-fix: npm run lint
- Post-fix: npm run security-scan
- Pre-publish: npm run changelog
- Post-publish: npm run notify-team
```

#### Podporované hooky — pouze čtyři

| Hook | Kdy se spustí | Typické použití |
|------|--------------|-----------------|
| `Pre-fix` | Po code-analyst, před fixer | Lint, statická analýza — fail fast před drahým opus voláním |
| `Post-fix` | Po fixer + build, před reviewer | Security scan, dependency check |
| `Pre-publish` | Po test-engineer / e2e-test-engineer, před publisher | Generování changelog, kontrola verzí |
| `Post-publish` | Po publisher (PR vytvořen) | Notifikace, metriky, deploy preview |

#### Chování při selhání

- Pre-fix / Post-fix / Pre-publish selže → **Block** (issue komentář, pokračuj dalším bugem)
- Post-publish selže → **Varování** (PR už je vytvořen, nelze rollbackovat)

### 2. Custom agenti

#### Filosofie: Reference, ne auto-discovery

Custom agenti musí být explicitně jmenováni v Automation Config. Pipeline je automaticky nedetekuje.

#### Kde custom agent žije

V projektu uživatele, v adresáři `agents/`, standardní CLAUDE-agents formát:

```markdown
---
name: migration-reviewer
description: Reviewuje databázové migrace
model: opus
---

You are a Database Migration Reviewer...

## Goal
## Expertise
## Process
## Constraints
```

#### Registrace v Automation Config

```markdown
### Custom Agents

- Post-fix agent: agents/migration-reviewer.md
- Pre-publish agent: agents/compliance-checker.md
```

#### Dva insertion points

| Insertion point | Kdy | Typické použití |
|----------------|-----|-----------------|
| `Post-fix agent` | Po fixer + build, před reviewer | DB migrace, API breaking changes |
| `Pre-publish agent` | Po test-engineer, před publisher | Compliance check, licence scan |

Výchozí agenti (fixer, reviewer, publisher) nelze nahradit — garance pluginu musí zůstat.

#### Custom agent je one-shot gate

Custom agent běží jednou. Pokud vrátí BLOCK, fixer nedostane šanci opravit. Záměrné — custom agenti jsou jednosměrné quality gates.

## Alternativy

### Hooks

**A1: Hook jako agent místo shell příkazu** — Zamítnuto: zbytečné pro lint/scan; stírá hranici mezi hooks a custom agenty.

**A2: Pouze post-hooky** — Zamítnuto: `Pre-fix` (fail fast před opus) je nejpoptávanější use case.

**A3: Hooky v CI/CD** — Zamítnuto: plugin by nemohl blokovat před publish.

### Custom agenti

**B1: Auto-discovery ze složky `agents/`** — Zamítnuto: nepředvídatelné.

**B2: Custom agent jako override výchozího** — Zamítnuto: ztrácejí se garance pluginu.

**B3: Inline definice v Automation Config** — Zamítnuto: CLAUDE.md se stane neúnosně dlouhý.

## Nové / upravené soubory

### V CLAUDE-agents repozitáři

| Soubor | Změna |
|--------|-------|
| `commands/fix-ticket.md` | Přidat kroky pro hooky a custom agenty |
| `commands/fix-bugs.md` | Totéž; hooky i agenti per-worktree |
| `CLAUDE.md` | Config Contract: dokumentace `Hooks` a `Custom Agents` sekcí |

Žádné nové soubory v pluginu.

### V projektu uživatele (jeho zodpovědnost)

| Co přidá | Popis |
|----------|-------|
| `agents/<name>.md` | Definice custom agenta |
| `## Automation Config → Hooks` | Shell příkazy pro hooky |
| `## Automation Config → Custom Agents` | Reference na custom agent soubory |

## Designová rozhodnutí

**#1: Klíč-hodnota formát pro hooky** — odpovídá stylu ostatních sekcí Automation Config.

**#2: Bez timeout konfigurace** — YAGNI. Bash tool má vlastní timeouty.

**#3: Explicitní registrace custom agentů** — odpovídá principu "Commands contain zero project-specific logic".

**#4: Pouze dva insertion points** — přirozené hranice. Insertion pointy dovnitř reviewer smyčky = nepřiměřená komplexita.

**#5: Hooks a custom agenti jsou ortogonální** — Hooks = shell příkazy. Custom agenti = AI operace s kontextem.

**#6: Post-publish hook je pouze Varování** — PR už existuje, rollback by znamenal smazat PR.

---

## Vyřešené otázky (2026-02-25)

**OQ-1: Pořadí Post-fix hook vs Post-fix custom agent:** Hook před agentem (mechanická validace před AI analýzou).

**OQ-2: Custom agent v reviewer smyčce:** Ne v 2.0. Příliš komplexní.

**OQ-3: Validace custom agentů:** Ne. Fail gracefully s Block.

**OQ-4: analyze-bug command:** Beze změny (read-only, hooks nedávají smysl).
