# Design: v1.5.0 — Pipeline Reporting (Dry-Run + Summary)

**Datum:** 2026-02-25
**Status:** APPROVED (2026-02-25)

---

## Přehled

### Co

Dvě nové funkce pro zvýšení transparentnosti pipeline:

1. **Dry-run mód** — pipeline proběhne jen přes triage + code-analyst a vygeneruje report per issue (závažnost, dotčené soubory, odhadovaná složitost). Žádné změny kódu, žádné issue tracker state changes, žádné worktrees.

2. **Souhrnný report** — na konci `/fix-bugs` se zobrazí strukturovaná tabulka: co bylo opraveno, co zablokováno a proč.

### Proč

**Dry-run** řeší reálný problém: před spuštěním celé pipeline (která může trvat desítky minut a ovlivnit issue tracker + git) chce uživatel vědět, co ho čeká. Kolik bugů je triviálních, kolik vysokorizikových, které jsou duplicity.

**Souhrnný report** řeší chybějící uzávěr pipeline. Aktuálně `/fix-bugs` tiše skončí, uživatel musí sám procházet issue tracker nebo git log, aby zjistil výsledek.

---

## Navrhovaný přístup

### Dry-run mód

#### Mechanismus předávání příznaku

Dry-run se implementuje jako **flag na existujících commands**, ne jako samostatný command.

Syntaxe:
```
/CLAUDE-agents:fix-bugs 5 --dry-run
/CLAUDE-agents:fix-ticket PROJ-123 --dry-run
```

Commands v Claude Code přijímají `$ARGUMENTS` jako prostý string. Přidáme explicitní instrukci do každého dotčeného commandu:

```
Pokud $ARGUMENTS obsahuje `--dry-run`, aktivuj dry-run mód (viz sekce Dry-run níže).
```

`analyze-bug` dry-run nepotřebuje — je již read-only analýzou.

#### Co dry-run dělá (fix-bugs)

1. **Načti bugy** — stejné jako normálně (limit z `$ARGUMENTS`)
2. **Triage (paralelně)** — spusť `triage-analyst` pro každý bug, ale:
   - Duplicity → zaznamená jako `DUPLICATE`, neprovede close v issue trackeru
   - Nejasné → zaznamená jako `UNCLEAR`, neprovede Block v issue trackeru
   - Žádné issue tracker state changes v dry-run módu
3. **Analýza (paralelně)** — pro bugy, které prošly triage, spusť `code-analyst`
   - Žádné worktrees se nevytváří — code-analyst pracuje read-only v CWD
4. **Vygeneruj dry-run report** — viz formát níže
5. **Konec** — žádný fixer, reviewer, test-engineer, publisher

#### Co dry-run dělá (fix-ticket)

Originální kroky fix-ticket.md a jejich stav v dry-run:

1. Nastav issue tracker — **PŘESKOČIT** (dry-run nesmí měnit stav)
2. Vytvoř branch — **PŘESKOČIT**
3. triage-analyst — **SPUSTIT** (ale nezapisovat do issue trackeru)
4. code-analyst — **SPUSTIT**
5. fixer — **PŘESKOČIT**
6. build — **PŘESKOČIT**
7. reviewer — **PŘESKOČIT**
8. test-engineer — **PŘESKOČIT**
9. výsledek — **NAHRADIT** dry-run reportem per ticket

#### Formát dry-run reportu

```
## Dry-Run Report — {N} bugů analyzováno

| Bug ID | Summary | Triage | Severity | Affected Files | Risk | Est. Složitost |
|--------|---------|--------|----------|----------------|------|----------------|
| PROJ-1 | Login fails on ... | OK | HIGH | auth/login.ts, auth/session.ts | MEDIUM | S (≤20 řádků) |
| PROJ-2 | Button not visible | OK | LOW | ui/button.tsx | LOW | XS (≤5 řádků) |
| PROJ-3 | ... | DUPLICATE of PROJ-1 | — | — | — | — |
| PROJ-4 | Crash on load | UNCLEAR | — | — | — | — |

### Souhrn
- Připraveno k fixu: 2 bugů
- Duplicity: 1 (PROJ-3 → PROJ-1)
- Nejasné / k dořešení: 1 (PROJ-4)

### Odhadovaná složitost
Celkem: 1× XS, 1× S
Bez HIGH RISK bloků — pipeline by pravděpodobně proběhla bez komplikací.
```

Stupnice složitosti:

| Stupeň | Popis |
|--------|-------|
| XS | ≤5 řádků, 1 soubor, LOW risk |
| S | ≤20 řádků, 1–2 soubory, LOW/MEDIUM risk |
| M | ≤100 řádků (fixer limit), 3–5 souborů, MEDIUM risk |
| L | Potenciálně přes 100 řádků nebo HIGH risk — pipeline může selhat |

### Souhrnný report (Summary) po fix-bugs

Aktuální krok 6 v `fix-bugs.md` se nahradí detailnější instrukcí:

```
### Pipeline Summary

| Bug ID | Summary | Status | PR | Blokující důvod |
|--------|---------|--------|----|-----------------|
| ... | ... | FIXED | #42 | — |
| ... | ... | BLOCKED | — | Build selhal po 3 pokusech |
| ... | ... | BLOCKED | — | Reviewer: 5× REQUEST_CHANGES |
| ... | ... | DUPLICATE | — | Uzavřeno jako duplikát PROJ-1 |

**Výsledek:** {N_fixed} opraveno, {N_blocked} zablokováno, {N_dup} duplicit
```

Status hodnoty: `FIXED`, `BLOCKED`, `DUPLICATE`, `SKIPPED`.

## Alternativy

### A1: Dry-run jako samostatný command `/dry-run`

**Výhoda:** Čistá separace, vlastní `allowed-tools` (jen read-only).
**Nevýhoda:** Duplicitní orchestrační logika — dry-run a normální `/fix-bugs` sdílejí 80 % kódu. Při změně pipeline nutné aktualizovat dva soubory.
**Zamítnuto.** Hlavní riziko je desynchronizace.

### A2: Dry-run jako rozšíření `analyze-bug` (batch mode)

**Nevýhoda:** analyze-bug nemá batch koordinaci ani integraci s bug query.
**Zamítnuto.**

### A3: Summary report jako samostatný command

**Nevýhoda:** Stav pipeline se nikde neukládá persistentně — command by neměl co číst.
**Zamítnuto.** Summary je vždy součástí fix-bugs kroku 6.

### A4: Strukturovaný JSON výstup

**Nevýhoda:** Plugin je čistý markdown — JSON by nikdo nezpracoval.
**Zamítnuto.** Markdown tabulka je dostačující.

## Nové/upravené soubory

| Soubor | Změna | Rozsah |
|--------|-------|--------|
| `commands/fix-bugs.md` | Přidat dry-run sekci + rozšířit krok 6 (Summary) | ~20 řádků |
| `commands/fix-ticket.md` | Přidat dry-run sekci | ~15 řádků |

| `skills/bug-workflow/SKILL.md` | Upravit | Přidat intent mapping pro "dry run / co když / simulace" |

Agenti se **nemění** — triage-analyst a code-analyst již vrací strukturované výstupy odpovídající potřebám dry-run reportu.

## Designová rozhodnutí

### R1: Dry-run nesmí mít žádné side effects

Dry-run nesmí měnit stav v issue trackeru, vytvářet git branches/worktrees, volat fixer/reviewer/test-engineer/publisher. Smí jen číst z issue trackeru a repozitáře, spouštět triage-analyst a code-analyst (oba read-only).

### R2: Složitost se odhaduje v commandu, ne v agentovi

Code-analyst vrací: Affected files, Risk level, Root cause candidates. Z těchto dat command deterministicky vypočítá složitostní stupeň (XS/S/M/L).

### R3: Summary report je synchronní a inline

Report se generuje po dokončení posledního bugu v kroku 6. Nevznikají žádné persistentní soubory. Výstup je pouze do Claude Code chatu.

### R4: `--dry-run` se parsuje LLM instrukcí, ne regex logikou

Command file obsahuje plaintext instrukci: "Pokud `$ARGUMENTS` obsahuje `--dry-run`, aktivuj dry-run mód." LLM tuto instrukci spolehlivě vyhodnotí z kontextu.

## Vyřešené otázky (2026-02-25)

### O1: fix-ticket summary formát
Fix-ticket zpracovává jeden bug — žádná tabulka. Explicitní výpis výsledku a případného blokujícího důvodu.

### O2: Dry-run code-analyst pro UNCLEAR/DUPLICATE
Přeskočit. Code-analyst se nevolá pro bugy s triage výsledkem UNCLEAR nebo DUPLICATE.

### O3: Odhad času a tokenů v dry-run reportu
**ANO — odhadovat čas i spotřebu tokenů.** Deterministický výpočet z počtu bugů × složitostní stupeň × typický profil agentů.

Časový odhad per bug dle složitosti:

| Stupeň | Odhad per bug |
|--------|---------------|
| XS | ~2 min |
| S | ~5 min |
| M | ~10 min |
| L | ~15+ min |

Tokenový odhad per agent:

| Agent | Model | Input | Output |
|-------|-------|-------|--------|
| triage-analyst | sonnet | ~2k | ~500 |
| code-analyst | sonnet | ~5k | ~1k |
| fixer | opus | ~10k | ~3k |
| reviewer | opus | ~8k | ~2k (× N iterací) |
| test-engineer | sonnet | ~5k | ~2k |
| publisher | haiku | ~2k | ~500 |

Formát v dry-run reportu:
```
### Odhad zdrojů
Odhadovaná doba pipeline: ~22 min (2× XS, 1× S, 1× M)
Odhadovaná spotřeba tokenů: ~120k input + ~35k output
  - Sonnet: ~48k input + ~14k output
  - Opus: ~64k input + ~18k output
  - Haiku: ~8k input + ~2k output
```

Rozpad per model — opus je výrazně dražší, uživatel vidí kde jde většina nákladů. Přesnost ±30 %.

### O4: Logovat dry-run do issue trackeru
Ne. Dry-run = nula side effects.
