## Critical (3)

### C1 + C2: Verze v3.2.0 je v rozporu s paralelizovatelností

Dokument přiřazuje v3.2.0 a současně říká "implementujte PŘED Fází 1" a "plně nezávislý na Phase 1 a 2". Ale v3.2.0 implikuje, že v3.0.0 a v3.1.0 už vyšly. Plugin.json bump 2.0.0 → 3.2.0 přeskakuje intermediate verze.

**Doporučení:** Verzi přiřadit až při release dle skutečného pořadí (pokud shippne první → v2.1.0).

### C3: Pipeline stage rekonstrukce závisí na neexistujících datech

Tabulka "poslední komentář → stage" předpokládá, že pipeline píše komentáře při každém stage přechodu. Ale reálně pipeline píše jen:
- Triage checkpoint (`[CLAUDE-agents] Triage dokončen.`)
- Block komentáře (`[CLAUDE-agents] 🔴 Pipeline Block`)

Úspěšně postupující issue (code-analyst hotov → fixer pracuje → reviewer schválil) nemá žádné intermediate komentáře → dashboard ukáže špatný stage.

**Doporučení:** (A) Audit jaké komentáře pipeline reálně zapisuje, nebo (B) explicitně zdokumentovat jako known limitation L1 ("stage pro aktivně postupující issues může být nepřesný").

## Important (6)

### I1: Sekce "Doporučení" schválena v Q&A ale chybí v designu

Q&A odpověď říká "Ano, přidej 3-5 řádků doporučení." Ale wireframe, datový model, command flow ani HTML template tuto sekci neobsahují.

**Doporučení:** Integrovat do designu, nebo explicitně označit jako "deferred to implementation decision".

### I2: Success rate formula je nekonzistentní

Datový model: `successRate = completed / (completed + blocked)`. Ale wireframe ukazuje "47 z 60 issues = 78%", přičemž `47 + 3 = 50 ≠ 60`. Chybí definice "total processed".

### I3: Filtr se aplikuje pozdě

Command flow: fetch ALL issues (step 4) → parse ALL comments (step 5) → query ALL branches (step 6) → apply filter (step 8). Pro `--filter "state:blocked"` (3 issues) se zbytečně stahují data pro 200+ issues.

**Doporučení:** State-level filtry aplikovat už na úrovni issue tracker query.

### I4: "build" v blocksByAgent není agent

JSON příklad: `"blocksByAgent": { ..., "build": 2 }`. Žádný agent se nejmenuje "build". Block Comment Template má `Agent: {název agenta}` — vždy skutečný agent.

**Doporučení:** Přejmenovat field na `blocksBySource` nebo opravit data.

### I5: Custom CSS path je scope creep pro L1

Přidává file-path dependency do "zero dependencies" pluginu. Nikde v wireframu, command flow ani HTML generation sekci není zmíněna. Žádná jiná optional config sekce v existujícím pluginu nemá file-path dependency.

**Doporučení:** Odložit na L2.

### I6: Command flagy jsou LLM-parsed, ne CLI-parsed

Flagy (`--output <path>`, `--format html|json`, `--filter <query>`) implikují strukturované parsování, ale reálně je LLM parsuje z volného textu `$ARGUMENTS`. Edge cases (`--output "path with spaces"`) závisí na LLM úsudku.

**Doporučení:** Zdokumentovat a zvážit zjednodušení pro L1.
