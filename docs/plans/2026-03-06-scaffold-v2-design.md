# Scaffold v2 — From Description to Working App

**Date:** 2026-03-06
**Status:** PROPOSED
**Version:** target v4.0.0 (breaking change — new default behavior for `/scaffold`)
**Depends on:** Existing scaffold pipeline (v3.0), feature pipeline agents (fixer, reviewer, test-engineer, architect)

---

## 1. Vision & Goals

### Problem

Current `/scaffold` generates a skeleton project (10-20 files, 1 smoke test, no business logic). The user must then manually:
1. Fill in TODO sections in CLAUDE.md
2. Create issues in the issue tracker
3. Run `/implement-feature` for each issue

This manual gap between "skeleton" and "working app" defeats the purpose of automation.

### Goal

One command takes a project description and produces a **fully functional application** with implemented features, comprehensive tests, and validated behavior. Zero manual steps required (except optional checkpoints).

### Success Criteria

- User runs `/scaffold "description"` → gets a buildable, tested, working application
- Works with or without an issue tracker
- Specification quality is enforced by spec-writer ↔ spec-reviewer loop
- Every feature has acceptance criteria that are verified by tests
- Application is validated by automated tests (unit, integration, E2E)

---

## 2. Three Modes

User chooses mode at the start of every scaffold run:

| Mode | Spec Phase | Checkpoints | Use When |
|------|-----------|-------------|----------|
| **Interactive** | Spec-writer asks questions one at a time, iterates with spec-reviewer | User approves spec + feature plan | You don't know exactly what you want, or it's complex |
| **YOLO with checkpoint** | Spec-writer generates autonomously, iterates with spec-reviewer | User approves spec + feature plan | You know what you want and can describe it well — **most common mode** |
| **Full YOLO** | Spec-writer generates autonomously, iterates with spec-reviewer | None — goes straight to implementation | Prototypes, experiments, "let's see what comes out" |

All three modes run the same spec-writer ↔ spec-reviewer loop (up to Spec iterations, default 5). The difference is only in how input is gathered and whether human checkpoints exist.

## 3. Input Sources

The system accepts input from multiple sources:

| Source | Flag | Behavior |
|--------|------|----------|
| Direct text | (default) | `/scaffold "REST API for user management with auth and roles"` — spec-writer starts from this description |
| Issue tracker card | `--issue <ID>` | Reads description from issue tracker, uses it as input for spec-writer |
| Custom template | `--template path/to/template.md` | Spec-writer uses this template instead of built-in default |
| Ready specification | `--spec path/to/spec/` | Skips spec-writer, spec-reviewer validates completeness, then continues |

### Validation of User-Supplied Spec (`--spec`)

When user provides a ready specification:
1. Spec-reviewer validates it against the built-in template (checks for missing REQUIRED sections)
2. If complete → pipeline continues
3. If sections missing → depends on mode:
   - **Interactive** → tells user what's missing, asks them to fill in
   - **YOLO / YOLO with checkpoint** → spec-writer fills in missing sections autonomously

### Input Source Conflicts

The input source flags are **mutually exclusive** — only one can be used at a time:
- `--spec`, `--template`, `--issue` cannot be combined (each defines a different input source)
- If multiple are provided → error: "Only one input source allowed. Use --spec, --template, or --issue."

Tech stack preset flags (`--lang`, `--framework`, `--db`, `--ci`) are **compatible with all input sources**. They are passed to spec-writer as constraints — spec-writer incorporates them into the Tech Stack section of the specification.

`--no-implement` is **incompatible with `--spec`, `--template`, `--issue`** (it skips the spec phase entirely). If combined → error: "--no-implement skips specification phase. Remove --spec/--template/--issue or remove --no-implement."

## 4. Specification Structure

Specification is always saved as a folder in the repository: `spec/`

### Folder Layout

```
spec/
├── README.md           # vision, goals, success criteria, tech stack, status
├── architecture.md     # architecture, data flow, integrations, NFR, constraints
├── verification.md     # test strategy, definition of done, risks, assumptions
└── epics/
    ├── 01-auth.md      # stories, acceptance criteria, API, data model
    ├── 02-users.md
    └── ...
```

### Built-in Specification Template

#### `spec/README.md`

```markdown
# Project Specification: {name}

> Generated by: spec-writer + spec-reviewer ({N} iterations)
> Date: {date}
> Mode: {interactive | yolo-checkpoint | yolo}
> Status: DRAFT | APPROVED

## Vision & Goals                                    [REQUIRED]
- **What:** one paragraph — what the project does
- **Why:** what problem it solves
- **Success criteria:** SMART metrics (measurable conditions for "done")

## Users & Personas                                  [REQUIRED]
- Primary users (persona + main use case)
- Secondary users

## Tech Stack                                        [REQUIRED]
| Component | Choice | Rationale |
|-----------|--------|-----------|
| Language | | |
| Framework | | |
| Database | | |
| CI/CD | | |
| Containerization | | |

## Out of Scope                                      [REQUIRED]
- What is explicitly NOT being built
```

#### `spec/architecture.md`

```markdown
# Architecture

## High-Level Overview                               [REQUIRED]
- Component diagram (text-based)
- How components communicate

## Data Flow                                         [REQUIRED]
- How data moves through the system

## Data Model                                        [IF APPLICABLE]
- Entity definitions + relationships
- Key constraints

## API Design                                        [IF APPLICABLE]
- Endpoints (method, path, request/response)
- Authentication/authorization model

## External Integrations                             [IF APPLICABLE]
- 3rd party services, APIs

## Non-Functional Requirements                       [REQUIRED]
- Performance (response time, throughput targets)
- Security (auth model, input validation, data protection)
- Reliability (error handling, recovery strategy)
- Observability (logging, monitoring, health checks)

## Design Constraints                                [IF APPLICABLE]
- Regulatory/compliance requirements
- Infrastructure limitations
- Compatibility requirements
```

#### `spec/verification.md`

```markdown
# Verification Plan

## Test Strategy                                     [REQUIRED]
- Unit tests: scope and coverage target
- Integration tests: what integrations to test
- E2E tests: critical user flows to cover

## Definition of Done                                [REQUIRED]
- Conditions that must be met for the project to be considered complete

## Risks & Assumptions                               [REQUIRED]
- Assumptions (what we assume to be true)
- Known risks + mitigation strategies

## Open Questions                                    [IF ANY]
- Unresolved questions (for interactive mode)
```

#### `spec/epics/NN-name.md`

```markdown
# Epic: {name}

## Description
One paragraph — what this epic delivers.

## User Stories                                      [REQUIRED]
- As a {persona}, I want {action}, so that {benefit}
  - **Acceptance criteria:**
    - [ ] {testable condition}
    - [ ] {testable condition}

## API Endpoints                                     [IF APPLICABLE]
| Method | Path | Description | Auth |
|--------|------|-------------|------|

## Data Model Changes                                [IF APPLICABLE]
- New/modified entities

## Dependencies
- Depends on: {other epics or "none"}

## Priority
- must | should | could
```

### Template Principles

- **REQUIRED** sections must always be filled — spec-reviewer blocks if missing
- **IF APPLICABLE** sections are skipped with explanation (e.g., "No API — CLI tool")
- Every acceptance criterion must be testable — spec-reviewer enforces this
- Feature Breakdown in epics is the input for the architect agent
- All projects use the same template structure — consistency across applications

## 5. New Agents

### spec-writer (model: opus)

> **Model justification:** Opus instead of sonnet (which handles specification in other agents) because spec-writer's output drives the entire downstream pipeline — errors in the specification cascade to architecture, implementation, and testing. Unlike spec-analyst (which analyzes an existing feature request), spec-writer creates the complete application specification from scratch, making critical scoping and design decisions.

**Role:** Generates project specification from user input.

**Input sources:**
- Direct text description (from user or issue tracker card)
- Answers from interactive questioning (interactive mode)
- Custom template (uses it instead of built-in)
- Tech stack constraints from flags (--lang, --framework, --db, --ci) — incorporated into Tech Stack section

**Process:**
1. Parse input (description, flags, template)
2. In interactive mode: ask clarifying questions one at a time (purpose, users, features, constraints, tech preferences)
3. Generate specification following the folder template
4. Ensure all REQUIRED sections are filled
5. Ensure every user story has testable acceptance criteria
6. Output: complete `spec/` folder content

**Constraints:**
- NEVER skip REQUIRED sections
- NEVER write vague acceptance criteria ("works correctly" → must be specific and testable)
- In interactive mode: one question at a time, prefer multiple-choice when possible
- Must generate rationale for tech stack choices

### spec-reviewer (model: opus)

**Role:** Reviews specification quality, completeness, and consistency.

**Process:**
1. Read entire specification (all files in `spec/`)
2. Check completeness: all REQUIRED sections present and filled
3. Check quality: every acceptance criterion is testable and specific
4. Check consistency: no contradictions between sections (e.g., tech stack vs architecture)
5. Check feasibility: are the requirements implementable within the stated constraints
6. Check scope: flag overengineered requirements, suggest simplification
7. Output: list of issues with severity (BLOCK / WARN) + specific improvement suggestions

**Constraints:**
- NEVER approve specs with missing REQUIRED sections
- NEVER approve vague acceptance criteria
- NEVER modify the specification — only review and suggest changes
- Must flag overengineered requirements (YAGNI enforcement)

### Spec-writer ↔ Spec-reviewer Loop

```
spec-writer generates/updates spec
    │
    ▼
spec-reviewer reviews
    │
    ├── All clear → spec is ready
    └── Issues found → feedback to spec-writer → next iteration
         (max iterations from Retry Limits → Spec iterations, default 5)

If max iterations exhausted and BLOCK issues remain:
    → Report to user with remaining issues
    → User decides: approve anyway / provide input / abort
```

**Note:** Spec iterations limit is configurable via Automation Config → Retry Limits → Spec iterations (default: 5). Scaffolder generates this key in the CLAUDE.md it creates.

## 6. Full Pipeline

```
/scaffold "project description" [--template X] [--spec X] [--issue X]
    │
    ▼
Step 0: Mode selection
    If --no-implement → skip to stack-selector → scaffolder (v3.x flow), exit pipeline
    │
    "How do you want to proceed?"
    (a) Interactive — we'll build the spec together
    (b) YOLO with checkpoint — I'll design everything, you approve
    (c) Full YOLO — I'll design and implement, no stops
    │
    ▼
Step 1: Specification phase
    ├── --spec provided → spec-reviewer validates (fill gaps per mode)
    ├── --template provided → spec-writer uses custom template
    └── default → spec-writer uses built-in template
    │
    spec-writer ↔ spec-reviewer (up to Spec iterations, default 5)
    │
    Output: spec/ folder written to target directory
    │
    ▼
Step 2: Spec checkpoint (skip in Full YOLO)
    User reviews spec/ folder → approve / edit / reject
    │
    ▼
Step 3: Scaffold skeleton
    Scaffolder generates project skeleton into temp directory:
    - Build config, entry point, project structure
    - .gitignore, configs
    - 1 smoke test, linter config
    - Dockerfile, CI config
    - CLAUDE.md with Automation Config
    │
    Scaffolder receives tech stack from spec/README.md (Tech Stack section).
    If --lang/--framework/--db/--ci flags were provided, they are already
    incorporated into the spec by spec-writer — scaffolder reads from spec.
    │
    Validation: build + test + lint (max 3 retries)
    Move skeleton to target directory (which already contains spec/)
    │
    ▼
Step 4: Git init
    git init → git add . → commit "feat: initial project scaffold"
    (commits both spec/ and skeleton files)
    │
    ▼
Step 5: Architecture & decomposition
    Command reads spec/epics/*.md and formats each epic's user stories
    into the structured specification format that architect expects
    (same format as spec-analyst output in implement-feature pipeline).
    │
    Architect receives formatted input → decomposes user stories into
    subtasks and groups them into dependency-aware batches:

    Batch 1: Auth + User model (foundation)
    Batch 2: Roles + Permissions (depends on auth)
    Batch 3: Notifications + Reports (independent)
    │
    Each user story from epics becomes one or more subtasks (fixer tasks).
    Each subtask ≤ 100 lines diff (architect constraint).
    Max subtasks limit applies to the TOTAL across all epics.
    If exceeded → architect must merge/simplify subtasks.
    │
    ▼
Step 6: Feature plan checkpoint (skip in Full YOLO)
    User sees batches → approve / reorder / remove features
    │
    ▼
Step 7: Feature implementation loop
    For each batch:
    │
    ├── For each feature (subtask) in batch:
    │   ├── Fixer (opus) — implements feature
    │   ├── Reviewer (opus) — reviews (fixer ↔ reviewer loop, max Fixer iterations)
    │   ├── Test-engineer (sonnet) — writes tests
    │   ├── Commit — "feat: {feature name}"
    │   │
    │   └── On block (fixer/reviewer/test-engineer):
    │       ├── Rollback-agent reverts to last successful commit
    │       ├── Block reported to stdout (no issue tracker in scaffold)
    │       └── Continue or stop per Fail strategy from Decomposition config
    │
    ├── Batch validation: run full test suite
    │   └── If failure → fixer repairs (max Build retries)
    │       └── If still failing → stop and report
    │
    └── Next batch
    │
    ▼
Step 8: E2E tests
    E2E-test-engineer writes and runs end-to-end tests
    covering critical user flows from spec
    │
    If failures → fixer repairs → re-run (max 3 retries)
    │
    ▼
Step 9: Issue tracker (optional)
    "Create cards in issue tracker? [Y/n]"
    If yes → create epics/features as cards from spec/epics/
    │
    ▼
Step 10: Final report
    ## Scaffold Complete

    **Project:** {name}
    **Mode:** {mode}
    **Stack:** {summary}
    **Spec:** {N} iterations, {status}
    **Features:** {implemented} / {total}
    **Tests:** {unit} unit, {integration} integration, {e2e} e2e

    ### Generated files: {count}
    ### Spec: spec/
    ### Remaining TODOs in CLAUDE.md:
    - [ ] Issue Tracker instance
    - [ ] Source Control remote
```

## 7. Decomposition & Batching

### How features are grouped

Architect reads `spec/epics/*.md` and produces a dependency-aware batch plan:

1. **Dependency analysis:** which features depend on which (from `Dependencies` field in epic files)
2. **Batch grouping:** features with no unmet dependencies go into the same batch
3. **Batch ordering:** batches are ordered so dependencies are always satisfied
4. **Batch size:** 2-3 features per batch (keeps context manageable, limits rollback blast radius)

### Failure handling

- **Per-feature failure:** rollback-agent reverts last feature commit, then follow Fail strategy (`fail-fast` = stop pipeline, `continue` = skip and proceed to next feature in batch)
- **Per-batch failure:** if batch integration tests fail after 3 retries, stop and report
- **Decomposition config:** reads `Max subtasks`, `Fail strategy`, `Commit strategy` from Automation Config (generated by scaffolder with scaffold-optimized defaults)

### Scaffold-optimized Decomposition defaults

Scaffolder generates these defaults in CLAUDE.md Automation Config:

| Key | Scaffold default | Standard default | Rationale |
|-----|-----------------|------------------|-----------|
| Max subtasks | 5 | 7 | Conservative for new projects |
| Fail strategy | fail-fast | fail-fast | Same — failing fast is correct for scaffold |
| Commit strategy | individual | squash | Better git history for new projects |

**Important:** Scaffolder MUST generate an `E2E Test` section in Automation Config when running in scaffold v2 mode (not --no-implement). The E2E-test-engineer agent blocks if this section is missing. Scaffolder determines the E2E framework from the tech stack (e.g., Playwright for web apps, supertest for APIs).

## 8. Issue Tracker Interaction

### Principle

Issue tracker is **fully optional** in scaffold pipeline. Specification in the repo (`spec/`) is the single source of truth.

### Three scenarios

| Scenario | Behavior |
|----------|----------|
| No tracker configured | Pipeline runs without tracker. Block comments go to stdout. Publisher skipped (local commits only). |
| Tracker configured, user declines cards | Same as above — tracker exists but not used for this scaffold run. |
| Tracker configured, user accepts cards | After implementation, epics/features created as cards. Spec links to cards. |

### What gets created in tracker (when opted in)

- One epic per `spec/epics/*.md`
- Features as sub-issues under epics (from user stories)
- Each card links back to spec file in repo
- Status set per Automation Config state transitions

## 9. Command Interface

### Updated `/scaffold` command

```
/scaffold "project description"
    [--template path/to/template.md]   # custom spec template
    [--spec path/to/spec/]             # ready specification (skip spec-writer)
    [--issue <ID>]                     # read description from issue tracker
    [--lang <language>]                # preset language (passed to spec-writer as constraint)
    [--framework <framework>]          # preset framework
    [--db <database>]                  # preset database

```

### Existing commands — no changes needed

- `/scaffold-add` — remains as-is (add component to existing project)
- `/scaffold-validate` — remains as-is (validate project state)

## 11. Specification Template Customization

### Built-in template

Default template is defined in section 4. Used when no `--template` flag is provided.

### Custom template

User provides a markdown file with their own sections. Spec-writer follows the custom template structure instead of built-in. Spec-reviewer still validates that key concepts are covered (vision, features with acceptance criteria, tech stack) but accepts different section names/organization.

### Ready specification

User provides a complete `spec/` folder. Spec-writer is skipped for generation. Spec-reviewer validates:
1. At least one epic file exists in `spec/epics/`
2. Each epic has user stories with acceptance criteria
3. Tech stack is defined somewhere in the spec
4. Missing sections are handled per mode (interactive → ask user; YOLO → spec-writer is invoked to fill gaps only, not regenerate)

## 12. Risks & Mitigation

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Spec-writer ↔ spec-reviewer converge on flawed spec (LLM blind spots) | MEDIUM | HIGH | Human checkpoint after spec (except Full YOLO); 2 minutes of review catches 90% of issues |
| Too many features — pipeline runs too long, exhausts tokens | HIGH | HIGH | Decomposition limit (Max subtasks default 5); batching with checkpoints |
| Scaffold skeleton incompatible with features | MEDIUM | HIGH | Skeleton validated (build+test) before feature implementation starts |
| Feature implementation diverges from spec intent (LLM interpretation drift) | MEDIUM | MEDIUM | Reviewer catches divergence; acceptance criteria from spec are explicit and testable |
| Fixer exceeds 100-line diff limit on complex features | HIGH | MEDIUM | Architect must decompose features into small subtasks; fixer constraint enforced by reviewer |
| Custom template missing critical sections | LOW | MEDIUM | Spec-reviewer validates key concepts regardless of template structure |

## 13. Versioning Impact

This is a **MAJOR** version bump (v4.0.0) because:
- Changes default behavior of `/scaffold` command (adds mode selection)
- Introduces 2 new required agents (spec-writer, spec-reviewer)
- Introduces new `spec/` folder convention
- Modifies scaffold command behavior for publisher integration (command-level — skips tracker steps when not configured; publisher agent definition unchanged)

Backwards compatibility: running `/scaffold` still works — user just gets a mode selection prompt first. Old behavior is equivalent to choosing "Full YOLO" mode without implementation (skeleton only), but this requires adding a 4th mode or `--skeleton-only` flag for pure backwards compat.

### Backwards compatibility option

Add `--no-implement` flag:
```
/scaffold "description" --no-implement
```
This produces skeleton only (current v3.x behavior). No spec phase, no feature implementation. Stack-selector + scaffolder only.

**Note:** There is no "spec-only" mode (generate spec without implementing). This is deferred to Future Extensions. Users who want spec-only output can use Full YOLO mode and abort after the spec checkpoint, or use --spec flag in a future run to reuse the generated spec.

## 14. Future Extensions (Not in v1)

| Extension | Description | Why deferred |
|-----------|-------------|-------------|
| `/scaffold-implement` standalone command | Run feature implementation on existing project (not just after scaffold) | Scope — scaffold v2 is already large |
| `--remote` and `--tracker-instance` flags | Auto-fill CLAUDE.md TODOs during scaffold | Nice-to-have, manual fill is acceptable |
| Spec diffing | Compare spec changes across versions | Needs usage patterns first |
| Template marketplace | Share spec templates across teams | Needs users first |
| Autonomous browser validation | Playwright MCP agent navigates app, clicks through features, validates UI | Underspecified (no assigned agent, app startup undefined); E2E tests are primary safety net |
| `--spec-only` flag | Generate spec/ without implementation | Low demand — users can abort after spec checkpoint |
