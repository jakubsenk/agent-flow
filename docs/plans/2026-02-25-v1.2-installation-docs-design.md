# Design: v1.2.0 — Instalační dokumentace

**Datum:** 2026-02-25
**Status:** APPROVED (2026-02-25)

## Přehled

Plugin CLAUDE-agents má v README sekci `## Instalace` se dvěma řádky (`/plugin install` příkaz + marketplace URL). To nestačí pro nového uživatele — existují tři blokující závislosti, které README vůbec nepokrývá:

1. **Přístup k Gitea** — plugin je na interním Gitea (`gitea.internal.ceosdata.com`). Bez SSH klíče nebo HTTPS tokenu `git clone` (a tedy `/plugin install`) selže.
2. **MCP servery** — agenti komunikují s issue trackerem a source control přes MCP. Bez `.mcp.json` s platnými tokeny pipeline neudělá nic.
3. **Automation Config** — bez `## Automation Config` v CLAUDE.md projektu commands sice běží, ale nemají co číst.

Cíl v1.2.0: pokrýt celou cestu od "nemám nic" po "pipeline funguje" srozumitelným, krok za krokem návodem.

---

## Navrhovaný přístup

### Kam dokumentace patří

**Hlavní volba: separátní `docs/` soubory + shrnutí v README.**

README zůstane stručné (jak je teď) a přidá odkaz na detailní průvodce. Detailní obsah patří do `docs/` ze tří důvodů:

1. Instalační návod je dlouhý — vložit ho do README by zdvojnásobilo délku a pohřbilo architekturní sekce.
2. Různé části se aktualizují různě — token scopes v YouTrack se mění nezávisle na plugin verzi.
3. `docs/` je zavedené místo pro delší dokumenty (plány, ADR). Instalační dokumenty sem přirozeně patří.

### Struktura nových souborů

```
docs/
├── plans/              ← ADR (stávající)
└── setup/              ← nová složka
    ├── installation.md       ← krok za krokem: Claude Code, plugin, git přístup
    ├── tokens.md             ← generování tokenů: YouTrack API, Gitea/Forgejo API
    └── mcp-configuration.md  ← .mcp.json — struktura, příklady, ověření
```

README dostane novou sekci `## Rychlý start` (nad stávající `## Instalace`) s odkazem na `docs/setup/installation.md` a inline blokem pro nejjednodušší případ.

## Alternativy

### A) Vše v README

**Pro:** Vše na jednom místě, žádné klikání.
**Proti:** README se nafukuje. Na Gitea (webový pohled) by bylo nutné rolovat přes 300+ řádků než uživatel uvidí architekturní sekce. Instalační kroky se mění nezávisle na architektuře.
**Zamítnuto.**

### B) Wiki na Gitea

**Pro:** Oddělené od repo kódu, verzované zvlášť.
**Proti:** Interní Gitea, ne všichni mají přístup k wiki editaci. Plugin je veřejně distribuovatelný — wiki je nesdílitelná. Dokumentace by nebyla součástí `git clone`.
**Zamítnuto.**

### C) Jeden velký `docs/SETUP.md`

**Pro:** Méně souborů, méně odkazů.
**Proti:** Jeden soubor by míchalo tři různé osy: instalace Claude Code, token generování, MCP konfigurace. Každá osa má jiného "zákazníka" (admin, developer, new joiner) a jiný aktualizační rytmus.
**Zamítnuto.**

### D) `docs/setup/` se třemi soubory (navrhovaný přístup)

**Pro:** Každý soubor má jasný scope, dá se linkovat selektivně ("tady je jen token setup"), snadná údržba.
**Přijato.**

## Obsah jednotlivých souborů

### `docs/setup/installation.md`

Sekce:

1. **Předpoklady**
   - Claude Code CLI nainstalovaný (`claude --version`)
   - Git nainstalovaný a nakonfigurovaný (`git config --global user.email`)
   - Přístup k internímu Gitea (SSH klíč nebo HTTPS token)

2. **Přístup k Gitea** (platforma-specificky)
   - SSH varianta: vygenerovat klíč, přidat do `~/.ssh/config`, přidat veřejný klíč do Gitea profilu
   - HTTPS varianta: vygenerovat Gitea Personal Access Token (odkaz na `tokens.md`)
   - Ověření: `git ls-remote git@gitea.internal.ceosdata.com:fsabacky/CLAUDE-agents.git`

3. **Instalace pluginu**
   - Spustit Claude Code, zadat `/plugin install CLAUDE-agents@CLAUDE-agents`
   - Ověření: `/CLAUDE-agents:analyze-bug` musí být dostupný (tab-complete v Claude Code)
   - Update postup (marketplace cache) pro případ, že uživatel upgraduje

4. **Setup projektu** (stručně, odkaz na `mcp-configuration.md`)
   - Vytvořit `.mcp.json`
   - Přidat `## Automation Config` do CLAUDE.md
   - Ověřit: `/CLAUDE-agents:analyze-bug PROJ-1`

Platformové varianty: Windows (primární, stávající dev prostředí), Linux (v1.4.0 roadmap), macOS (analogické Linuxu). V1.2.0 pokryje Windows plně, ostatní s poznámkou "analogicky — detaily v v1.4.0".

### `docs/setup/tokens.md`

Sekce:

1. **Přehled tokenů** — co k čemu: YouTrack token pro MCP server, Gitea token pro MCP server. Jasně oddělit: jeden token = jedna služba.

2. **YouTrack API token**
   - Navigace: YouTrack → profil → Hub → Authentication → New token
   - Požadované scope: `YouTrack` (nebo specificky `Issue tracker read/write`)
   - Pojmenování tokenu (doporučení: `CLAUDE-agents-PROJEKT`)
   - Jak zkopírovat a bezpečně uložit (token se zobrazí jen jednou)
   - Platnost tokenu — doporučení: bez expirace nebo 1 rok

3. **Gitea / Forgejo API token**
   - Navigace: Gitea → Settings → Applications → Generate New Token
   - Požadované scope: `repository:read`, `repository:write`, `issue:write` (pro PR + issue komentáře)
   - Pojmenování tokenu
   - Platnost tokenu

4. **Bezpečnost tokenů**
   - `.mcp.json` NIKDY do git — přidat do `.gitignore`
   - Tokeny nepsat do CLAUDE.md (je v gitu)
   - Pokud token unikne: okamžitě revokovat v příslušné službě

### `docs/setup/mcp-configuration.md`

Sekce:

1. **Co je `.mcp.json`**
   - Soubor v root projektu (nebo `~/.claude/`)
   - Obsahuje MCP server definice s tokeny
   - Claude Code ho načítá automaticky při spuštění v daném adresáři

2. **Struktura souboru**
   - Ukázka s oběma servery (YouTrack MCP + Gitea/Forgejo MCP)
   - Placeholder syntax pro tokeny

3. **Gitea/Forgejo MCP server setup**
   - Nainstalovat MCP server (příkaz/zdroj)
   - Konfigurační klíče: `url`, `token`
   - Ověření: `mcp__gitea__list_repos` v Claude Code

4. **YouTrack MCP server setup**
   - Nainstalovat MCP server
   - Konfigurační klíče: `url`, `token`, `project` (nebo globální)
   - Ověření: `mcp__youtrack__get_issue` v Claude Code

5. **Ověření celého setupu**
   - Spustit `/CLAUDE-agents:analyze-bug PROJ-TESTISSUE`
   - Co znamená každá chybová hláška (token neplatný, MCP server nedostupný, Automation Config chybí)

## Designová rozhodnutí

### 1. Separátní soubory místo jednoho mega-dokumentu

Každý soubor odpovídá jedné "akci" s jasným vstupem a výstupem. Uživatel, který jen potřebuje obnovit YouTrack token, otevře `tokens.md` — nemusí hledat v 500-řádkovém dokumentu.

### 2. Tokeny a MCP konfigurace odděleny od instalace

Token generování a `.mcp.json` konfigurace jsou per-projekt kroky, které se opakují pro každý nový projekt. Instalace pluginu je jednorázová (globální). Sloučení by mátlo uživatele, kteří plugin mají, ale nastavují nový projekt.

### 3. README odkazuje, neobsahuje

README je "vstupní bod" — má říct "co to je" a "kde začít". Detailní kroky patří do `docs/setup/`. Tím README zůstane čitelné i při budoucím rozšiřování dokumentace.

### 4. Windows-first v1.2.0

Stávající uživatelé (BIFITO, interní dev prostředí) jsou na Windows. Linux je v roadmapě jako v1.4.0. Psát cross-platform návod, kde 60 % je untested, by bylo zavádějící. V1.2.0 dokumentuje Windows, přidá poznámku "Linux — viz v1.4.0".

### 5. Bezpečnostní sekce v `tokens.md`

`.mcp.json` obsahuje tajné tokeny. Dokumentace musí explicitně upozornit na `.gitignore` — nejen v poznámce, ale jako pojmenovanou sekci. Pokud uživatel `.mcp.json` omylem commitne, token unikne do gitu.

### 6. Ověřovací kroky na konci každé sekce

Každá sekce instalační dokumentace končí "jak ověřit, že to funguje". Uživatel tak nepokračuje dál s nesprávnou konfigurací a dostane jasný signal, kde se pipeline zasekla.

## Vyřešené otázky (2026-02-25)

1. **YouTrack MCP server:** `@vitalyostanin/youtrack-mcp` (npm). Spouštěný přes `npx -y @vitalyostanin/youtrack-mcp@latest` — zajistí vždy aktuální verzi bez manuální instalace. Ověřeno na BIFITO projektu.

2. **Gitea/Forgejo MCP server:** `forgejo-mcp` — oficiální Go binary od Forgejo, stahovaný z [codeberg.org/forgejo/forgejo-mcp/releases](https://codeberg.org/forgejo/forgejo-mcp/releases). Binary se ukládá do `bin/forgejo-mcp.exe` (Windows) / `bin/forgejo-mcp` (Linux) v projektu. Kompatibilní s Gitea i Forgejo.

3. **Scope tokenů YouTrack:** YouTrack Cloud — permanent token se scope `YouTrack` (nebo specificky `Issue tracker read/write`). Ověřeno na cmd.youtrack.cloud (BIFITO instance).

4. **`.mcp.json` — per-projekt.** Důvody: izolace tokenů (různé projekty = různé instance), `.mcp.json.example` trackovaný v gitu jako template, production `.mcp.json` v `.gitignore`. Pattern ověřený na BIFITO. Globální `~/.claude/.mcp.json` se nepoužívá.

5. **Jazyk setup dokumentace:** Česky (per jazykovou politiku projektu: česky pro docs, anglicky pro agent definice). Potvrzeno.
