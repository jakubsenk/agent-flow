## 1. Vize & cíl

### Proč dashboard

CLAUDE-agents pipeline dnes běží v terminálu. Člověk, který pipeline spustí, vidí output přímo v Claude Code session. Nikdo další ne. To je fundamentální problém pro týmy:

- **PM** nevidí, kolik bugů se opravuje, kolik je zablokovaných, jaký je throughput
- **QA** neví, které issues prošly celou pipeline a které zůstaly blocked — musí se ptát nebo hledat v issue trackeru manuálně
- **Team lead** nemá přehled o zdraví pipeline — poměr úspěšných vs. blocked, průměrná doba na stage, trendy
- **Developer**, který pipeline nespustil, neví, co se děje s jeho issues

### Co dashboard přinese

Jednoduchá statická HTML stránka, která agreguje data z issue trackeru a gitu do přehledné vizualizace:

1. **Týmová viditelnost** — každý v týmu vidí stav všech issues v pipeline na jednom místě
2. **PM oversight** — success rate, throughput, trendy — bez nutnosti chodit do terminálu
3. **QA transparentnost** — blocked issues s důvodem a doporučením, PR linky pro review
4. **Historická data** — timeline aktivit, statistiky za zvolené období

### Quick win argument

Dashboard L1 je ideální quick win:

- **Nulová infrastruktura** — žádný server, žádná databáze, žádné dependencies. Výstup je jeden HTML soubor.
- **Nulové nové agenty** — command sám čte data z issue trackeru a gitu, žádný specialista není potřeba
- **Data už existují** — issue tracker stavy, `[CLAUDE-agents]` komentáře, git branches, PRs — všechno je k dispozici
- **Nízký effort** — 1 nový command, 1 optional config sekce, rozšíření skill routing
- **Okamžitá hodnota** — i s 5 issues v pipeline je přehled užitečný

### Srovnání s existujícím `/status` commandem

`/status` command (`commands/status.md`) dnes poskytuje:

| Aspekt | `/status` | `/dashboard` |
|--------|-----------|-------------|
| **Formát** | Tabulka v terminálu | HTML stránka v prohlížeči |
| **Data** | Aktivní issues (In Progress, Blocked, For Review) | Aktivní + uzavřené issues za období |
| **Detail** | Issue ID, název, fáze, branch, PR | + blocked důvod, agent, doporučení, timeline, statistiky |
| **Audience** | Developer v terminálu | Celý tým (sdílitelné) |
| **Persistence** | Jednorázová — zmizí se session | Soubor na disku — lze sdílet, commitnout, refreshnout |
| **Filtrování** | Žádné | Podle stavu, stage, časového rozmezí |
| **Statistiky** | Počty (N in progress, M blocked) | Success rate, průměrné časy, throughput |
| **Blocked detail** | Žádný | Agent, krok, důvod, doporučení (z `[CLAUDE-agents]` komentářů) |

`/dashboard` NENAHRAZUJE `/status`. Jsou komplementární:

- `/status` = rychlý pohled v terminálu pro developera (live data, žádný soubor)
- `/dashboard` = sdílený přehled pro tým (HTML soubor, historická data, statistiky)

### Detaily jednotlivých sekcí

#### Header

- Název projektu (z Automation Config → Issue Tracker → Project)
- Timestamp generování (ISO 8601, lokální časová zóna)
- Verze pluginu (z `plugin.json`)
- Odkaz na Automation Config referenci (typ issue trackeru, remote)

#### Pipeline Overview (3 karty)

- **Aktivní** — issues ve stavech In Progress, For Review (z Automation Config → State transitions)
- **Zablokované** — issues ve stavu Blocked
- **Dokončené** — issues uzavřené/done za zvolené období (default: 30 dní)

Barvy:
- Aktivní: modrá (#3B82F6)
- Zablokované: červená (#EF4444)
- Dokončené: zelená (#22C55E)

#### Issue Table

Hlavní tabulka se všemi issues v pipeline. Sloupce:

| Sloupec | Zdroj dat | Sortovatelný |
|---------|-----------|-------------|
| ID | Issue tracker | Ano (default: desc) |
| Název | Issue tracker (summary) | Ano |
| Stav | Issue tracker (state) | Ano |
| Pipeline stage | Odvozen z `[CLAUDE-agents]` komentářů (viz sekce 3) | Ano |
| Agent | Poslední agent dle komentáře | Ano |
| Timestamp | Poslední `[CLAUDE-agents]` komentář timestamp | Ano |
| PR | Source control MCP — link na PR pokud existuje | Ne |

#### Blocked Issues Panel

Detailní pohled na zablokované issues. Parsuje `[CLAUDE-agents]` block komentáře:

```
[CLAUDE-agents] 🔴 Pipeline Block
Agent: {agent}
Krok: {krok}
Důvod: {důvod}
Detail: {detail}
Doporučení: {doporučení}
```

Panel zobrazuje:
- Issue ID + název
- Blokující agent
- Důvod (max 2 věty)
- Doporučení (co by měl člověk udělat)
- Detail je skrytý pod "Zobrazit detail" expander (JavaScript)

#### Recent Activity Timeline

Posledních N pipeline eventů (default: 20). Zdroj: `[CLAUDE-agents]` komentáře v issue trackeru, seřazené dle času desc.

Každý event:
- Timestamp (HH:MM nebo datum pokud starší než dnes)
- Issue ID (link do issue trackeru)
- Agent name
- Stručný popis akce (parsovaný z komentáře)

Barevné kódování a klasifikace `type`:

| Komentář obsahuje | Timeline `type` | Barva |
|-------------------|----------------|-------|
| "Triage dokončen" | success | zelená tečka |
| "Pipeline Block" | block | červená tečka |
| PR link komentář | success | zelená tečka |
| Ostatní `[CLAUDE-agents]` komentáře | progress | modrá tečka |

#### Statistics Panel

Metriky za zvolené období:

1. **Success rate** — poměr úspěšně dokončených vs. celkem zpracovaných issues (`completed / total`, kde `total = completed + blocked + active`)
   - Vizualizace: progress bar s procentem
2. **Throughput** — počet dokončených issues za období, průměrný denní throughput
3. **Průměrný čas na stage** (odhad) — kolik času (medián) stráví issue v každém pipeline stage
   - Zdroj: časové rozdíly mezi `[CLAUDE-agents]` komentáři
   - **L1 known limitation:** Časy jsou odhadnuté z existujících komentářů (triage checkpoint, block). Přesné časy vyžadují checkpoint komentáře pro každý stage přechod (L2). Dashboard zobrazí "~N min (odhad)" místo přesných hodnot.
4. **Rozdělení bloků** (`blocksBySource`) — který agent blokuje nejčastěji a kolikrát. Zdroj: pole `Agent:` z block komentáře (vždy skutečný agent name).
   - Vizualizace: jednoduchá tabulka s procenty

#### Recommendations Panel (Doporučení)

Na základě dat dashboard vygeneruje 3-5 řádků stručných doporučení. Pravidla pro generování jsou threshold-based (ne AI-generated):

| Podmínka | Doporučení |
|----------|-----------|
| blocked > 0 | "{N} issues jsou blocked — zkontrolujte je a rozhodněte o dalším postupu" |
| Jeden agent blokuje > 30% issues | "{agent} blokuje {X}% issues — zvažte úpravu retry limitů nebo dekompozici složitých bugů" |
| Průměrný čas na stage roste vs. předchozí období | "Průměrná doba ve fázi {stage} roste — zkontrolujte {relevant infrastructure}" |
| Success rate < 50% | "Success rate je pod 50% — zvažte review pipeline konfigurace" |
| Žádné blocked issues | "Žádné zablokované issues — pipeline běží hladce" |

Doporučení se zobrazí jen pokud data splňují alespoň jednu podmínku. Pokud pipeline běží bez problémů, sekce zobrazí pozitivní zprávu.

## 3. Data Sources

### 3.1 Issue tracker (primární zdroj)

Data dostupná přes MCP server odpovídající `Type` z Automation Config:

| Data | MCP operace | Dostupné dnes |
|------|------------|---------------|
| Issue ID, název, stav | Issue query (Bug query z config) | Ano |
| Issue komentáře | Issue comments read | Ano |
| State transitions | Issue state history (pokud tracker podporuje) | Částečně (závisí na trackeru) |
| Custom fields (priority, severity) | Issue field read | Ano |

#### Parsování `[CLAUDE-agents]` komentářů

Všechny pipeline komentáře začínají prefixem `[CLAUDE-agents]`. Dva formáty:

**Triage checkpoint:**
```
[CLAUDE-agents] Triage dokončen. Severity: {severity}. Area: {area}.
```
Regex: `^\[CLAUDE-agents\] Triage dokončen\. Severity: (.+)\. Area: (.+)\.$`

**Block comment:**
```
[CLAUDE-agents] 🔴 Pipeline Block
Agent: {agent}
Krok: {krok}
Důvod: {důvod}
Detail: {detail}
Doporučení: {doporučení}
```
Regex (multiline): `^\[CLAUDE-agents\] 🔴 Pipeline Block\nAgent: (.+)\nKrok: (.+)\nDůvod: (.+)\nDetail: (.+)\nDoporučení: (.+)$`

Z těchto komentářů lze extrahovat:
- **Pipeline stage** — poslední agent v komentářích určuje stage
- **Block důvod** — Agent, Krok, Důvod, Detail, Doporučení
- **Severity a area** — z triage checkpointu
- **Timeline** — timestamp komentáře = časová osa pipeline

#### Rekonstrukce pipeline stage

Pipeline stage není uložený explicitně — je odvozen z posledního `[CLAUDE-agents]` komentáře:

| Poslední komentář obsahuje | Odvozený stage |
|----------------------------|----------------|
| "Triage dokončen" | analysis (triage hotov, code-analyst následuje) |
| Block od triage-analyst | triage (zablokováno při triage) |
| Block od code-analyst | analysis |
| Block od fixer | fix |
| Block od reviewer | review |
| Block od test-engineer/e2e-test-engineer | test |
| PR link komentář | publish (hotovo) |
| Žádný `[CLAUDE-agents]` komentář | triage (ještě nezahájeno nebo právě běží) |

**Known limitation (L1):** Pipeline dnes zapisuje komentáře pouze ve dvou situacích: (1) triage checkpoint po úspěšné triage, (2) block komentář při selhání. Úspěšně postupující issue (code-analyst hotov, fixer pracuje, reviewer schválil) nemá žádné intermediate komentáře. To znamená, že **pipeline stage pro aktivně postupující issues může být nepřesný** — dashboard zobrazí poslední známý stage, ne nutně aktuální. Toto je akceptovatelné pro L1, protože:
- Zablokované issues (hlavní diagnostická hodnota) mají vždy přesný stage z block komentáře
- Dokončené issues mají PR link komentář
- Řešení pro L2: přidat checkpoint komentáře pro každý stage přechod do pipeline commands (`fix-bugs.md`, `fix-ticket.md`)

### 3.2 Git / Source Control (sekundární zdroj)

Data dostupná přes source control MCP:

| Data | MCP operace | Dostupné dnes |
|------|------------|---------------|
| Branches pro issue ID | List branches (match Branch naming pattern) | Ano |
| PR stav, URL, labels | List/search PRs | Ano |
| PR merge stav | PR detail | Ano |
| Commit count na branch | Git log (compare base..branch) | Ano |

### 3.3 Co je dostupné dnes vs. co potřebuje novou instrumentaci

**Dostupné dnes (ZERO nový kód):**
- Issue ID, název, stav, komentáře — z issue trackeru
- `[CLAUDE-agents]` komentáře — triage checkpoint + block comments
- Branches, PRs — ze source control
- Block důvody a doporučení — z block komentářů

**Potřebuje novou instrumentaci (pro plné statistiky):**
- **Časová data per stage** — dnes pipeline nepíše komentář při ZAHÁJENÍ každého stage, jen při UKONČENÍ triage a při blocku. Abychom změřili čas na stage, potřebovali bychom checkpoint komentáře pro každý stage.
  - **Řešení pro L1:** Odhadnout z timestampů existujících komentářů. Nepřesné, ale lepší než nic.
  - **Řešení pro L2 (budoucnost):** Přidat checkpoint komentáře do pipeline commands.
- **Token spotřebu** — dnes se loguje jen na konci pipeline (summary). Není uložena per issue.
  - **Řešení pro L1:** Vynechat z dashboardu.
- **Počet fixer iterací** — není explicitně logovaný do issue trackeru.
  - **Řešení pro L1:** Vynechat, nebo odhadnout z počtu komentářů fixeru v issue.

### 3.4 Datový model

Interní datová struktura pro dashboard:

```
DashboardData {
  project: string           // z Automation Config
  generated: ISO8601        // čas generování
  pluginVersion: string     // z plugin.json
  period: { from: date, to: date }

  issues: [
    {
      id: string
      title: string
      state: string         // In Progress | Blocked | For Review | Done | ...
      stage: string         // triage | analysis | fix | review | test | publish
      lastAgent: string     // jméno agenta z posledního komentáře
      lastUpdate: ISO8601   // timestamp posledního komentáře
      branch: string | null
      prUrl: string | null
      prNumber: number | null
      severity: string | null  // z triage checkpointu
      area: string | null      // z triage checkpointu
      blocked: {
        agent: string
        step: string
        reason: string
        detail: string
        recommendation: string
      } | null
    }
  ]

  timeline: [
    {
      timestamp: ISO8601
      issueId: string
      agent: string
      action: string       // parsovaný z komentáře
      type: "success" | "block" | "progress"
    }
  ]

  stats: {
    total: number
    active: number
    blocked: number
    completed: number
    successRate: number     // completed / total (kde total = completed + blocked + active)
    throughputPerDay: number
    blocksBySource: { [source: string]: number }  // source = agent name z block komentáře (vždy skutečný agent, ne "build")
  }
}
```

## 4. `/dashboard` Command Design

### Command definice

```markdown
---
description: Vygeneruje HTML dashboard se stavem pipeline
allowed-tools: mcp__*, Read, Glob, Grep, Bash
---
```

Allowed tools:
- `mcp__*` — issue tracker query, source control PR/branch query
- `Read` — CLAUDE.md, plugin.json
- `Glob/Grep` — hledání branch patterns
- `Bash` — zápis HTML souboru, otevření v prohlížeči

### Plná syntaxe

```
/CLAUDE-agents:dashboard [flags]
```

| Flag | Popis | Default |
|------|-------|---------|
| `--output <path>` | Cesta pro HTML výstup | `./reports/dashboard.html` (nebo z Automation Config) |
| `--open` | Automaticky otevři v prohlížeči po generování | false |
| `--format html\|json` | Výstupní formát | `html` |
| `--days <N>` | Časové rozmezí zpětně od dnešku | `30` (nebo z Automation Config) |

**L2 flagy** (odložené — budou přidány v L2 spolu s filter syntaxí):

| Flag | Popis | Default |
|------|-------|---------|
| `--filter <query>` | Filtr issues (key:value AND logika) | (všechny aktivní) |
| `--include-closed` | Zahrnout uzavřené/done issues | false (jen aktivní) |

**Poznámka k parsování flagů:** Flagy jsou LLM-parsed z volného textu `$ARGUMENTS`, nikoliv CLI-parsed. To znamená, že Claude Code interpretuje flagy z přirozeného jazyka. Edge cases (cesty s mezerami, speciální znaky) závisí na LLM úsudku. Pro L1 je to akceptovatelný trade-off — plugin je "pure markdown definitions" bez CLI parseru. Uživatelé by měli používat jednoduché cesty bez mezer a speciálních znaků.

> **[ROZHODNUTO]** L1 redukuje na 4 klíčové flagy (Varianta B): `--output`, `--open`, `--days`, `--format`. Flagy `--filter` a `--include-closed` odloženy na L2 — snižuje LLM parsing chybovost a `--filter` s vlastní key:value syntaxí je nejnáchylnější na chyby. `--include-closed` dává smysl až v kombinaci s `--filter`.

#### Filter syntaxe (L2)

> Tato sekce bude implementována v L2 spolu s flagy `--filter` a `--include-closed`.

Filter query používá jednoduchý key:value formát s AND logikou:

```
--filter "state:blocked"
--filter "stage:review"
--filter "agent:fixer"
--filter "state:blocked,stage:fix"      // AND: blocked AND ve fix stage
--filter "severity:HIGH"
```

Podporované filtry:

| Key | Hodnoty | Popis |
|-----|---------|-------|
| `state` | blocked, in-progress, for-review, done | Stav v issue trackeru |
| `stage` | triage, analysis, fix, review, test, publish | Pipeline stage |
| `agent` | triage-analyst, code-analyst, fixer, reviewer, test-engineer, publisher | Poslední agent |
| `severity` | LOW, MEDIUM, HIGH | Severity z triage |

### Command flow (krokový postup)

```
1. Načti Automation Config z CLAUDE.md
2. Načti Dashboard sekci (pokud existuje) pro defaults
3. Zpracuj argumenty (flags overridují config)
4. Zpracuj --filter (pokud zadán):
   a. State-level filtry (state:blocked, state:in-progress) → použij přímo v issue tracker query (krok 5)
   b. Post-fetch filtry (stage:*, agent:*, severity:*) → aplikuj po parsování komentářů (krok 7)
5. Query issue tracker:
   a. Aktivní issues (stavy z State transitions, s aplikovaným state filtrem z kroku 4a)
   b. Blocked issues (stav Blocked)
   c. Pokud --include-closed: uzavřené issues za --days období
6. Pro každý issue:
   a. Načti komentáře → parsuj [CLAUDE-agents] komentáře
   b. Určení pipeline stage (dle tabulky v sekci 3)
   c. Extrakce severity/area z triage checkpointu
   d. Extrakce block dat z block komentáře
7. Aplikuj post-fetch filtr (pokud --filter obsahuje stage/agent/severity)
8. Query source control:
   a. Pro každé (filtrované) issue ID → hledej branch dle Branch naming pattern
   b. Pro každý nalezený branch → hledej PR
9. Agregace statistik:
   a. Počty (aktivní/blocked/dokončené)
   b. Success rate
   c. Throughput
   d. Block distribuce dle agenta
10. Generování:
    a. --format html → vygeneruj HTML z dat
    b. --format json → serializuj datový model
11. Zápis do souboru
12. Pokud --open → otevři v prohlížeči
13. Zobraz summary v terminálu:
    "Dashboard vygenerován: {cesta} ({N} issues, {M} blocked, {K} dokončeno)"
```

**Optimalizace filtrování:** State-level filtry (`state:blocked`, `state:in-progress`) se aplikují na úrovni issue tracker query, čímž se vyhneme stahování dat pro stovky irrelevantních issues. Filtry vyžadující parsování komentářů (`stage:*`, `agent:*`, `severity:*`) se aplikují po parsování, ale ještě před query na source control (krok 8), aby se minimalizoval počet branch/PR lookupů.

## 5. HTML Generation

### Template approach

**Rozhodnutí: Raw string concatenation (template literals)**

Důvody:
- Plugin je "pure markdown definitions" — žádné dependencies, žádný build system
- Template engine (Handlebars, EJS) by přidal dependency a build step
- HTML generuje command přímo — je to markdown soubor s instrukcemi pro Claude Code
- Claude Code umí generovat HTML napřímo z dat bez pomocných knihoven
- Jednoduchost > elegance pro L1

Alternativa (zamítnutá):
- Externí HTML template soubor v plugin repu (`templates/dashboard.html`) s placeholders
- Problém: command by musel číst template soubor z pluginu, což je nepraktické (plugin cesta se mění dle instalace)

**Přístup:** Command obsahuje přesný popis HTML struktury a stylu. Claude Code generuje HTML přímo do string a zapíše na disk.

### JavaScript: minimální inline JS

Žádné frameworky, žádné externí knihovny. Pouze vanilla JS pro:

1. **Sortování tabulky** — klik na hlavičku sloupce → seřadit asc/desc
2. **Filtrování** (client-side) — textový input pro rychlé filtrování tabulky
3. **Expander pro blocked detail** — zobrazit/skrýt technický detail
4. **Sticky header** — hlavička tabulky se drží při scrollu

Příklad sort funkce (~ 30 řádků JS):

```javascript
document.querySelectorAll('th[data-sort]').forEach(th => {
  th.addEventListener('click', () => {
    const table = th.closest('table');
    const tbody = table.querySelector('tbody');
    const rows = Array.from(tbody.querySelectorAll('tr'));
    const col = th.cellIndex;
    const asc = th.dataset.order !== 'asc';
    th.dataset.order = asc ? 'asc' : 'desc';
    rows.sort((a, b) => {
      const va = a.cells[col].textContent.trim();
      const vb = b.cells[col].textContent.trim();
      return asc ? va.localeCompare(vb) : vb.localeCompare(va);
    });
    rows.forEach(row => tbody.appendChild(row));
  });
});
```

Celkový rozsah JS: < 100 řádků.

Všechno je inlinované v jednom `.html` souboru:

| Klíč | Popis | Typ | Default | Příklady |

**Output path:**
- Relativní cesta k repo root
- Adresář se vytvoří automaticky (mkdir -p)
- Soubor se přepíše při každém generování
- Flag `--output` overriduje tuto hodnotu

**Include closed:**
- Formát: `last N days` nebo `all`
- Určuje, jak daleko zpět se hledají uzavřené issues
- Flag `--days` overriduje tuto hodnotu
- `all` = bez časového omezení (pozor na performance)

**Auto-open:**
- Pokud `true`, dashboard se po generování automaticky otevře v default prohlížeči
- Flag `--open` overriduje (vždy otevře)
- Na headless serveru se ignoruje s warningem

**Refresh hint:**
- `manual` — dashboard zobrazuje "Refresh: spusťte /dashboard znovu"
- `after-publish` — dashboard zobrazuje "Refresh: automaticky po /publish" (nápověda pro integraci s post-publish hookem)
- `hourly` — dashboard zobrazuje "Doporučený refresh: 1x za hodinu" (nápověda, neřídí automaticky)
- Toto je jen textová nápověda v dashboardu — L1 nemá automatický refresh

## 7. Refresh & Aktualizace

### Static snapshot model

Dashboard L1 je **statický snapshot** — jednorázově vygenerovaný HTML soubor. Data se NEaktualizují automaticky.

Pro čerstvý pohled je nutné:
1. Spustit `/CLAUDE-agents:dashboard` znovu
2. Refresh prohlížeče (F5) po regeneraci

Dashboard sám zobrazuje:
- **Timestamp generování** — v headeru i patičce
- **Staleness indikátor** — JavaScript snippet, který porovnává čas generování s aktuálním časem:

```javascript
const generated = new Date('2026-02-27T14:32:05Z');
const now = new Date();
const hours = Math.floor((now - generated) / 3600000);
const indicator = document.getElementById('staleness');
if (hours < 1) {
  indicator.textContent = 'Aktuální (< 1 hodina)';
  indicator.className = 'badge badge-green';
} else if (hours < 24) {
  indicator.textContent = hours + ' hodin starý';
  indicator.className = 'badge badge-yellow';
} else {
  indicator.textContent = Math.floor(hours/24) + ' dní starý — doporučujeme refreshnout';
  indicator.className = 'badge badge-red';
}
```

### Commitování dashboardu do repa?

**Doporučení: NE (default), ale nebráníme tomu.**

Pro:
- Historické snapshoty — git log ukazuje vývoj pipeline v čase
- Sdílení bez serveru — kolega si pullne a otevře

Proti:
- Binary-like churn — každá regenerace = velký diff v gitu
- Velikost repa roste zbytečně
- `.gitignore` je čistší řešení

**Doporučení:** Přidat `reports/` do `.gitignore`. Pokud tým chce archivovat, ať commitne explicitně.

Dashboard bude v patičce obsahovat nápovědu:
```
Tip: Pro archivaci dashboardu commitněte soubor do repa.
Pro vynechání přidejte 'reports/' do .gitignore.
```

### Integrace s post-publish hookem

Tým může nakonfigurovat automatické regenerování dashboardu po každém publish:

```markdown
### Hooks
| Klíč | Hodnota |
|------|---------|
| Post-publish | /CLAUDE-agents:dashboard --output ./reports/dashboard.html |
```

Tato integrace je dobrovolná. Pipeline ji neřídí — je to jen hook konfigurace.

### Napojení na budoucí Level 2

Dashboard L1 pokládá základy pro Level 2 (live monitoring):

1. **Datový model** (sekce 3.4) — stejný model může být serializován jako JSON a konzumován realtime serverem
2. **HTML šablona** — Level 2 může znovupoužít layout a CSS, jen přidat SSE/WebSocket update
3. **JSON formát** — `--format json` umožňuje externím nástrojům číst data bez parsování HTML
4. **Refresh hint** — příprava na automatické refreshe (Level 2 přepíše "manual" na "realtime")
5. **`[CLAUDE-agents]` komentáře** — stabilní formát pro parsování, Level 2 pouze přidá real-time monitoring nad stejným zdrojem

Co Level 2 bude potřebovat navíc (a L1 NEPŘIPRAVUJE):
- Event emitting z commands (JSONL stream)
- Background server process
- SSE/WebSocket komunikace
- In-memory state management

| Uživatel říká... | Směruj na | Proč |
|-------------|-----------|------|
| "status" / "stav" | `/status` | Rychlý pohled v terminálu (existující chování) |
| "dashboard" / "přehled" / "report" | `/dashboard` | HTML export s plnými daty |
| "jak to vypadá" / "overview" | `/status` (default) | Obecný dotaz → rychlý terminal output |
| "vygeneruj report" / "exportuj" | `/dashboard` | Explicitně chce soubor |

Rozhodovací logika:

```
Pokud intent obsahuje "dashboard", "report", "přehled", "zpráva", "exportuj":
  → /dashboard

Pokud intent obsahuje "status", "stav", "jak to vypadá":
  → /status (stávající chování)

Pokud nejednoznačné:
  → Zeptej se: "Chcete rychlý přehled v terminálu (/status) nebo HTML dashboard (/dashboard)?"
```

### Aktualizace SKILL.md

Do Intent Mapping tabulky přidat:

```markdown
| Vygeneruj dashboard/report | `CLAUDE-agents:dashboard` | Flags | Yes |
```

Dashboard je označen jako **destructive** (zapíše soubor na disk), takže skill bude po uživateli žádat potvrzení před spuštěním.

## 11. Rizika & edge cases

### Riziko 1: Velký počet issues — performance

**Popis:** Pokud projekt má stovky nebo tisíce issues, query na issue tracker může být pomalá a výsledný HTML soubor příliš velký.

**Pravděpodobnost:** Střední (velké projekty s dlouhou historií)

**Dopad:** Pomalé generování (minuty), velký HTML soubor (> 500 KB), pomalý browser rendering.

**Mitigace:**
- Default `--days 30` omezuje časové rozmezí
- Paginace v issue tracker query (max 100 issues per request)
- HTML paginace: zobrazit 50 issues, "Zobrazit další" tlačítko
- Warning při > 200 issues: "Velký počet issues. Zvažte užší časové rozmezí (--days)."
- `--filter` umožňuje zúžit scope

### Riziko 2: Chybějící data — částečné pipeline runy

**Popis:** Některé issues mohly projít pipeline před instalací CLAUDE-agents (žádné `[CLAUDE-agents]` komentáře). Nebo pipeline byla přerušena manuálně.

**Pravděpodobnost:** Vysoká (obzvláště při prvním spuštění)

**Dopad:** Neúplné data v dashboardu, chybějící pipeline stage, prázdné blocked detaily.

**Mitigace:**
- Pipeline stage "unknown" pro issues bez `[CLAUDE-agents]` komentářů
- Graceful degradation: pokud komentář chybí, zobraz "—" místo chyby
- Staleness indikátor: issues bez recentního komentáře označit jako "stale"
- Dashboard explicitně uvede: "Pipeline data dostupná pouze pro issues zpracované CLAUDE-agents pluginem."

### Riziko 3: Cross-browser kompatibilita

**Popis:** Dashboard používá inline CSS a vanilla JS. Starší prohlížeče nebo specifické prohlížeče mohou renderovat jinak.

**Pravděpodobnost:** Nízká (modern CSS features jsou dobře podporované)

**Dopad:** Rozpadlý layout, nefunkční sort/filtr.

**Mitigace:**
- CSS custom properties: podporované od Chrome 49, Firefox 31, Safari 9.1, Edge 15
- CSS Grid/Flexbox: podporované od 2017+
- `prefers-color-scheme`: Chrome 76+, Firefox 67+, Safari 12.1+
- Testovat v Chrome, Firefox, Safari (hlavní tři)
- Fallback pro staré prohlížeče: bez dark mode (světlý režim default)
- Žádné experimentální CSS features (žádný :has(), container queries, etc.)

### Riziko 4: Citlivá data — názvy issues, PR obsah

**Popis:** Dashboard obsahuje issue titulky, PR linky, block důvody. Pokud se sdílí mimo tým, mohou uniknout citlivé informace.

**Pravděpodobnost:** Střední (dashboard je lokální soubor, ale může být commitnutý nebo odeslán emailem)

**Dopad:** Únik interních informací (security bugy, business logika).

**Mitigace:**
- Dashboard se generuje lokálně — neodesílá se na externí server
- Doporučení v dokumentaci: "Dashboard obsahuje issue titulky a technické detaily. Sdílejte jen v rámci týmu."
- `.gitignore` doporučení pro `reports/` adresář
- Budoucí rozšíření: `--redact` flag pro anonymizaci issue titulků (nahradit "Login crash" → "PROJ-42")

### Riziko 5: MCP server různorodost — různé issue trackery

**Popis:** Plugin podporuje různé issue trackery (YouTrack, Jira, GitHub Issues, Linear). Každý má jiný MCP server s jiným API.

**Pravděpodobnost:** Vysoká (různí uživatelé, různé trackery)

**Dopad:** Dashboard command musí fungovat s libovolným MCP serverem. Různé query formáty, různé response struktury.

**Mitigace:**
- Command čte `Type` z Automation Config a používá odpovídající MCP server (stejný princip jako existující commands)
- `[CLAUDE-agents]` komentáře jsou format-agnostic — nezáleží na trackeru
- Pipeline stage rekonstrukce záleží na komentářích, ne na tracker-specifických datech
- Testovat minimálně s YouTrack (primární) a jedním dalším (GitHub Issues)

### Riziko 6: Zastaralý dashboard — uživatel zapomene refreshnout

**Popis:** Dashboard je statický snapshot. Pokud uživatel otevře den starý dashboard, vidí zastaralá data.

**Pravděpodobnost:** Vysoká (lidé si zvyknou otvírat bookmark)

**Dopad:** Rozhodnutí na základě zastaralých dat. PM si myslí, že 3 issues jsou blocked, ale 2 už jsou opravené.

**Mitigace:**
- Staleness indikátor (JavaScript) — červené upozornění pokud > 24 hodin starý
- Timestamp v headeru i patičce
- Nápověda: "Data ke dni {datum}. Pro aktuální stav spusťte /dashboard znovu."
- Post-publish hook integrace — automaticky regenerovat po každém publish
- Budoucnost: Level 2 (live monitoring) řeší tento problém úplně

### Riziko 7: Souběžné spuštění více dashboardů

**Popis:** Dva uživatelé spustí `/dashboard` současně a píšou do stejného souboru.

**Pravděpodobnost:** Nízká (jednorázové generování, krátkodobé)

**Dopad:** Poškozený HTML soubor.

**Mitigace:**
- Každý uživatel může použít `--output` s vlastní cestou
- Zápis je atomický (write celého souboru najednou)
- V praxi nepravděpodobné — dashboard se generuje v sekundách

## 12. Otevřené otázky

```
Q: Máme pro Issue Table použít kompletní tabulku se 7 sloupci, nebo zjednodušení se 4 sloupci (ID, Název, Stav, Stage)?
→ Navrhovaná odpověď: Kompletní tabulka se 7 sloupci na desktopu, zjednodušená (ID, Stav, Stage) na mobilu.
  Důvod: Desktop má dostatek místa, uživatel očekává plný detail. Na mobilu je horizontální
  scroll nepříjemný — lepší je zobrazit jen klíčové informace a detaily dát do expandovatelného
  řádku. Toto je běžný responsive pattern (klik na řádek → rozbalí detail kartu).
```

```
Q: Jak často bude typický uživatel generovat dashboard? Jednou denně? Po každém pipeline runu?
→ Navrhovaná odpověď: Po každém pipeline runu (via post-publish hook) + 1x denně manuálně.
  Důvod: Post-publish hook je set-and-forget — dashboard se regeneruje automaticky při každém
  publish. Navíc PM nebo QA si může spustit manuálně, když chce aktuální stav. Toto je konzistentní
  s "static snapshot" modelem — generuj často, protože je to levné (žádná infrastruktura).
```

```
Q: Má JSON formát (--format json) exportovat stejný datový model jako interní reprezentace, nebo zjednodušený?
→ Navrhovaná odpověď: Stejný datový model (sekce 3.4), bez zjednodušení.
  Důvod: JSON formát je určený pro programatické konzumenty (Grafana, custom skripty, CI/CD).
  Ti potřebují plná data. Pokud chtějí subset, mohou filtrovat sami (jq, Python, etc.).
  Zjednodušení by znamenalo udržovat dva formáty — zbytečná komplexita.
```

```
Q: Má dashboard zobrazovat odhad token spotřeby per issue, nebo je to mimo scope L1?
→ Navrhovaná odpověď: Mimo scope L1. Vynechat.
  Důvod: Token spotřebu dnes pipeline loguje jen jako celkový odhad v summary (viz fix-bugs.md
  krok 9). Není uložena per issue v issue trackeru. Abychom ji zobrazili, museli bychom
  přidat novou instrumentaci do pipeline commands — to je práce navíc s nízkou hodnotou pro L1.
  Zvážit pro Level 2, kde event emitting může zahrnovat token metriky.
```

```
Q: Jak řešit situaci, kdy Automation Config neobsahuje Dashboard sekci — použít hard-coded defaults, nebo vynutit konfiguraci?
→ Navrhovaná odpověď: Hard-coded defaults. Dashboard sekce je opcionální.
  Důvod: To je konzistentní se všemi ostatními opcionálními sekcemi (Retry Limits, Hooks,
  Worktrees, etc.). Pokud sekce chybí, command použije defaults (output: ./reports/dashboard.html,
  days: 30, auto-open: false). Vynucování konfigurace by bylo breaking change (MAJOR verze)
  a zbytečně zvyšovalo onboarding friction. Flags overridují defaults — power users nepotřebují
  config.
```

```
Q: Má dashboard obsahovat link "otevřít v issue trackeru" u každého issue, nebo jen zobrazit ID?
→ Navrhovaná odpověď: Link na issue tracker. Issue ID je klikatelný odkaz.
  Důvod: Dashboard je přehledový nástroj — uživatel ho používá k identifikaci problému a pak
  chce přejít na detail. Klikatelný link je minimální effort (URL = Instance + /issue/ + ID
  pro YouTrack, nebo jiný pattern pro Jira/GitHub) a maximální hodnota. URL pattern je odvoditelný
  z Automation Config (Instance + Project).
```

```
Q: Máme přidat do dashboardu sekci "Doporučení" (co by měl tým dělat jako další krok)?
→ Navrhovaná odpověď: Ano, jednoduchých 3–5 řádků na konci dashboardu.
  Důvod: Dashboard je diagnostický nástroj. Pokud zobrazí jen data, uživatel musí sám vyvodit
  závěry. Jednoduchá sekce "Doporučení" může říct:
  - "3 issues jsou blocked — zkontrolujte je a rozhodněte o dalším postupu"
  - "Fixer blokuje 40% issues — zvažte úpravu retry limitů nebo dekompozici složitých bugů"
  - "Průměrná doba ve fázi test roste — zkontrolujte test infrastrukturu"
  Toto je low-effort, high-value přidání. Pravidla pro generování doporučení mohou být
  jednoduchá (threshold-based), ne AI-generated.
```

```
Q: Má /dashboard command být non-destructive (jako /status) nebo destructive (zapíše soubor)?
→ Navrhovaná odpověď: Destructive — zapíše soubor na disk.
  Důvod: Command zapíše HTML/JSON soubor na disk, což je side effect. V terminologii
  bug-workflow skillu je to "destructive" operace — skill bude žádat potvrzení před spuštěním.
  To je správné, protože uživatel by měl vědět, že se zapíše soubor (a kde). Alternativa
  "jen zobrazit v terminálu" je už pokrytá přes /status.
```

```
Q: Jak řešit lokalizaci — dashboard v češtině nebo angličtině?
→ Navrhovaná odpověď: Čeština jako default (konzistentně s ostatními commands a docs), bez lokalizačního systému.
  Důvod: Všechny existující commands (status.md, check-setup.md, onboard.md, etc.) jsou v češtině.
  Dashboard je interní nástroj pro český tým. Angličtina by byla nekonzistentní. Pokud v budoucnu
  bude potřeba lokalizace, lze přidat Language klíč do Dashboard config sekce — ale to je
  scope pro budoucí verze, ne pro L1. Agent definitions zůstanou v angličtině (konzistentně
  s existující konvencí), ale command output a HTML jsou v češtině.
```

```
Q: Má dashboard ukazovat pipeline pro feature issues (z Fáze 1), nebo jen bug-fix?
→ Navrhovaná odpověď: Oba — dashboard je agnostický vůči typu pipeline.
  Důvod: Dashboard čte [CLAUDE-agents] komentáře a issue tracker stavy. Nezáleží na tom, jestli
  issue je bug nebo feature — formát komentářů je stejný. Pokud Fáze 1 (Feature Pipeline) používá
  stejný [CLAUDE-agents] prefix a Block Comment Template, dashboard ho automaticky zobrazí.
  To je další argument pro implementaci dashboardu před Fází 1 — dashboard bude připravený
  pro feature pipeline bez jakýchkoliv změn.
```

```
Q: Jak prezentovat statistiku "průměrný čas na stage", když nemáme přesné timestamps pro zahájení každého stage?
→ Navrhovaná odpověď: Odhadnout z časových rozdílů mezi po sobě jdoucími komentáři. Označit jako "odhad".
  Důvod: Dnes pipeline píše komentář při ukončení triage (checkpoint) a při blocku. Další
  komentáře (od publisher, reviewer, etc.) jsou zapisovány do issue trackeru. Časový rozdíl
  mezi nimi je přibližný čas strávený na stage. Není to přesné (komentář může být zapsán až
  po dokončení stage, ne přesně na začátku), ale pro L1 je to dostatečné. Dashboard zobrazí
  "~8 min (odhad)" místo "8 min (přesně)". V Level 2 s event emitting budou časy přesné.
```

```
Q: Má se Custom CSS path načítat za běhu (při otevření HTML) nebo při generování (inlinovat)?
→ Navrhovaná odpověď: Odloženo na L2. Custom CSS path byl odstraněn z L1 scope (viz I5 review nález).
  Pokud se v L2 přidá, odpověď je: při generování — inlinovat do HTML (zachová single-file princip).
```
