## Motivace

CLAUDE-agents v3.0 pokrývá: bug-fix pipeline, feature pipeline, scaffold, decomposition engine, dashboard. Plugin je v ekosystému Claude Code pluginů unikátní — žádný jiný nástroj nekombinuje issue-tracker-to-PR pipeline se specializovanými agenty, rollback, hooks a zero dependencies.

Tento dokument definuje unified release v3.1, který vyčerpá potenciál pure-markdown pluginu. Všechny features (intelligence, community, autonomie) jdou do jednoho releasu — žádný z nich nemá breaking config change, takže jde o jeden MINOR bump.

### Proč jeden release místo tří

- Žádní external uživatelé (marketplace deferred) → granulární semver je overhead bez příjemce
- Většina features je vzájemně nezávislá → lze implementovat v libovolném pořadí
- Jeden release = jeden changelog, jeden tag, kompletní produkt
- Cross-version dependencies se zjednodušují na interní dependencies

### Kde CLAUDE-agents vede (a nikdo jiný to nemá v kombinaci)

1. Issue tracker jako kanonický vstup (YouTrack+, rozšiřitelné přes MCP)
2. 6-stage sequential pipeline (triage → analysis → fix → review → test → publish)
3. 13 specializovaných agentů s model selection per stage
4. Rollback agent jako first-class participant
5. Block comment protokol + `/resume-ticket` (checkpoint-based recovery)
6. 4 hook injection points + custom agents
7. Scaffold pipeline (stack-selector → scaffolder → validate → git init)
8. Decomposition s configurable fail/commit strategy
9. Pure markdown, zero dependencies, installable jako plugin
10. Generic Automation Config contract — 100% project-agnostic

## Strategický přístup

**v3.1:** Vyčerpat potenciál pure-markdown pluginu — každá feature, která JDE udělat bez runtime. Intelligence, analytics, community readiness, multi-tracker — vše v jednom releasu.

**Runtime (future):** Companion projekt `claude-agents-runtime` pro features, které vyžadují persistent state, web server, paralelizaci na OS úrovni. Rozhodnutí o startu po 3.1.16 Scalability Assessment.

### Architektonické pravidlo

Nový agent jen pokud má UNIKÁTNÍ ROLI, kterou existující agent nemůže převzít rozšířením promptu. V celém v3.1 je jen 1 nový agent: `priority-engine`.

### Versioning

| Level | Trigger |
|-------|---------|
| MAJOR | Breaking change v Automation Config — nový required key, přejmenovaná sekce |
| MINOR | Nová backward-compatible feature — nový optional key, nový command/agent |
| PATCH | Behavior fix bez contract change |

**v3.1.0 = MINOR** — nové features přidávají optional config, optional commands, nebo dokumentaci. Žádný breaking change.

## v3.1 Features

### A — Pipeline Enhancement

#### 3.1.1 — Parallel Batch Execution

**Co:** Worktree-based paralelní zpracování ticketů v `/fix-bugs`.

**Jak:**
- `/fix-bugs N` spustí N ticketů, každý ve vlastním git worktree
- Pipeline per ticket je nezávislá (triage → ... → publish)
- Orchestrace: command vytvoří worktrees, spustí Agent tool paralelně, sbírá výsledky
- Merge strategy: každý ticket = vlastní branch → vlastní PR (žádné cross-ticket dependencies)

**Config (existující sekce Worktrees):**
```markdown
### Worktrees (optional)
| Klíč | Hodnota |
|------|---------|
| Batch size | 3 |
| Base path | `.worktrees/` |
| Cleanup | auto |
```

**Aktuální stav:** `fix-bugs` command UŽ obsahuje worktree orchestrační logiku a Worktrees config sekci. Zbývá: ověřit reálné paralelní spuštění Agent tool, result collection, error coordination across worktrees, cleanup.

**Impact:** HIGH — 3-5x rychlejší batch processing.
**Effort:** LOW-MEDIUM

#### 3.1.2 — Pipeline Profiles

**Co:** Konfigurovatelné profily pro různé typy úkolů — skip/add stages podle potřeby.

**Proč:**
- Typo fix nepotřebuje triage + code analysis
- Critical production bug potřebuje full pipeline + E2E testy
- Refactoring nepotřebuje triage, ale potřebuje enhanced review
- One-size-fits-all pipeline je neefektivní

**Config:**
```markdown
### Pipeline Profiles (optional)
| Profil | Skip stages | Extra stages |
|--------|-------------|-------------|
| fast | triage, code-analyst | — |
| strict | — | e2e-test-engineer |
| minimal | triage, code-analyst, test-engineer | — |
```

**Použití:**
- `/fix-ticket ISSUE-123 --profile fast`
- `/implement-feature ISSUE-456 --profile strict`

**Poznámka:** Profily se aplikují na oba pipeline typy (fix i feature). Stage-skip list může být odlišný per pipeline — např. "fast" pro feature pipeline skipuje spec-analyst, zatímco pro bug pipeline skipuje triage.

**Impact:** HIGH — flexibilita, šetří čas a tokeny.
**Effort:** MEDIUM

#### 3.1.3 — Fix Verification Stage

**Co:** Volitelný post-publish verifikační krok — ověření, že fix skutečně opravil bug.

**Jak:**
- Po merge PR + deploy zkontroluje, jestli se bug reprodukuje
- Implementace přes nový optional config klíč `Verify` v Build & Test sekci
- Pokud verifikace selže → issue se znovu otevře s komentářem

**Proč je to kritické:**
- Aktuální pipeline: triage → ... → publish → ???
- Nikdo neověřuje, jestli fix SKUTEČNĚ opravil bug v produkci/staging
- Closed-loop verification je unikátní differentiator — žádný konkurent to nemá

**Config:**
```markdown
### Build & Test
| Klíč | Hodnota |
|------|---------|
| Build | `dotnet build` |
| Test | `dotnet test` |
| Verify | `npm run verify-fix` |
```

**Impact:** HIGH — closed-loop feedback. Buduje trust v pipeline.
**Effort:** LOW

### B — Analytics & Intelligence

#### 3.1.4 — Pipeline Metrics & Agent Effectiveness

**Co:** Command `/metrics` — sbírá data z issue trackeru + git, generuje markdown report s pipeline-level i per-agent analytics.

**Pipeline metriky:**
- Success rate (issues fixed / issues attempted)
- Průměrný čas na fix (od start po merged PR)
- Breakdown by stage (kolik issues blocked v triage vs. fixer vs. test)
- Top blocked reasons (z `[CLAUDE-agents]` komentářů)
- Token cost estimate per run (z modelu a počtu agentů)

**Per-agent metriky:**
- Success rate per stage (kolikrát stage prošel vs. blokoval)
- Průměrný počet iterací fixer↔reviewer
- Top důvody selhání per agent (z `[CLAUDE-agents]` blocked komentářů)
- Failure pattern detection: "65% bloků v test-engineer stage → top důvod: chybějící test fixtures"

**Jak:**
- Nový command `/metrics` — query issue tracker přes MCP, parsuje `[CLAUDE-agents]` komentáře, čte git log
- Výstup: markdown report (`metrics-report.md`) nebo stdout
- Žádný nový agent — command přímo používá MCP tools

**Config:**
```markdown
### Metrics (optional)
| Klíč | Hodnota |
|------|---------|
| Output | `./reports/metrics.md` |
| Period | last 30 days |
```

**Vztah k `/dashboard`:** Dashboard (v3.0) vizualizuje aktuální stav pipeline. Metrics analyzuje historická data (success rate, trendy, failure patterns). Long-term: dashboard by měl zobrazovat metrics data jako sekci.

**Proč je effectiveness tracking kritický:**
- V pure markdown pluginu jsou agent prompty CELÝ produkt
- Bez měření effectiveness nemůžeš cíleně zlepšovat prompty
- Failure analysis feeduje zpátky do prompt improvement — skutečný competitive advantage

**Impact:** HIGH — první plugin s pipeline analytics. Žádný konkurent to nemá.
**Effort:** LOW

#### 3.1.5 — Cost Estimation (Range-Based)

**Co:** Command `/estimate ISSUE-ID` — predikuje token usage PŘED spuštěním pipeline.

**Jak:**
- Přečte issue z trackeru → odhadne velikost (LOC affected, počet souborů)
- Na základě historických dat (z `/metrics`, pokud dostupné) nebo heuristiky spočítá expected tokens
- **Range estimate** (ne point estimate):
  - Best case: 1 fixer iterace, žádná decomposition
  - Typical: 2-3 fixer iterace
  - Worst case: max iterations + decomposition
- Výstup: tabulka s odhadem tokenů a ceny per stage (best / typical / worst)
- Respektuje Pipeline Profiles (3.1.2) — pokud uživatel specifikuje profil, estimate reflektuje skipped stages

**Proč range místo point estimate:**
- Token usage závisí na: složitost issue, velikost codebase, počet fixer iterací (1-5), zda se spustí decomposition
- Point estimate s ±30% accuracy poškodí trust. Range je upřímný.

**Impact:** MEDIUM — unikátní feature. Pomáhá s budget planning.
**Effort:** LOW

#### 3.1.6 — AI Backlog Prioritization

**Co:** Command `/prioritize` — analyzuje backlog a navrhne pořadí fixů.

**Jak:**
- Přečte otevřené issues z trackeru
- Pro každý issue odhadne: impact (kolik uživatelů/modulů ovlivňuje), risk (jak kritická oblast kódu), effort (velikost change), dependencies (co musí být fixnuto první)
- Pokud jsou dostupná metrics data (z 3.1.4), použije historické success rate a failure patterns
- Výstup: ranked list s doporučeným pořadím a zdůvodněním
- Nový agent: `priority-engine` (opus) — potřebuje opus pro komplexní reasoning přes celý backlog

**Proč nový agent (výjimka z architektonického pravidla):**
- Žádný existující agent nedělá cross-issue analýzu. Všechny jsou per-issue.
- Priority engine potřebuje vidět CELÝ backlog najednou — jiná role než triage-analyst (který vidí jeden issue).

**Impact:** HIGH — unikátní feature. Žádný konkurent to v pluginu nemá.
**Effort:** MEDIUM

#### 3.1.7 — Historical Context for Code Analyst

**Co:** Rozšíření `code-analyst` agenta o sekci "Relevant Past Fixes".

**Jak:**
- Code analyst přečte posledních 10 commitů v dotčených souborech
- Parsuje `[CLAUDE-agents]` blocked komentáře pro stejný modul/area
- Předá context fixeru: "tento soubor měl 3 bugy v posledním měsíci, pattern: off-by-one errors v pagination"

**Proč NE nový agent:**
- Code analyst UŽ čte codebase a mapuje impact zone
- Přidání historického kontextu je rozšíření jeho expertízy, ne nová role
- Nový agent by přidal zbytečný context window overhead

**Impact:** MEDIUM — vyšší success rate fixeru díky kontextu z minulých fixů.
**Effort:** LOW

### C — Multi-Tracker & Ecosystem

#### 3.1.8 — Multi-Tracker Support

**Co:** End-to-end podpora GitHub Issues + Jira + Linear jako first-class trackery vedle YouTrack.

**Aktuální stav:** Config klíč `Type` v Issue Tracker sekci existuje od v1.6 a publisher agent ho podporuje. ALE: reálně bylo testováno jen s YouTrack přes MCP. Pro ostatní trackery chybí:
- End-to-end ověření celé pipeline (triage → ... → publish) s non-YouTrack trackerem
- `check-setup` validace per tracker type (správné MCP tool names, state transition syntax)
- MCP configuration guide per tracker (jaký MCP server, jaké env vars)

**Co se změní:**
- Agents: žádná změna (čtou data z issue, nezáleží odkud přijdou)
- Commands: rozšíření `check-setup` o validaci per tracker type
- Docs: MCP configuration guide per tracker

**Config příklad (GitHub):**
```markdown
### Issue Tracker
| Klíč | Hodnota |
|------|---------|
| Type | github |
| Instance | `github.com` |
| Project | `owner/repo` |
| Bug query | `is:issue is:open label:bug` |
| State transitions | In Progress: `add label:in-progress`, Blocked: `add label:blocked` |
```

**Impact:** HIGH — většina potenciálních uživatelů je na GitHub.
**Effort:** MEDIUM

#### 3.1.9 — Example Library & Templates

**Co:** Adresář `examples/` s ukázkami + command `/template STACK` pro rychlé generování Automation Config.

**Struktura:**
```
examples/
├── configs/                  ← Pre-built Automation Config templates
│   ├── github-nextjs.md
│   ├── github-python-fastapi.md
│   ├── github-dotnet.md
│   ├── gitea-spring-boot.md
│   ├── jira-react.md
│   └── youtrack-python.md
├── custom-agents/            ← Ukázkové custom agenti pro hooks
│   ├── security-analyst.md
│   ├── dependency-analyst.md
│   ├── migration-reviewer.md
│   └── compliance-checker.md
└── mcp-configs/              ← .mcp.json příklady per tracker
    ├── github.json
    ├── jira.json
    ├── linear.json
    └── youtrack.json
```

**Template command:**
- `/template nextjs` → vygeneruje kompletní Automation Config pro Next.js + GitHub
- `/template list` → seznam dostupných templates
- Rozšíření `/onboard` — nabídne template jako starting point

**Proč examples místo core agentů:**
- Security scanning, dependency checking, migration review — to jsou use-casy pro CUSTOM AGENTS hook systém
- Přidávat je do core = bloat. Dát je jako examples = ukazuje flexibilitu hook systému

**Impact:** HIGH — snižuje barrier to entry. Nový uživatel za 1-5 minut nakonfiguruje plugin.
**Effort:** LOW

#### 3.1.10 — Plugin Composability

**Co:** Dokumentace + guardrails pro koexistenci s jinými pluginy.

**Jak:**
- Namespace prefix `CLAUDE-agents:` na všech commands (už máme)
- Dokumentace: "jak používat CLAUDE-agents vedle jiných pluginů"
- Conflict detection v `/check-setup` — varování pokud jiný plugin registruje stejné command names
- Doporučení pro custom agents: namespace konvence

**Impact:** LOW-MEDIUM — prevence problémů.
**Effort:** LOW

### D — Developer Experience

#### 3.1.11 — Config Migration

**Co:** Command `/migrate-config` — upgraduje Automation Config ze starší verze na novou.

**Jak:**
- Detekuje aktuální verzi configu (z přítomnosti/nepřítomnosti sekcí)
- Navrhne změny pro upgrade (nové optional sekce, přejmenované klíče)
- Uživatel schválí → command aplikuje změny

**Proč je to důležité:**
- Při budoucím upgrade (v3 → v4) — co se stane s user's Automation Config?
- Bez migration path = frustrovaní uživatelé = ztráta community trust

**Impact:** MEDIUM — insurance pro budoucí breaking changes.
**Effort:** LOW

#### 3.1.12 — Automated Test Harness

**Co:** Automatické testy pro plugin samotný (ne pro uživatelské projekty).

**Jak:**
- Mock MCP server (jednoduchý bash/node script co vrací předpřipravené responses)
- Test runner: spustí command s mock MCP, ověří expected output/behavior
- Scenáře: happy path, triage block, fixer retry, reviewer reject, test fail, publish success
- CI: Gitea Actions workflow co spustí testy při push

**Proč je to důležité:**
- Plugin s 22 commands a 13 agenty (po v3.1) bez automatických testů je risk
- Pro community adoption je automated testing prerequisite pro důvěru
- Dnes máme jen manuální smoke testy v `tests/`

**Impact:** HIGH pro reliability a community trust.
**Effort:** HIGH — mock MCP server je netriviální. Ale jednorázová investice.

### F — Strategic Assessment

#### 3.1.16 — Pure Markdown Scalability Assessment

**Co:** Architektonické posouzení, zda pure markdown orchestrace ještě škáluje pro budoucí features.

**Posouzení by mělo odpovědět:**
- Je 22+ commands v pure markdown ještě udržitelné?
- Kde je break-even point pro přechod na hybrid (markdown prompty + code orchestrace)?
- Měl by runtime převzít orchestraci, zatímco markdown zůstane jen pro agent prompty?
- Jaký je context window overhead pluginu? (13 agentů + 22 commands + routing skill = kolik tokenů se loaduje do Claude Code session?)

**Impact:** Strategické rozhodnutí pro long-term architekturu. Prerekvizita pro runtime rozhodnutí.
**Effort:** LOW — research, žádná implementace.

### Continuous Process: Prompt Quality Iteration

Průběžný proces (ne jednorázová feature):
- Po release analyzovat per-agent effectiveness data z `/metrics`
- Identifikovat nejslabší agenty a cíleně zlepšovat prompty

- Mechanismus: review `/metrics` outputu, manuální analýza per-agent failure patterns, cílené prompt edity v `agents/*.md`, měření efektu

## Runtime Platform (Companion Project: `claude-agents-runtime`)

**DŮLEŽITÉ:** Runtime NENÍ verze pluginu. Je to samostatný projekt s vlastním repem a semver. Plugin zůstává na svém semver. Rozhodnutí o startu závisí na výsledku 3.1.16 Scalability Assessment.

**Architektura:**
```
┌─────────────────────────────────────────┐
│  CLAUDE-agents plugin (pure markdown)    │
│  Commands + Agents + Skills              │
│  = jádro pipeline orchestrace            │
└──────────────┬──────────────────────────┘
               │ events (JSONL)
               │
┌──────────────▼──────────────────────────┐
│  claude-agents-runtime (Node.js/Python)  │
│                                          │
│  ├─ Live Dashboard (SSE/WebSocket)       │
│  ├─ Webhook Server (příjem triggerů)     │
│  ├─ Fleet Orchestrator (parallel VMs)    │
│  ├─ Web Approval UI (diff viewer)        │
│  ├─ Persistent Store (SQLite)            │
│  ├─ CLI Companion (claude-agents run)    │
│  └─ Multi-Repo Coordinator              │
└─────────────────────────────────────────┘
```

**Klíčový princip:** Pure markdown plugin zůstává jádrem. Runtime je OPTIONAL companion.

**PROVISIONAL:** Všechny detaily runtime jsou aspirační vize. Finální design závisí na výsledku 3.1.16 Scalability Assessment.
