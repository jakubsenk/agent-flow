### K1: Automation Config — celková struktura po v2.0

Po sloučení všech designů bude Automation Config obsahovat:

```markdown

| Klíč | Povinný | Zdroj |
|------|---------|-------|
| Type | ano (default: youtrack) | v1.6 — NOVÉ |
| Instance | ano | stávající |
| Project | ano | stávající |
| Bug query | ano | stávající |
| State transitions | ano | stávající |
| On start set | ano | stávající |

### Source Control
(beze změny)

### PR Rules
(beze změny — PR description template vyčlenit jako podsekci?)

### Build & Test
(beze změny)

### Retry Limits (optional)           ← v1.7 NOVÉ
| Klíč | Default |
|------|---------|
| Fixer iterations | 5 |
| Test attempts | 3 |
| Build retries | 3 |

### Hooks (optional)                  ← v2.0 NOVÉ
| Klíč | Hodnota |
```

**Rozhodnutí:** `/check-setup` (v1.3) a `/onboard` (v1.9) musí znát CELOU tuto strukturu. Design v1.9 se doplní o otázky pro nové sekce (Type, Retry Limits, Hooks, Custom Agents, Notifications).

### K2: fix-bugs.md — pořadí nových kroků

Aktuální pipeline:
```
1. Načti bugy (query)
2. Pro každý bug → triage
3. Code-analyst
4. Fixer
5. Build
6. Reviewer ⟲ max 5x
7. Test-engineer ⟲ max 3x
8. Publisher
9. Summary
```

Po sloučení všech designů:
```
0.  [DRY-RUN CHECK] — pokud --dry-run, běž jen kroky 1-3, pak report   ← v1.5
1.  Načti bugy (query) — čti retry limity z config                      ← v1.7
2.  Pro každý bug → triage (čti Type z config)                          ← v1.6
3.  Code-analyst
3a. [PRE-FIX HOOK] — spusť Pre-fix z Hooks                             ← v2.0
4.  Fixer (retry limit z config)                                        ← v1.7
5.  Build (retry limit z config)                                        ← v1.7
5a. [POST-FIX HOOK] — spusť Post-fix z Hooks                           ← v2.0
5b. [POST-FIX CUSTOM AGENT] — spusť Post-fix agent z Custom Agents     ← v2.0
6.  Reviewer ⟲ (limit z config)                                         ← v1.7
7.  Test-engineer ⟲ (limit z config)                                    ← v1.7
7a. [E2E TEST-ENGINEER] — pokud E2E Test config existuje                ← stávající
7b. [PRE-PUBLISH HOOK]                                                   ← v2.0
7c. [PRE-PUBLISH CUSTOM AGENT] — spusť Pre-publish agent               ← v2.0
8.  Publisher
8a. [POST-PUBLISH HOOK]                                                  ← v2.0
8b. [WEBHOOK] — notifikace PR created                                   ← v2.1
9.  Summary (strukturovaná tabulka + token odhad)                        ← v1.5 + v2.1
9a. [WEBHOOK] — notifikace pipeline-complete                            ← v2.1
X.  [BLOCK HANDLER] — při blocku od fixer/reviewer/test-engineer:       ← v1.6 + v2.1
    rollback-agent + webhook issue-blocked
    (NEROLOVAT při blocku od triage/code-analyst — žádné git změny)
```

**Opravy vs původní verze:**
- Přidán krok 7a (E2E test-engineer — stávající, chyběl)
- Přidán krok 7b (Pre-publish custom agent — CHYBA-7 z review)
- Block handler specifikuje, že rollback jen po fixer/reviewer/test (VAROVANI-2)

**Rozhodnutí:** Implementovat jako jeden koherentní dokument, ne postupné patche.

### K3: fix-ticket.md — unified flow

```
0.  [DRY-RUN CHECK] — pokud --dry-run, běž jen kroky 1-3, pak report   ← v1.5
1.  Nastav issue v trackeru (čti Type z config)                          ← v1.6
2.  Vytvoř branch
3.  Triage-analyst (čti Type z config)                                   ← v1.6
4.  Code-analyst
4a. [PRE-FIX HOOK]                                                       ← v2.0
5.  Fixer (retry limit z config)                                         ← v1.7
6.  Build (retry limit z config)                                         ← v1.7
6a. [POST-FIX HOOK]                                                      ← v2.0
6b. [POST-FIX CUSTOM AGENT]                                              ← v2.0
7.  Reviewer ⟲ (limit z config)                                          ← v1.7
8.  Test-engineer ⟲ (limit z config)                                     ← v1.7
8a. [E2E TEST-ENGINEER] — pokud E2E Test config existuje                 ← stávající
8b. [PRE-PUBLISH HOOK]                                                   ← v2.0
8c. [PRE-PUBLISH CUSTOM AGENT]                                           ← v2.0
9.  Publisher (uživatel rozhodne)
9a. [POST-PUBLISH HOOK]                                                  ← v2.0
9b. [WEBHOOK] — notifikace PR created                                   ← v2.1
9c. [TOKEN ODHAD] — zobraz odhadovanou spotřebu tokenů                  ← v2.1
X.  [BLOCK HANDLER] — při blocku od fixer/reviewer/test-engineer:       ← v1.6 + v2.1
    rollback-agent:
      - v worktree: git reset --hard {base_branch}
      - v CWD: git stash (ochrana necommitted user work) + git reset --hard {base_branch}
        (fixer commity jsou záměrně zahozeny resetem, stash chrání jen uživatelskou práci)
    + webhook issue-blocked
    (NEROLOVAT při blocku od triage/code-analyst — žádné git změny)
```

**Rozdíly oproti fix-bugs:** žádný batch, žádná summary tabulka, žádné worktrees. Publisher volaný manuálně (uživatel rozhodne).

### K4: Block Comment Template vs Checkpoint komentář

**Block Template (při selhání):**
```
[CLAUDE-agents] 🔴 Block
Agent: {název}
Krok: {krok kde selhalo}
Důvod: {max 2 věty}
Detail: {technický output}
Doporučení: {co by měl člověk udělat}
```

**Checkpoint komentář (při úspěchu triage):**
```
[CLAUDE-agents] ✓ Checkpoint: Triage
Severity: {severity}
Area: {area}
```

**Rozhodnutí:** Konzistentní prefix `[CLAUDE-agents]` pro oba. Různý formát pro různý účel.

**Jak se template dostane k agentům:** Commands předávají Block Template agentům jako context instruction (stejný mechanismus jako retry limity). Agenti samotní se NEMĚNÍ — template reference je v commands.

## 4. Sdílené konvence (unified)

### Command argument parsing

Všechny commands používají `$ARGUMENTS` jako prostý string. Konvence:
- Flags: `--dry-run`, `--skip-build` (double-dash prefix)
- Positional: issue ID, počet bugů (první argument)
- Instrukce v commandu: "Pokud $ARGUMENTS obsahuje `--flag`, aktivuj mód X."

### Model selection

**Commands nemají model v frontmatter** — model selection je vlastnost agentů, ne commands. Tabulka níže uvádí **doporučený model pro agenta**, ne command:

| Nový agent | Model | Důvod |
|-----------|-------|-------|
| rollback-agent | haiku | Mechanický (git reset + komentář) |

Commands (check-setup, resume-ticket, status, onboard, changelog, version-check) nemají model — běží v aktuální Claude Code session.

### Automation Config formát

**Unifikovaný formát:** Všechny sekce používají tabulkový formát (`| Klíč | Hodnota |`). Žádné odrážkové seznamy v config sekcích.

### Jazyk

- Agent definice: **anglicky** (stávající konvence)
- Commands: **česky** (stávající konvence — všech 6 existujících commands je česky)
- Skills: **anglicky** (stávající konvence — bug-workflow/SKILL.md je anglicky)
- Dokumentace (docs/, README): **česky**
- CHANGELOG.md: **česky** (Keep a Changelog formát)
