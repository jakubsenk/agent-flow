# Phase 8 Security Adversary Report — ceos-agents v6.8.0

## Methodology

I attacked the v6.8.0 implementation across the three feature surfaces (Autopilot, Observability Hooks, Cost Visibility) assuming a hostile-but-realistic operator and an unknown downstream project consuming the plugin. I probed: webhook payload leakage (what PII / code / file paths escape), `Webhook URL` injection surface (`file://`, `gopher://`, redirects, DNS hangs), `curl --max-time 5` semantics, `--dangerously-skip-permissions` risk disclosure, lock-directory and log-file symlink attacks, config-field shell injection, state.json atomicity under crash, and `run_id` / `issue_id` path traversal. I read the full Phase 4 spec (requirements.md, design.md, formal-criteria.md), the implementation in `skills/autopilot/SKILL.md`, `core/post-publish-hook.md` Section 4, `core/state-manager.md`, `core/config-reader.md`, `state/schema.md`, `skills/fix-ticket/SKILL.md` (webhook fire sites), `skills/fix-bugs/SKILL.md`, `skills/metrics/SKILL.md`, `skills/dashboard/SKILL.md`, `docs/guides/autopilot.md`, and `CLAUDE.md`, then cross-checked spec obligations against implementation. I also ran `./tests/harness/run-tests.sh` (139/140 pass; the 1 failure is a version-bump artifact unrelated to security).

## Findings

### [SECURITY-FINDING-1] severity=MEDIUM

**Location:** `skills/autopilot/SKILL.md:16` and `skills/autopilot/SKILL.md:22`, `skills/autopilot/SKILL.md:353`
**Description:** The skill file tells operators to invoke Autopilot with `claude -p "Run /ceos-agents:autopilot" --dangerously-skip-permissions` three times but never discloses what `--dangerously-skip-permissions` actually means, what the blast radius is, or what mitigations are appropriate (e.g., runtime sandbox, container, limited POSIX user, outbound firewall). Phase 0 `verify.md` asks explicitly: *"Does the `--dangerously-skip-permissions` guidance document the risk in the skill file itself?"* It does not. The guide at `docs/guides/autopilot.md` also uses the flag four times (lines 78, 88, 110, 126, 138, 147) with no risk note. A cron-scheduled operator running this verbatim grants the child skills full unattended Bash + Edit + Write + Skill tool power with zero prompting.
**Attack scenario:** Operator copy-pastes the crontab line verbatim; a triage-analyst / fixer agent processing a poisoned issue description (prompt-injection payload from the tracker) executes arbitrary Bash because permissions were pre-skipped. The blast radius includes `git push`, filesystem writes anywhere the cron user can reach, and outbound network via any tool the child skills may invoke. The `core/external-input-sanitizer.md` boundary markers are the only defense — but those are advisory string markers, not sandbox isolation.
**Remediation:** Add a `## Security / Permission Model` section to `skills/autopilot/SKILL.md` explaining (1) what `--dangerously-skip-permissions` disables, (2) the recommended containment (dedicated user, rootless container, outbound allowlist), and (3) that the tracker MCP output flows unfiltered (beyond marker wrapping) into opus-powered fixers. Cross-link to `docs/guides/autopilot.md` and add the same note next to every invocation example.

### [SECURITY-FINDING-2] severity=MEDIUM

**Location:** `docs/reference/config.md:53-81` and `CLAUDE.md:144-189`
**Description:** Design spec `3.6` explicitly requires: *"Add operator-trust note for Webhook URL (mirrors CLAUDE.md note in f)"* in `docs/reference/config.md`, and an *"operator-trust note under Notifications"* in `CLAUDE.md`. Neither file contains the specified note: `The Webhook URL value is dispatched via curl without scheme/host validation. Operators are responsible for restricting this value to trusted internal observability endpoints.` A grep for `without scheme`, `trusted internal`, `SSRF`, `loopback`, and `operator-trusted` against both files returns zero matches. The forward-compat paragraph is present in `CLAUDE.md:189`, but the operator-trust / no-scheme-validation warning is missing.
**Attack scenario:** An operator reading CLAUDE.md or config.md to pick a `Webhook URL` never sees that the URL is passed raw to `curl` without scheme checking. They might paste `file:///etc/passwd` (accidentally), a cloud metadata URL like `http://169.254.169.254/`, or an internal-loopback URL, and the plugin will faithfully dispatch a POST to it on every pipeline lifecycle event. Because Known Limitation §8.4 declares no SSRF guard, documentation becomes the sole mitigation — and the documentation is missing.
**Remediation:** Add the exact paragraph from design.md §3.6(f) to both `CLAUDE.md` (under `## Webhook Payloads` or a new sub-section) and to `docs/reference/config.md` (in `## Notifications`). This is a documentation-only fix but is load-bearing under the WONTFIX (§6.21) decision.

### [SECURITY-FINDING-3] severity=MEDIUM

**Location:** `core/post-publish-hook.md:107-111`, `skills/fix-ticket/SKILL.md:93`, `skills/fix-bugs/SKILL.md:102`
**Description:** Webhook invocation is `curl --max-time 5 --retry 0 -X POST -H "Content-Type: application/json" --data-binary @- "${Webhook_URL}" <<EOF`. No `--proto "=https"` or `--proto "=http,https"` restriction, no `-L` (so 3xx redirects are NOT followed — good), and no `--resolve` pinning. Combined with SECURITY-FINDING-2, this means a malicious or typo `Webhook URL` like `file:///sensitive/path`, `gopher://internal-smtp:25/`, or `dict://127.0.0.1:11211/` will be delivered to curl's scheme handlers. While curl will not *exfiltrate file contents via POST* to `file://` (POST to file is a no-op), it will still exercise protocol handlers that could hit internal services (`gopher://`, `scp://`, `smb://` depending on curl build).
**Attack scenario:** An operator inadvertently sets `Webhook URL: http://169.254.169.254/latest/meta-data/` (cloud metadata service). Every pipeline lifecycle event GETs (POST-to-GET is common for metadata) that URL with no SSRF guard. Exfiltration of `run_id`, `issue_id`, `pr_url` values to the metadata service logs is minor; the real concern is that the payload construction and endpoint reach are entirely operator-controlled with zero plugin-side validation.
**Remediation:** Either (1) add `--proto "=http,https"` to the curl invocation in `core/post-publish-hook.md` Section 3 and Section 4 (1-line change, zero behaviour impact for legitimate HTTPS endpoints), or (2) explicitly document at §8.4 that `--proto` restriction is WONTFIX per the operator-trusted model. Option 1 is preferable: it costs nothing and removes the `file://`/`gopher://` vectors without extra config.

### [SECURITY-FINDING-4] severity=MEDIUM

**Location:** `skills/fix-ticket/SKILL.md:93,170` etc. (all 3 new webhook fire sites in all 4 pipeline skills) and `core/post-publish-hook.md:107`
**Description:** The JSON payload is constructed via bash heredoc with variable interpolation: `{"event":"pipeline-started","run_id":"${run_id}","issue_id":"${issue_id}", ...}`. `issue_id` flows unsanitized from the tracker MCP (the input is then classified via `core/external-input-sanitizer.md` ONLY for agent prompts, not for webhook payloads). If an attacker controls the issue title or ID field in a tracker (e.g., a malicious reporter on a public GitHub instance), a quote character in `issue_id` (rare but not impossible: Jira/YouTrack project keys are constrained, but GitHub `#123` has no colons, and Gitea uses `owner/repo#123` — a slash is already live). A literal `"` in `issue_id` would break the JSON payload; worse, a crafted value like `PROJ-42","extra":"injected` would inject arbitrary JSON keys into the payload — polluting the webhook consumer's schema contract.
**Attack scenario:** An attacker with write access to a tracker issue title changes it to `PROJ-42","outcome":"success","pr_url":"evil","x":"x`. If the publisher / pipeline derives the webhook `issue_id` from the tracker title (it shouldn't, but no validation is enforced in the skill), the emitted payload becomes `{"event":"pipeline-started","run_id":"...","issue_id":"PROJ-42","outcome":"success","pr_url":"evil","x":"x",...}` — a structured-data injection that could mislead a downstream observability consumer.
**Remediation:** Require JSON-encoding of `issue_id`, `pr_url`, `step_name`, and `run_id` before heredoc interpolation. Document this in `core/post-publish-hook.md` Section 4 curl pattern as "field values MUST be JSON-encoded via `jq -Rn` or equivalent before interpolation". Alternative: constrain `issue_id` to `[A-Z]+-[0-9]+` or the tracker's declared regex in config-reader.md. The AC-10 regex `^[A-Z]+-[0-9]+_[0-9]{8}T[0-9]{6}Z$` validates `run_id` shape but not `issue_id` in isolation.

### [SECURITY-FINDING-5] severity=LOW

**Location:** `skills/autopilot/SKILL.md:53` (config declaration) and spec `design.md §3.1` Step 4
**Description:** `Log file` config key defaults to `.ceos-agents/autopilot.log` and is appended to at end of run ("summary append to log file" per design.md §3.1). No implementation code is shown in the skill — but the declared behavior is "append-only ISO-8601 summary line per run". If an attacker with write access to the repo's `.ceos-agents/` directory (or the configured `Log file` path) replaces `autopilot.log` with a symlink (e.g., `ln -sf /etc/passwd .ceos-agents/autopilot.log`), then every Autopilot run will append (not truncate, since `>>` mode) adversary-chosen text to the symlink target. Append-mode does NOT truncate the file, but it does write ~1 line of operator-controlled text per run to `/etc/passwd` or any other path the Autopilot user can reach.
**Attack scenario:** Multi-tenant cron host where user A writes `.ceos-agents/` and user B reads it — if user A symlinks `autopilot.log` to a file user B trusts (or to a shared config), user B's system accumulates garbage. On single-tenant systems this is a non-issue.
**Remediation:** Before appending, the skill (or an implementation step in the SKILL.md body) should check `[ ! -L "$LOG_FILE" ]` or use `O_NOFOLLOW` semantics (e.g., Python `open(..., "a")` with `os.O_NOFOLLOW` flag). At minimum, document this in `docs/guides/autopilot.md#troubleshooting` as an operator-responsibility note. The autopilot.log path is operator-trusted, so this is LOW.

### [SECURITY-FINDING-6] severity=LOW

**Location:** `skills/autopilot/SKILL.md:123` (`LOCK_DIR="$(pwd)/.ceos-agents/autopilot.lock"`) and `rm -rf "$LOCK_DIR"` invocations at lines 165, 178, 192, 210
**Description:** The lock uses `mkdir` (atomic, symlink-safe on the create path), but the stale-recovery path does `rm -rf "$LOCK_DIR"` when it decides the existing lock is stale or malformed. If an attacker plants a symlink *inside* `.ceos-agents/autopilot.lock/` pointing to `../../etc` before Autopilot's `rm -rf` runs, the recursive delete would follow the symlink. Note: `rm -rf` on modern coreutils does NOT follow symlinks by default for the *directory argument itself*, but it DOES traverse into contained directories. The owner.json parsing uses `grep -o` on a potentially adversarial file — LOW but non-zero risk.
**Attack scenario:** Adversarial process on the same host creates `.ceos-agents/autopilot.lock/owner.json` with `acquired_at` 200 minutes ago AND a symlink `.ceos-agents/autopilot.lock/target -> /etc`. Autopilot declares the lock stale and calls `rm -rf .ceos-agents/autopilot.lock/` which traverses and deletes `/etc` contents — catastrophic.
**Remediation:** Use `find "$LOCK_DIR" -xdev -depth -delete` with `--one-file-system` semantics, or explicitly check `[ ! -L "$LOCK_DIR/owner.json" ]` before treating the file as valid JSON. Modern `rm -rf` on GNU coreutils 8.25+ resists this specific vector, but Windows Git Bash and BusyBox may not. Document this or add a defensive `find -type l -delete` pass before `rm -rf`.

### [SECURITY-FINDING-7] severity=LOW

**Location:** `core/post-publish-hook.md:57-98` (payload definitions)
**Description:** I probed whether the three new webhook payloads leak PII, issue descriptions, code diffs, or internal paths. They do NOT: the payloads carry only `event`, `run_id`, `issue_id`, `pipeline`, `timestamp`, `step_name`, `duration`, `iteration_count`, `status`, `outcome`, `pr_url`. No raw issue description, no file paths with usernames, no code diffs. This is GOOD design (contrast with typical over-shared payloads). The only field that could carry operator-sensitive info is `pr_url` which leaks the repository host+path — but that is expected and documented as part of the pipeline-completed contract.
**Attack scenario:** Minimal. A malicious webhook receiver gathers `{run_id, issue_id, pr_url}` tuples, which is metadata the operator already consciously chose to share.
**Remediation:** None needed for v6.8.0. Flag as a POSITIVE finding in the commander aggregation (minimal disclosure surface). The ONLY residual concern is `run_id = "{issue_id}_{timestamp}"` which embeds the `issue_id` — so enforcement of a safe `issue_id` regex (see SECURITY-FINDING-4 and -8) is the consolidated fix.

### [SECURITY-FINDING-8] severity=MEDIUM

**Location:** `skills/fix-ticket/SKILL.md:87,89` (`run_id` generation) and `core/state-manager.md:23` (`.ceos-agents/{RUN-ID}/state.json` path derivation)
**Description:** `run_id = "{issue_id}_{YYYYMMDDTHHMMSSZ}"` and the state.json path is derived as `.ceos-agents/{ISSUE-ID}/state.json` (note: state.json uses `ISSUE-ID`, not `run_id` as directory — per core/state-manager.md:23 and schema.md). There is NO validation step anywhere that constrains `issue_id` to a safe character set (e.g., `[A-Za-z0-9._-]+`). A maliciously crafted issue_id `../etc` or `../../somewhere/state.json` would cause the state.json write to escape the intended directory. Spec Known-Limitation §8.1 addresses `run_id` UNIQUENESS caveats but NOT input validation. AC-10's regex `^[A-Z]+-[0-9]+_[0-9]{8}T[0-9]{6}Z$` is applied only to the webhook payload assertion, not as a validation gate at pipeline start.
**Attack scenario:** A hostile tracker admin creates an issue with key `../../../../tmp/evil` (most trackers won't permit this, but some tracker types like Gitea or GitHub use numeric IDs like `#123` that get substituted client-side; a MCP layer could accept arbitrary string). The fix-ticket skill sees `issue_id = "../../../../tmp/evil"`, constructs `.ceos-agents/../../../../tmp/evil/state.json`, and writes pipeline state outside the repo. Paired with SECURITY-FINDING-6 (`rm -rf` on lock), this becomes a privilege-bypass path.
**Remediation:** Add an explicit issue-id validation step in every pipeline skill's Step 0 (MCP pre-flight): `issue_id =~ ^[A-Za-z0-9._-]+$` — block with `[ceos-agents] 🔴 Pipeline Block` if not. Or canonicalise via `basename` before any filesystem interpolation. Document in `core/state-manager.md` "Input Contract" that `run_id` / `ISSUE-ID` MUST match `^[A-Za-z0-9._-]+$`.

### [SECURITY-FINDING-9] severity=LOW

**Location:** `core/state-manager.md:29-31` (atomic write protocol)
**Description:** The tmp+rename protocol is sound on POSIX — `write .tmp, rename to .json` is atomic if same-filesystem. Docs note "best-effort on Windows" (line 30). On Windows Git Bash, rename-over-existing-file is NOT atomic pre-Windows-10; a crash between tmp-write and rename *does* leave a stale .json (safe) or a partial .tmp (which `state-manager.md:166` handles via the `.state.json.corrupt.{ts}` rename). The risk of corruption mid-write is LOW. However, the schema.md:465 notes retry once after 100ms on rename failure — no exponential backoff; a persistent race could lose state silently.
**Attack scenario:** Concurrent access by two pipeline runs on the same ISSUE-ID (explicitly disallowed in state-manager.md:168 "last-write-wins — acceptable"). Not an attack surface, but a data-integrity note.
**Remediation:** Documented behaviour is acceptable for v6.8.0. Consider upgrading the retry to 2-3 attempts with backoff in v6.9.0.

## Top 3 Risks

1. **`--dangerously-skip-permissions` undisclosed risk surface** (SECURITY-FINDING-1, MEDIUM) — the skill invites operators to run with permissions disabled in cron. No sandboxing guidance, no containment recommendation, no pointer to what permissions are being waived. Poisoned tracker content flows to opus-powered fixers with write-anywhere authority. Fix is documentation-only but blocks a responsible v6.8.0 ship.
2. **`run_id` / `issue_id` path traversal unvalidated** (SECURITY-FINDING-8, MEDIUM) — no regex gate on `issue_id` before it is interpolated into `.ceos-agents/{ISSUE-ID}/state.json` path construction. Combined with SECURITY-FINDING-6's `rm -rf` on lock-dir, a malicious tracker ID like `../../etc` could escape the pipeline's directory jail. Trivial to fix (`issue_id =~ ^[A-Za-z0-9._-]+$` validation in Step 0) but currently absent.
3. **Operator-trust note missing from CLAUDE.md / config.md** (SECURITY-FINDING-2, MEDIUM) — spec design.md §3.6 mandated this paragraph; implementation dropped it in both files. Because §8.4 / §6.21 explicitly WONTFIX'd SSRF guard, the operator-trust note IS the v6.8.0 mitigation — and it is missing. Documentation-only fix; load-bearing under the operator-trusted threat model.
