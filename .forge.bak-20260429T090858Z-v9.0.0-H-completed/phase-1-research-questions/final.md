## Questions

1. **Question:** In the v8.0.0 codebase today, what concrete evidence exists that absent machine-readable I/O contracts cause observable failures — specifically: (a) do `tests/scenarios/frontmatter-completeness.sh`, `read-only-agents.sh`, and `section-order.sh` contain stale pre-v8 agent names that would mask contract violations, and (b) in the forge-2026-04-25-001 archive (219/62/15 harness results), how many of the 62 failures are traceable to output section name mismatch or structural inconsistency rather than logic errors?
   **Why it matters:** Anchors the "do nothing" baseline in measurable data from the actual project. If the 62 failures are unrelated to output shape and existing tests cover all structural contracts, the cost-benefit of formalization is weak. If output-shape failures are present but undetected by the harness, the case for contracts is concrete and quantifiable. This is the primary WHETHER gate.
   **Source hint:** `tests/scenarios/frontmatter-completeness.sh`, `tests/scenarios/read-only-agents.sh`, `tests/scenarios/section-order.sh` (inspect for stale agent lists); `.forge/` archive for forge-2026-04-25-001 (failure attribution); v8.0.0 commit history.
   **Cluster:** agent-3 Q1 + agent-2 Q11

2. **Question:** When Claude Code's Task tool dispatches an agent, what validation (if any) does the runtime apply to the agent's output before returning it to the calling skill — specifically: does it enforce section presence, frontmatter fields, or output token patterns, or does it return the raw LLM response verbatim — and does the Task tool pass the full agent file (frontmatter + body) verbatim as the system prompt, or does it strip or reformat frontmatter before model invocation?
   **Why it matters:** Determines whether adding `## Inputs` / `## Outputs` body sections is visible to the model at dispatch time (frontmatter stripping would make embedded YAML invisible), and whether explicit contracts buy runtime enforcement or are purely static/documentation-level. If the Task tool returns raw text with no enforcement, the formalization choice shifts entirely to lint-time validation. Core HOW and WHETHER gate simultaneously.
   **Source hint:** Anthropic Claude Code documentation on the Task tool (docs.anthropic.com/claude-code), `/agents` command reference; inspect `skills/fix-bugs/SKILL.md` Task tool call shape for what the invocation passes; compare Anthropic Tool Use API `output` field contract (platform.openai.com analogy).
   **Cluster:** agent-3 Q2 + agent-1 Q3

3. **Question:** What structured output contract format do the Model Context Protocol (MCP) `inputSchema`/`outputSchema` fields and CrewAI `expected_output` / `output_pydantic` Task fields use — specifically: are contracts co-located with the agent/tool definition or in sidecar files, are they mandatory or optional, and what happened to backward compatibility when CrewAI added `output_json` / `output_pydantic` to existing crew definitions in v0.28?
   **Why it matters:** MCP is the closest structural analog to ceos-agents (markdown-adjacent tool definitions dispatched by an orchestrator). CrewAI's addition of structured output fields to an existing natural-language-first framework is the closest procedural analog (18 agents, existing definitions must work unmodified). Together they provide concrete evidence for the co-location pattern (HOW: same-file vs. sidecar) and whether such an addition was MINOR/additive or required MAJOR migration.
   **Source hint:** MCP specification at `modelcontextprotocol.io/specification`, MCP `schema/2024-11-05/schema.json`; CrewAI changelog (github.com/crewAIInc/crewAI), `crewai/task.py` `output_json`/`output_pydantic` fields, CrewAI migration guide for v0.28.
   **Cluster:** agent-2 Q2 + agent-3 Q9

4. **Question:** What is the minimal schema expressiveness actually needed to describe the I/O surface of polymorphic ceos-agents agents like `analyst` (`--phase triage` vs `--phase impact`, each with distinct inputs and output sections) and `test-engineer` (`--e2e` variant) — and does JSON Schema 2020-12's `oneOf`/`if-then-else` add enough expressive power over a typed field list (markdown table: field name + type + required/optional) to justify the readability overhead in a file that must remain human-readable by a Claude model?
   **Why it matters:** The HOW schema-language decision. If the most complex agents' I/O surface can be described adequately with a simple typed-list table, full JSON Schema complexity in a markdown file is not warranted. If mode-dispatch polymorphism genuinely requires discriminated-union schemas, a typed list is structurally insufficient and a richer schema or sidecar file is needed. Bounds schema complexity from real contract requirements rather than theoretical completeness.
   **Source hint:** JSON Schema 2020-12 spec (json-schema.org), `agents/analyst.md` `## Phase Dispatch` section, `agents/test-engineer.md` for `--e2e` variant; LangChain `StructuredTool` Pydantic field types in production tool definitions; MCP `inputSchema` patterns in published MCP servers.
   **Cluster:** agent-1 Q4 + agent-2 Q12

5. **Question:** In production agent frameworks, what has driven the choice between "advisory schema" (declared contract, validation skipped at runtime, LLM output accepted as-is) and "enforced schema" (validation runs in the hot path, output rejected if malformed) — specifically, what was smolagents' rationale for making `output_type` advisory, and what did OpenAI's introduction of `strict: true` in function-calling (August 2024) reveal about failure modes that strict enforcement surfaces in previously-working tool definitions?
   **Why it matters:** ceos-agents must decide enforcement posture. If advisory schema is the production consensus for prose-generating agents (because LLM non-determinism makes hard enforcement impractical), this strengthens the "static lint only" option and weakens the case for runtime validation hooks in skills. If `strict: true` surfaced real previously-undetected failures, the enforcement investment is justified. Directly informs HOW validation mechanism and whether new sections should be opt-in or mandatory across all 18 agents.
   **Source hint:** smolagents `smolagents/tools.py` `Tool` base class (github.com/huggingface/smolagents); OpenAI structured outputs announcement blog (August 2024), OpenAI API reference `tools[].function.strict`, OpenAI developer forum threads on `strict: true` edge cases.
   **Cluster:** agent-2 Q5 + agent-2 Q10

6. **Question:** Does the `core/agent-override-injector.md` mechanism append override files strictly verbatim after all existing agent body sections (making any new `## Inputs` / `## Outputs` body sections additive and invisible to existing override files) — or does any skill inspect agent body structure before injection — and if new contract section headings (e.g., `## Inputs`) collide with heading names that consuming projects' `customization/{agent-name}.md` files currently use, what does the model receive and how do production LLM systems handle duplicate markdown heading context?
   **Why it matters:** The hard backward-compat constraint: v8.0.0 `customization/` override files MUST work unmodified. If the injector is append-only and verbatim, new sections are safe unless heading names collide with override content. Heading collision creates ambiguous model context even without breaking the injector. The design must either reserve heading names or choose names unlikely to appear in override files. This determines WHERE new sections must live in the agent body and what names are safe to use.
   **Source hint:** `core/agent-override-injector.md` (read injection mechanism); `skills/fix-bugs/SKILL.md` and `skills/fix-ticket/SKILL.md` Task tool call sites; `examples/` directory for sample override files; `agents/fixer.md` and `agents/reviewer.md` for output block names (`## Fix Report`, `## Code Review`) that overrides might reference; Anthropic prompt engineering docs on system prompt structure with duplicate headings.
   **Cluster:** agent-1 Q6 + agent-1 Q7 + agent-3 Q3

7. **Question:** If `## Inputs` / `## Outputs` sections are added to all 18 agent files in v9.0.0, what is the minimal migration action required for a v8.0.0 consuming project (e.g., BIFITO pilot, drmax-readmine-test) — specifically: do existing `customization/{agent-name}.md` override files need any modification, or are they passively forward-compatible because the override injector appends them after all base sections without inspecting section names — and does the answer change if contracts are defined as optional (absent = uncontracted) versus mandatory?
   **Why it matters:** Determines whether a `docs/guides/migration-v8-to-v9.md` is needed (per `feedback_docs_coverage.md` discipline) and what it must contain. Zero-touch passive compatibility supports a MINOR bump; required override modifications demand MAJOR and a migration guide. Also determines whether the "opt-in contracts" design avoids migration entirely for v8.0.0 projects.
   **Source hint:** `core/agent-override-injector.md` (append mechanism); MEMORY entries for `project_bifito_autopilot_test.md` and `project_drmax_readmine_test.md` (check for referenced custom override files); `docs/guides/migration-v7-to-v8.md` for existing migration guide pattern.
   **Cluster:** agent-3 Q5

8. **Question:** Under ceos-agents' Versioning Policy (CLAUDE.md: "agent output format contract changes that external tooling/Agent Overrides may parse = MAJOR; adding optional config sections = MINOR"), does adding `## Inputs` / `## Outputs` sections to all 18 agent files constitute a MAJOR or MINOR increment — and what do LangChain's v0.1→v0.2 tool schema breaking-change policy and LangGraph's TypedDict state schema additive-field behavior provide as external precedent for classifying "new parseable section added to existing definition, no existing section renamed or removed"?
   **Why it matters:** Directly determines the version number and all associated doc/changelog/tag work. The MEMORY allocates v9.0.0 to sub-projekt H — verifying whether that allocation is semantically correct (MAJOR) or whether the CLAUDE.md Versioning Policy itself needs an amendment to cover a "new structured contract section" category is a gate before Phase 6 can plan commit order. A misclassification creates a semver lie for downstream consumers.
   **Source hint:** CLAUDE.md `## Versioning Policy` table; git log for v7→v8 and v6→v7 bump commits to understand prior MAJOR trigger precedent; LangChain changelog (github.com/langchain-ai/langchain), v0.2 breaking-change announcement (May 2024); LangGraph `StateGraph` checkpointing docs and GitHub issues on state schema migration (langchain-ai/langgraph).
   **Cluster:** agent-1 Q8 + agent-3 Q4 + agent-2 Q1 + agent-2 Q6

9. **Question:** In the 297 existing test scenarios, how many assert properties of agent body sections (Goal, Expertise, Process, Constraints content) versus frontmatter only — and which specific scenarios (`v8-agents-analyst-shape.sh`, `section-order.sh`, `read-only-agents.sh`) would require modification if a new mandatory body section (`## Inputs` / `## Outputs`) were added to all 18 agents, versus which would be unaffected?
   **Why it matters:** The test-impact question that determines the edit surface of the existing harness before the HOW decision is made final. If tests assert exact section order or section count, adding new body sections may break them — inverting the backcompat constraint from agent files to the test suite itself. If 90%+ of tests only grep frontmatter, body additions are near-zero-impact. This is a codebase-inspection question answerable without running the harness.
   **Source hint:** `tests/scenarios/` directory — inspect `v8-agents-analyst-shape.sh`, `section-order.sh`, `read-only-agents.sh`, `frontmatter-completeness.sh`, `v8-agents-enumeration.sh` for body vs. frontmatter scope of assertions; grep for `awk '/^## /'` patterns to find body-section assertions.
   **Cluster:** agent-1 Q10

10. **Question:** What is the minimal bash assertion pattern — using the existing harness primitives (`grep -qE`, `awk`, `wc -l`, `diff -q`, no jq/yq/Python) — that can distinguish a structurally valid contract section (correct heading present, required subfields present with correct names) from a malformed one (heading present but subfields absent or misspelled) in a new scenario file; and what SKIP-guard pattern should that scenario use so it exits 77 on v8.0.0 agent files lacking the section, fails on v9.0.0 files with malformed sections, and passes on v9.0.0 files with valid sections?
    **Why it matters:** The HOW-to-test question that is a hard design input: if the chosen contract format (YAML block, typed table, structured list) cannot be validated with grep/awk, the format must be simplified to grep-friendly syntax, or a new test dependency (jq/yq) must be introduced. The SKIP-guard pattern determines whether new scenarios can be merged before agent files are updated without breaking the CI green bar. Both are load-bearing for the test strategy in Phase 6.
    **Source hint:** Inspect `tests/scenarios/v8-agents-analyst-shape.sh` for file-existence SKIP guard pattern; `tests/harness/run-tests.sh` for exit code 77 = SKIP handling; `tests/scenarios/section-order.sh` for `awk '/^## Process/{found=1}...'` generalization candidates; POSIX awk field-splitting capabilities for structured field parsing.
    **Cluster:** agent-1 Q9 + agent-3 Q6 + agent-3 Q7

11. **Question:** What grep-based cross-reference assertion could validate that every output section name declared in an agent's I/O contract also appears verbatim in the dispatching skill's SKILL.md — and which existing cross-reference scenario (`xref-agent-registry.sh`, `xref-core-registry.sh`) provides the structural template for a new `xref-io-contracts.sh` scenario implementing this check across all 18 agents and their dispatching skills?
    **Why it matters:** The value of formalizing I/O contracts is zero if dispatching skills reference output section names that differ from what agents declare. A cross-reference test closes the contract loop and makes naming divergence detectable at commit time without running the LLM. This determines whether a new cross-reference scenario is feasible with grep (or requires a parser), and which skill→agent pairings must be covered. Load-bearing for the test strategy and for the contract format choice (section names must be grep-extractable).
    **Source hint:** `tests/scenarios/xref-agent-registry.sh` and `tests/scenarios/xref-core-registry.sh` for cross-reference pattern; `skills/fix-bugs/SKILL.md` for current output section references (e.g., `## Fix Report`, `## Triage Analysis`); grep `agents/*.md` for `## [A-Z]` headings inside Process sections to enumerate current output section names.
    **Cluster:** agent-3 Q8
