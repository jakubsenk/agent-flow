### Q6: Does the acceptance gate condition differ correctly between bug and feature pipelines?
**Area:** 2

## Area 4: State.json Write Contract

### Q10: Does fix-ticket write `decomposition.decision`/`decomposition.strategy`/`decomposition.subtasks` as fix-bugs and implement-feature do?
**Area:** 4
**Files:** `commands/fix-ticket.md`, `commands/fix-bugs.md`, `commands/implement-feature.md`, `state/schema.md`
**Question:** `implement-feature.md` (step 5) explicitly writes `decomposition.decision`, `decomposition.strategy`, and `decomposition.subtasks` to state.json. `scaffold.md` (step 5) writes `decomposition.subtasks` and `decomposition.strategy`. However, `fix-ticket.md` has decomposition steps (4b, 4c) and `fix-bugs.md` has decomposition steps (3b, 3c), but neither contains any `decomposition.*` state write instruction. Is this an intentional gap (decomposition state only written by feature/scaffold pipelines) or a missing state write in the bug pipelines?

### Q11: Does the `mode` field initialization differ between pipelines and does it match the schema enum?
**Area:** 4
**Files:** `commands/fix-ticket.md` (line 78), `commands/fix-bugs.md` (line 82), `commands/implement-feature.md` (line 78), `commands/scaffold.md` (line 211), `state/schema.md`
**Question:** The schema defines `mode` as one of: `code-bugfix`, `code-feature`, `code-project`, `analysis`, `strategy`, `content`. fix-ticket initializes `mode: "code-bugfix"`, fix-bugs uses `mode: "code-bugfix"`, implement-feature uses `mode: "code-feature"`, scaffold uses `mode: "code-project"`. Do all four pipeline commands initialize the correct mode enum value from the schema, and is there a test that validates the mode value (not just the presence of `state.json` references)?

### Q12: Does implement-feature reuse `triage.*` and `code_analysis.*` state fields for spec-analyst and architect, and is this semantic reuse documented and tested?
**Area:** 4
**Files:** `commands/implement-feature.md` (lines 176, 186), `state/schema.md`, `tests/scenarios/state-schema.sh`
**Question:** `implement-feature.md` explicitly reuses `triage.status` and `triage.acceptance_criteria` for the spec-analyst phase output, and reuses `code_analysis.status` for the architect phase output. The schema.md does not document this cross-pipeline field reuse. `state-schema.sh` only checks minimum reference counts, not semantic correctness. Should the schema document the `mode`-specific field reuse semantics, and is there a risk of resume-ticket.md misinterpreting these fields when resuming an implement-feature run?

## Area 5: Core Contract Adherence

### Q13: Is `core/mcp-detection.md` tested or referenced by any pipeline command other than scaffold.md and init.md?
**Area:** 5
**Files:** `core/mcp-detection.md`, `commands/scaffold.md`, `commands/init.md`, `core/mcp-preflight.md`, `tests/scenarios/core-include-refs.sh`
**Question:** `core-include-refs.sh` validates 10 core files but its hardcoded list does NOT include `mcp-detection.md`. Zero test scenarios grep for `mcp-detection`. The contract is used by `core/mcp-preflight.md` (which delegates to it) and directly by `scaffold.md` (Step 0-MCP) and `init.md`. The bug fix pipelines all use `core/mcp-preflight.md` which internally uses `mcp-detection.md`. Is `core/mcp-detection.md` currently excluded from the `core-include-refs.sh` test, and should it be added?

### Q14: Does `core/block-handler.md` define rollback scope (which agents trigger rollback vs. which don't) consistently with how each pipeline command invokes the Block handler?
**Area:** 5
**Files:** `core/block-handler.md`, `commands/fix-ticket.md` (step X), `commands/fix-bugs.md` (step X), `commands/implement-feature.md` (step X)
**Question:** `core/block-handler.md` Process step 1 specifies: rollback if blocking agent is `fixer`, `reviewer`, or `test-engineer`; do NOT rollback for `triage-analyst` or `code-analyst`. `fix-bugs.md` step X explicitly re-documents the rollback condition "DO NOT rollback on block from triage/code-analyst". But `fix-ticket.md` step X says "Follow `core/block-handler.md`" without re-documenting the condition. `implement-feature.md` step X says "Run rollback-agent" without specifying the no-rollback condition. Is the no-rollback-on-analysis-agent condition consistently enforced across all three pipeline commands, or does the delegation-only approach in fix-ticket and implement-feature risk unintended rollback?

### Q15: Does `core/profile-parser.md` validate against the `not_applicable` status enum value and does the test cover profile stage name validation (warning on unknown names)?
**Area:** 5
**Files:** `core/profile-parser.md`, `tests/scenarios/profile-skip.sh`, `state/schema.md`
**Question:** `core/profile-parser.md` Process step 6 specifies that invalid stage names produce a warning: `[WARN] Unknown stage '{name}' in profile — ignored`. `profile-skip.sh` only checks that `--profile` text and `NEVER.*skip` text exist in the commands — it does not verify that the profile-parser contract's warning behavior or invalid-name handling is mentioned in the command files. Is the profile-parser's invalid-stage-name warning behavior referenced in any pipeline command, and is the `skipped` vs `not_applicable` distinction from the Step Status Enum used consistently when a profile skip is applied?
