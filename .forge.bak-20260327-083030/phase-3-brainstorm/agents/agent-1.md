# Persona A: The Pragmatic Incrementalist

I've shipped 50+ developer tools. The pattern is always the same: teams want "scaffold to production" but what actually saves time is the boring middle — the part where you catch a broken deploy at 2am because there was no health check. Let me look at what's actually in front of us.

## Grounding observations

1. **Scaffold already ends with "fill in TODOs in CLAUDE.md."** That's the real gap — the user has a running project locally but no verified path to a running deployment. The scaffold pipeline drops them at git-init with a list of homework.
2. **The Verify command already exists** in `core/fix-verification.md` — it runs post-merge. Repurposing it as a deploy trigger is free infrastructure.
3. **The reproducer agent** (`agents/reproducer.md`) already has the `run_in_background` + health-poll pattern. A deployment verifier can copy that pattern verbatim.
4. **Command chaining is impossible.** We cannot have `/scaffold` call `/check-deploy`. Each idea must be a standalone command or an inline addition to an existing command.
5. **No required config keys without a MAJOR bump.** Everything must be optional.

---

## Idea 1: `/check-deploy` — Standalone Deployment Smoke Test

**Description:** A new command that reads an optional `### Local Deployment` config section from CLAUDE.md, starts the app (Docker or native), polls a health endpoint, runs the existing Verify command if configured, and reports `[OK]/[FAIL]/[SKIP]` per check. It follows the exact same `[OK]/[FAIL]/[SKIP]` output format as `scaffold-validate` and `check-setup`. It does not deploy anywhere — it answers the question "does this thing actually start and respond?" on the developer's machine. This is the single highest-ROI addition because it catches the #1 scaffold failure mode: the generated skeleton builds and tests pass, but the app doesn't actually boot.

**Pros:**
- Standalone — zero coupling to scaffold or any other command. Users of existing projects benefit immediately.
- Follows the proven `scaffold-validate` output pattern — no new UX to learn.
- Optional config section = MINOR version bump. Precedent: Browser Verification section.

**Cons:**
- Requires Docker daemon for Docker-based checks (same limitation as scaffold-validate step 6).
- Another command to discover — though the skill router and `/check-setup` closing message can point to it.
- Health endpoint varies wildly per framework — the config must be user-provided, not auto-detected.

**Effort:** S (1 command file, 1 optional config section, 0 new agents)

**Files affected:**
- `commands/check-deploy.md` (new) — the command definition
- `docs/reference/automation-config.md` — document the new optional `Local Deployment` section
- `CLAUDE.md` — add command to the command list, add config section to optional table
- `skills/router.md` — add routing entry for deploy-related intents

**What the user types:**
```
/ceos-agents:check-deploy
```
Or after scaffold:
```
/ceos-agents:check-deploy --docker
```

**Config section (in consuming project's CLAUDE.md):**
```markdown
### Local Deployment

| Key | Value |
|-----|-------|
| Start command | `docker compose up -d` |
| Health URL | `http://localhost:3000/health` |
| Health timeout | `30` |
| Teardown command | `docker compose down` |
```

## Idea 2: Scaffold Step 10.5 — Inline Deploy Verification

**Description:** Add an optional step between Step 10 (Final Report) and the end of `/scaffold` that runs the app locally and verifies it responds. No new command, no new agent — just 15 lines of inline logic in `commands/scaffold.md` that: (1) reads the generated CLAUDE.md's Build & Test section, (2) starts the app via `run_in_background`, (3) polls a health endpoint for up to 30s, (4) kills the process, (5) adds a `Local run: [OK]/[FAIL]/[SKIP]` line to the final report. This catches "it builds but doesn't boot" at scaffold time, when the user is still paying attention.

**Pros:**
- Zero new files — just an addition to the existing scaffold command.
- Catches the most common scaffold failure mode at the earliest possible moment.
- No config needed — the scaffolder already generates a Start command (or the user has one in Build & Test).

**Cons:**
- Only helps scaffold users, not users of existing projects.
- Health endpoint must be guessable or generated by scaffolder (e.g., `/health` or `/`). If wrong, it's a false negative.
- Adds ~30s to scaffold time even when everything works.

**Effort:** XS (edit 1 existing file, ~15 lines of command logic)

**Files affected:**
- `commands/scaffold.md` — add Step 10.5 between Step 10 (Final Report) generation and the end
- `agents/scaffolder.md` — add a convention: scaffolder MUST generate a `/health` endpoint in the entry point (or equivalent root route)

**What the user types:**
Nothing new — it runs automatically as part of:
```
/ceos-agents:scaffold "a REST API for managing bookmarks"
```

The final report gains one line:
```
Local run: [OK] — http://localhost:3000/health responded 200 in 4.2s
```

## Idea 3: `deployment-verifier` Agent + Post-Scaffold Deploy Step

**Description:** A new agent (`agents/deployment-verifier.md`, model: sonnet) that follows the reproducer pattern: starts the app via `run_in_background`, polls health, optionally runs `curl` against key routes from the spec, collects evidence (response codes, timing, console output), and produces a structured verdict. This agent is called from scaffold Step 10.5 AND from `/check-deploy` (Idea 1), making it the reusable "does this thing run?" primitive. State is tracked in `state.json` via a new `deployment` phase object.

**Pros:**
- Reusable across scaffold and standalone deploy checks — single source of truth for "does it boot?"
- The reproducer already proved this pattern works (background start, poll, timeout, cleanup).
- Agent can be overridden via Agent Overrides for project-specific deploy logic.

**Cons:**
- More moving parts than Idea 2 (new agent + new state schema field + new command).
- The agent model (sonnet) costs tokens for what is fundamentally a bash script with a health check.
- Overkill if the only check is "GET /health returns 200" — an agent is not needed for that.

**Effort:** M (1 new agent, 1 new command, state schema update, config section)

**Files affected:**
- `agents/deployment-verifier.md` (new) — agent definition
- `commands/check-deploy.md` (new) — command that dispatches the agent
- `commands/scaffold.md` — add optional Step 10.5 that dispatches deployment-verifier
- `state/schema.md` — add `deployment` phase object
- `CLAUDE.md` — update agent count (18 -> 19), add command, add config section

**What the user types:**
```
/ceos-agents:check-deploy
```
Or it runs automatically at the end of scaffold.

## Idea 4: `/onboard --update` Post-Scaffold Auto-Fill

**Description:** After scaffold completes, the user has a CLAUDE.md with TODO placeholders for Instance and Remote. Instead of telling them to edit manually, scaffold's final report says: "Run `/ceos-agents:onboard --update` to fill in the remaining TODOs." The `/onboard` command already supports `--update` mode (Step U0-U3 in `commands/onboard.md`). This requires zero new code — just changing scaffold's Step 10 closing message to explicitly recommend `/onboard --update` instead of "Review CLAUDE.md — fill in TODO sections." The onboard wizard then walks them through only the missing values (Instance URL, Remote, etc.) interactively.

**Pros:**
- Zero new files, zero new logic. Pure documentation change.
- Solves the real UX problem: users don't know how to fill in the TODOs correctly.
- The onboard update mode already skips sections that are filled in — it will focus exactly on the gaps.

**Cons:**
- Does not add deployment verification — only improves the config-completion step.
- Still requires the user to know their Instance URL, Remote, etc. — onboard can't guess those.
- Marginal improvement — experienced users already know to run onboard.

**Effort:** XS (edit 2 lines in scaffold's Step 10)

**Files affected:**
- `commands/scaffold.md` — change Step 10 "Next steps" item 1 from "Review CLAUDE.md" to "Run `/ceos-agents:onboard --update`"

**What the user types:**
```
/ceos-agents:onboard --update
```
(After scaffold completes and tells them to.)

## Idea 5: Post-Scaffold Issue Tracker Seeding via Existing Step 9

**Description:** Scaffold already has Step 9 (Issue Tracker — Optional) that creates tracker cards for implemented features. Extend it to also create a "Deploy & verify" card as the last item, with acceptance criteria like "App starts locally via docker compose up", "Health endpoint responds 200", "CI pipeline passes on main." This gives the user a concrete deployment checklist in their tracker that they can then fix with `/ceos-agents:fix-ticket`. No new commands, no new agents — just enriching the existing Step 9 output.

**Pros:**
- Leverages the existing Step 9 infrastructure — no new plumbing.
- Creates a traceable deployment task that feeds back into the fix-ticket pipeline.
- Zero coupling to Docker, health checks, or any runtime — it's just a card with instructions.

**Cons:**
- Only works if the user configures an issue tracker (most skip this at scaffold time).
- The card is just text — it doesn't actually verify anything.
- Adds noise if the user's project doesn't need deployment (e.g., a CLI tool or library).

**Effort:** XS (add ~10 lines to scaffold Step 9)

**Files affected:**
- `commands/scaffold.md` — extend Step 9 to include a "Deploy & verify" card template

**What the user types:**
Nothing new — it happens automatically when they approve Step 9's "Create cards?" prompt:
```
/ceos-agents:scaffold "a REST API for managing bookmarks"
# ... scaffold runs ...
# Step 9: "Create cards in issue tracker for implemented features? [Y/n]"
# User: Y
# -> Creates feature cards + 1 "Deploy & verify" card
```

## Top Recommendation

**Ship Idea 1 (`/check-deploy`) first.** Here's why:

1. **Highest ROI per effort.** It's a standalone S-effort command that helps every project — not just freshly scaffolded ones. Any user with an existing project can run it tomorrow.

2. **It's the boring, obvious thing.** "Does my app start?" is the most common failure mode after "does it build?" We already have `scaffold-validate` for build/test/lint and `check-setup` for config/MCP. The missing piece is runtime verification. This completes the trilogy.

3. **It unblocks everything else.** Once `/check-deploy` exists, Idea 2 (scaffold inline) becomes a one-liner that says "run the same logic as check-deploy." Idea 3 (agent) becomes an optional upgrade if the check needs to get smarter. Idea 4 (onboard recommendation) is a 2-line change that should ship alongside regardless.

4. **No version bump drama.** Optional config section = MINOR bump. No new agents needed for v1 — it's just a command that runs bash commands and reports results. If we later want a smarter agent-based version (Idea 3), that's a separate MINOR.

**Ship Idea 4 alongside for free** — it's 2 lines and improves scaffold UX immediately.

**Skip Idea 3 for now.** An agent is overkill for "start app, GET /health, report result." If check-deploy proves insufficient (e.g., people need route-level verification), promote to an agent later.

**Skip Idea 5 unless users ask.** Most scaffold users skip Step 9 because they don't have a tracker configured yet. The card is theoretically nice but practically invisible.
