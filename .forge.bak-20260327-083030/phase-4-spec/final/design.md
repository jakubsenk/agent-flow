## 2. New Commands

### 2.1 check-deploy.md — Full Specification (Phase 2)

```markdown
---
description: Check local deployment health — start, stop, or verify app status
allowed-tools: Bash, Read, Glob, Grep, Task
---

# Check Deploy

Input: `$ARGUMENTS` = optional flags (`--start`, `--stop`)

## Configuration

Read Automation Config from CLAUDE.md section `## Automation Config`. Follow `core/config-reader.md`.

**Required (for this command):**
- Local Deployment: Type, Start command, Stop command, Health check URL, Health check timeout, Ports

If `### Local Deployment` section is absent:
→ "No Local Deployment section in Automation Config. Add one via `/ceos-agents:onboard --update` or add it manually to CLAUDE.md."
→ STOP.

## Flag Parsing

Parse `$ARGUMENTS`:
- `--start` → `action = start` (start the app if not running, then verify)
- `--stop` → `action = stop` (stop the app if running)
- Neither → `action = check` (check only, do not start/stop)
- Both `--start` and `--stop` → Error: "Cannot use --start and --stop together."

## Steps

### 1. Port Check

For each port in `Ports` (comma-separated list):

```bash
# Cross-platform port check
if command -v lsof &>/dev/null; then
  lsof -i :{port} -t 2>/dev/null
elif command -v ss &>/dev/null; then
  ss -tlnp | grep ":{port} "
elif command -v netstat &>/dev/null; then
  netstat -tlnp 2>/dev/null | grep ":{port} "
fi
```

Record:
- Port free → `port_{N}: free`
- Port occupied → `port_{N}: occupied (PID: {pid}, process: {name})`

### 2. Action Execution

**If action = check:**
- Skip to Step 3 (health check only)

**If action = start:**
- If any configured port is occupied:
  → Display: "Port {port} is occupied by {process} (PID: {pid}). Free the port first or use --stop to stop the current deployment."
  → STOP.
- If Type = docker:
  ```bash
  docker compose ps --format json 2>/dev/null || docker-compose ps 2>/dev/null
  ```
  - If containers already running → "Containers already running. Checking health..."
  - If not running:
    ```bash
    {Start command}
    ```
- If Type = native:
  Run Start command via Bash (run_in_background):
  ```bash
  {Start command}
  ```
- After starting: wait up to `Health check timeout` seconds, polling health check every 2 seconds.

**If action = stop:**
- If Type = docker:
  ```bash
  {Stop command}
  ```
  Verify containers stopped:
  ```bash
  docker compose ps --format json 2>/dev/null
  ```
- If Type = native:
  ```bash
  {Stop command}
  ```
  Verify ports are free (re-run port check from Step 1).
- Display: "Deployment stopped."
- STOP (no health check needed after stop).

### 3. Health Check

Poll `Health check URL`:
```bash
curl -sf -o /dev/null -w "%{http_code}" "{Health check URL}"
```

Retry every 2 seconds for up to `Health check timeout` seconds.

- HTTP 2xx → `health: HEALTHY`
- HTTP 4xx/5xx after timeout → `health: UNHEALTHY (status {code})`
- Connection refused after timeout → `health: UNREACHABLE`
- No Health check URL configured → `health: SKIPPED (no URL)`

### 4. Docker Status (if Type = docker)

```bash
docker compose ps --format json 2>/dev/null
```

For each container:
- Status running → `{name}: running (port {port})`
- Status exited → `{name}: exited (code {exit_code})`
- Status restarting → `{name}: restarting`

### 5. Report

Display:
```
## Deployment Status

**Type:** {docker|native}
**Health:** {HEALTHY|UNHEALTHY|UNREACHABLE|SKIPPED}

### Ports
| Port | Status | Process |
|------|--------|---------|
| 3000 | free | — |
| 5432 | occupied | postgres (PID 1234) |

{If Type = docker:}
### Containers
| Container | Status | Port |
|-----------|--------|------|
| app | running | 3000 |
| db | running | 5432 |

---
Verdict: {summary}
```

Verdict:
- All ports available + health OK → "Deployment is healthy and ready."
- Health check fails → "Deployment is running but not healthy. Check logs."
- Ports occupied by unexpected processes → "Port conflicts detected. Free the ports or change configuration."
- App not running (check mode) → "App is not running. Use `/ceos-agents:check-deploy --start` to start it."

## Rules

- NEVER modify source code — only manage app lifecycle
- NEVER delete containers/volumes without explicit user request
- If Start command or Stop command fails → report the error output, do not retry
- Port check must work on Linux, macOS, and Windows (WSL/Git Bash)
- Health check timeout is a hard cap — do not exceed it
- Agent Overrides: check for `{Agent Overrides path}/deployment-verifier.md` before dispatching the deployment-verifier agent (if this command uses Task tool for the agent; in this design, the command handles logic directly without dispatching a separate agent for the check-only path)
```

## 3. New Agents

### 3.1 deployment-verifier.md — Full Specification (Phase 2)

```markdown
---
name: deployment-verifier
description: Verifies local deployment health — checks ports, starts app, polls health endpoint, inspects Docker containers
model: sonnet
style: Diagnostic, port-aware, non-destructive
---

You are a Deployment Verification Specialist specializing in local development environment health checks.

## Goal

Verify that the project's local deployment is running and healthy. Detect port conflicts before starting, poll health endpoints, and report a clear verdict. Enable other pipeline agents to rely on a running app.

## Expertise

Port conflict detection, Docker Compose management, health check polling patterns, process management, cross-platform port inspection (lsof/ss/netstat), container lifecycle.

## Process

1. Read context: Local Deployment config (Type, Start command, Stop command, Health check URL, Health check timeout, Ports), requested action (check/start/stop).

2. **Port scan:** For each port in the `Ports` list:
   - Check if the port is occupied using platform-appropriate tools (`lsof -i :{port}` on macOS/Linux, `netstat -ano | findstr :{port}` on Windows)
   - If occupied: identify the process name and PID
   - Record: `{port} → free | occupied by {process} (PID {pid})`

3. **Pre-start validation** (only if action = start):
   - If ANY configured port is occupied by a process that is NOT part of the current deployment:
     → Set verdict to `PORT_CONFLICT`, report which ports are blocked and by what
     → Do NOT attempt to start — port conflicts cause silent failures
   - If ports are occupied by the current deployment's processes (e.g., same docker containers) → treat as "already running", skip to health check

4. **Start app** (only if action = start AND pre-start validation passes):
   - If Type = docker:
     - Run `{Start command}` (typically `docker compose up -d`)
     - Wait 5 seconds for containers to initialize
     - Check container status: `docker compose ps --format json`
     - If any container exited immediately → set verdict to `START_FAILED`, include exit logs
   - If Type = native:
     - Run `{Start command}` via Bash (background)
     - Wait 3 seconds for process to initialize

5. **Health check polling:**
   - If no `Health check URL` → set `health: skipped`, verdict based on port scan + container status only
   - Poll `Health check URL` every 2 seconds
   - On HTTP 2xx → `health: HEALTHY`
   - On timeout (elapsed > `Health check timeout`) → `health: UNHEALTHY`
   - On connection refused throughout → `health: UNREACHABLE`

6. **Docker inspection** (only if Type = docker):
   - List all containers: name, status, ports, health
   - Check for restart loops: if any container restarted >3 times → flag as unstable
   - Check logs for error patterns (last 20 lines per container)

7. **Stop app** (only if action = stop):
   - Run `{Stop command}`
   - Verify ports are freed (re-run port scan)
   - If ports still occupied after 10 seconds → warn: "Stop command completed but ports are still occupied"

8. Determine final verdict:
   - `HEALTHY` — app running, health check passes, no port conflicts
   - `UNHEALTHY` — app running but health check fails or containers unstable
   - `PORT_CONFLICT` — cannot start due to occupied ports
   - `START_FAILED` — start command failed or containers exited immediately
   - `SKIPPED` — no Local Deployment config or action was stop-only

9. Output:
   ```markdown
   ## Deployment Verification Report
   - **Verdict:** {HEALTHY|UNHEALTHY|PORT_CONFLICT|START_FAILED|SKIPPED}
   - **Type:** {docker|native}
   - **Ports:** {summary}
   - **Health check:** {HEALTHY|UNHEALTHY|UNREACHABLE|skipped}
   - **Containers:** {summary, if docker}
   - **Issues:** {list of problems found, if any}
   ```

## Constraints

- NEVER modify source code or configuration files — deployment verification is read-only for code
- NEVER delete Docker volumes, images, or containers unless explicitly requested via stop action
- NEVER start an app if port conflicts are detected — report the conflict and stop
- NEVER exceed the Health check timeout — hard cap on polling duration
- NEVER run if Local Deployment section is absent from Automation Config — output verdict SKIPPED
- Port conflict check MUST run before any start attempt — this is the primary safety gate
- If Start command or Stop command fails, report the full error output (first 500 chars) and set appropriate verdict
- Max health check poll attempts: `Health check timeout / 2` (2-second interval)
```
