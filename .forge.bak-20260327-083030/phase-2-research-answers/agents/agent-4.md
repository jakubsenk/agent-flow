# Research Answers — Deployment Verification Agents & Config (RQ-13 to RQ-16)

## RQ-13: browser-verifier extension vs new deployment-verifier agent

**Question:** Should browser-verifier be extended for deploy smoke, or is a new agent cleaner?

### Findings

`agents/browser-verifier.md` is deeply coupled to the bug-fix pipeline context:

- **Step 1** reads `reproduction-result.json` for issue ID, fixer diff, and acceptance criteria extracted from a bug report.
- **Step 3a** replays a `reproducer-script.js` generated by the reproducer agent specifically for an issue ID.
- **Step 3b** reads the fixer diff to identify up to 3 directly modified routes — entirely meaningless in a deploy context.
- **Step 3c** performs a visual AC check based on bug-fix acceptance criteria.
- **Step 5** writes results to `.ceos-agents/{ISSUE-ID}/verification-result.json` — namespaced to a bug issue ID.
- The `FAILED` verdict returns control to the fixer agent — the pipeline coupling is structural, not incidental.
- The `On events` config key controls invocation with values `reproduce` and `verify` — scoped to the fix pipeline events only.

A deploy smoke check has a fundamentally different goal: verify that a freshly deployed environment is healthy — no issue ID, no fixer diff, no reproducer script. The relevant inputs would be a deploy target URL, health endpoints, and expected page states.

**Answer:** A new `deployment-verifier` agent is the correct choice. Extending browser-verifier would require gutting its core logic (diff reading, issue ID namespacing, reproducer script replay, fixer feedback loop) and replacing it with deploy-specific behavior. The coupling to bug-fix context is structural, not superficial. A new agent keeps both agents focused and independently testable.

**Confidence:** HIGH

## RQ-14: Long-running docker orchestration via Bash tool — viability

**Question:** Can the browser-verifier pattern (generate script → Bash execute → read results) be applied to docker compose up?

### Findings

The pattern used by `agents/reproducer.md` (steps 2, 4, 5):

1. **Prerequisite check:** Agent runs `npx playwright --version` and `GET {Base URL}` via Bash to verify environment before proceeding.
2. **Script generation:** Agent generates a `.js` file and saves it to `.ceos-agents/{ISSUE-ID}/reproducer-script.js`.
3. **Bash execution with timeout:** `timeout {Timeout}s node .ceos-agents/{ISSUE-ID}/reproducer-script.js` — the key pattern.
4. **Result reading:** Agent reads the JSON output file written by the script.

The `reproducer.md` step 2 already handles a long-running process: if the app is not running and `Start command` is set, it starts it via `run_in_background`, waits up to 15 seconds, and retries a health check. This is exactly the pattern needed for `docker compose up`.

No Bash timeout constraints are mentioned in `commands/check-setup.md` or `agents/browser-verifier.md` beyond the per-operation `Timeout` config key (default 60s). The `run_in_background` parameter in reproducer step 2 is used for the dev server — docker compose up would work identically.

**Pattern applied to docker compose up:**

```
1. Check prerequisites: docker, docker compose version via Bash
2. Run: docker compose up -d (run_in_background)
3. Health check loop: poll /health or container status up to Timeout
4. Write result JSON: {status: ready|timeout|error, containers: [...], logs: [...]}
5. Agent reads result JSON and proceeds
```

The `docker compose down` cleanup step mirrors the reproducer's `pkill -f "{Start command pattern}"` cleanup in step 5.

**Viability assessment:** The pattern is directly applicable. The key constraint is that docker compose up is a background process — which reproducer already handles with `run_in_background`. The health check polling (retry with wait) is also precedented. No architectural blocker exists.

**Confidence:** HIGH — the reproducer pattern is purpose-built for "start a background process, wait for it to be ready, then proceed."

## RQ-15: New Local Deployment config section vs extending Build & Test

**Question:** Should deployment config be a new section or extend Build & Test?

### Findings

**`Build & Test` section** (from `core/config-reader.md` and `CLAUDE.md`):

```
build.build_command
build.test_command
build.verify_command  (optional key within this section)
```

The `Verify command` is already documented as "runs after PR merge" — it is the closest existing concept to a deploy check. However, deploying a local stack (DB + FE + BE) requires qualitatively different keys: compose file path, service health endpoints, startup timeout, environment variables, teardown strategy. These do not fit alongside `Build command` and `Test command` without semantic confusion.

**`Browser Verification` section** was added as a standalone optional section (not folded into `E2E Test`), establishing the precedent that browser/environment infrastructure gets its own section. The config reader in `core/config-reader.md` adds new optional sections by adding a new `### {Section}` parse block — this is an additive, non-breaking change.

**Versioning policy** (from `CLAUDE.md`): Adding an optional section = MINOR. Extending `Build & Test` with new optional keys is also MINOR, but creates semantic clutter and breaks the single-responsibility principle of that section.

**Precedent:** `Browser Verification` (8 keys) was added as a new section rather than extending `E2E Test` (2 keys), even though both deal with browser/test infrastructure. The same rationale applies here.

**Answer:** A new `Local Deployment` optional section is the correct approach. It follows the `Browser Verification` precedent, keeps `Build & Test` semantically clean (build artifacts and test runs), and allows deployment config to have its own keys without ambiguity. The `Verify command` in `Build & Test` can remain as a post-merge smoke URL check while `Local Deployment` handles full stack orchestration.

**Confidence:** HIGH — the Browser Verification precedent is a direct analogy.

## RQ-16: check-setup extension vs new check-deploy command

**Question:** Should check-setup be extended or a new check-deploy command created?

### Findings

`commands/check-setup.md` structure:

- **Block 1:** Automation Config structural check (required + optional sections, table format, placeholders)
- **Block 2:** MCP server presence and connectivity
- **Block 3:** Live connectivity to issue tracker and source control
- **Block 4:** Build & Test execution (can be skipped with `--skip-build`)
- **Block 5:** Plugin composability check

The command is explicitly "safe for repeated execution" and "read-only". Block 4 runs `Build command` and `Test command` — the only side-effectful operations.

**Extension analysis:**

- check-setup validates static config and MCP connectivity. Adding a deployment check (Block 6: start docker compose, verify services, tear down) would introduce a long-running, side-effectful operation inconsistent with the command's "safe for repeated execution" contract.
- The `--skip-build` flag already signals that users want to skip expensive operations. Adding `--skip-deploy` would compound the flag proliferation.
- check-setup validates infrastructure *used by the pipeline* (tracker, SCM, build tool). A deployment check validates infrastructure *created by the pipeline* — a different responsibility.
- Block 1 already validates optional sections including Browser Verification (step 5, line: "Exists and correct format → [OK]"). A `Local Deployment` section would be validated structurally for free, without needing check-deploy.

**New command analysis:**

- A `check-deploy` command would: start the local stack, run health checks, report service status, tear down. Scoped, single-responsibility.
- It maps cleanly to a standalone user intent: "is my deploy config correct and does my stack start?"
- It can be invoked as a pre-scaffold or pre-feature-implementation gate without coupling to the full pipeline setup validation.

**Answer:** A new `check-deploy` command is the correct choice. check-setup's contract (read-only, safe for repeated execution, infrastructure presence check) is incompatible with starting a docker stack. The check-deploy command would follow the same output format (`[OK]/[FAIL]/[SKIP]` blocks) but scope to: parse `Local Deployment` section, run `docker compose up`, poll health endpoints, report container status, run `docker compose down`. This keeps check-setup fast and non-destructive.

**Confidence:** HIGH — the "safe for repeated execution" and "read-only" constraints in check-setup.md are explicit contract violations if deploy orchestration is added inline.

**NEEDS_VALIDATION:** Confirm whether the intended `check-deploy` should be a standalone command or a subcommand invokable as `check-setup --deploy`. The flag approach is more discoverable but violates the read-only contract.
