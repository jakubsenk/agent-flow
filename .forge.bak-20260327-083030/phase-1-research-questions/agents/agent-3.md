# Research Questions — Local Deployment Orchestration

Q1: The scaffolder generates a `Dockerfile` and `.dockerignore` (Batch 4), and `scaffold-add docker` explicitly generates `Dockerfile + .dockerignore + docker-compose.yml` — but the scaffolder agent definition itself never mentions `docker-compose.yml`. Is the multi-service orchestration file (`docker-compose.yml`) only produced by `scaffold-add docker`, or is it also produced during the full scaffold pipeline? If the latter, where exactly in the scaffolder process should it be added, and what service definitions (DB, FE, BE + health checks) should it include by default for full-stack projects?

Q2: The reproducer agent pattern demonstrates "run a command, wait for output, verify success" via a three-step sequence: (1) start the app in the background with `run_in_background`, (2) wait up to 15s and retry a health-check GET to `{Base URL}`, (3) continue or stop based on the HTTP response. Can a new `/deploy-local` command or a `deploy-local` agent apply this exact same pattern to `docker-compose up --detach`, polling each service's health endpoint (or using `docker inspect --format='{{.State.Health.Status}}'`) until all containers report `healthy`, and then pass a structured readiness result to downstream verification — without requiring any runtime code beyond what Claude Code's Bash tool can invoke?

Q3: The `browser-verifier` agent is scoped entirely around a single bug fix: it replays a `reproducer-script.js`, checks adjacent pages, and produces a `VERIFIED/PARTIAL/FAILED/SKIPPED` verdict. Its prerequisite check (Playwright installed + app reachable via GET to `Base URL`) is already a deployment health check in miniature. Could `browser-verifier` be extended with a third sub-phase — "Deployment Smoke" — that, when triggered by a new `On events: deploy` value in the `Browser Verification` config, navigates to a configurable set of smoke URLs (e.g., `/health`, `/`, `/api/status`) and asserts HTTP 200 + no critical console errors, reusing the existing verdict taxonomy (`VERIFIED/PARTIAL/FAILED/SKIPPED`) — or would the coupling to bug-fix context (fixer diff, issue ID, `reproducer-script.js`) make a separate `deployment-verifier` agent a cleaner design?

Q4: The Automation Config `Build & Test` section currently holds `Build command` and `Test command`, and the optional `Browser Verification` section holds `Base URL` and `Start command`. A local deployment workflow needs at minimum: a `docker-compose up` invocation, per-service health check URLs, and a readiness timeout. Should these live in a new optional `Local Deployment` config section (analogous to `Browser Verification`), or should they be added as new optional keys to `Build & Test` (e.g., `Compose file`, `Health check URLs`, `Startup timeout`) — and what is the MINOR vs MAJOR versioning impact of each choice given the current config contract rules?

Q5: The `check-setup` command validates config structure, MCP connectivity, and build/test commands, but has no concept of runtime service health. If a `/deploy-local` command or a `deploy` pipeline stage is introduced, should `check-setup` be extended to verify deployment prerequisites — Docker daemon reachable (`docker info`), `docker-compose` binary present, image names resolvable — or should a dedicated `/ceos-agents:check-deploy` command own that responsibility, following the same read-only, safe-for-repeated-execution contract that `check-setup` currently enforces?
