# Research Questions — Plugin Constraints & Deployment Verification

Q1: Can a markdown command invoke another markdown command directly (i.e., true command chaining), or is the only supported cross-command delegation pattern the Task tool dispatching a named agent — meaning commands can only compose via agents, never via other commands?

Q2: The `allowed-tools` frontmatter in commands (e.g., `Bash, Read, Write, Edit, Glob, Grep, Task, mcp__*`) defines what Claude Code exposes to that command at runtime. Does this list represent a hard capability ceiling — specifically, is there any mechanism by which a pure-markdown command can trigger network I/O, start a process (e.g., `docker compose up`), or read environment variables beyond what the Bash tool permits within that list?

Q3: State in `.ceos-agents/{RUN-ID}/state.json` is written atomically by Claude during a session and read back in a subsequent `/resume-ticket` invocation — but there is no daemon, no lock file, and no session handshake. What is the exact failure mode when a Claude Code session terminates mid-pipeline (e.g., power loss, token limit hit) and a new session attempts to resume: does `core/state-manager.md` guarantee a consistent resume point, or can the pipeline silently restart a completed step?

Q4: The `browser-verifier` pattern (agent generates a Playwright JS script → Bash executes `node script.js` → agent reads the JSON result file) is the only precedent in the plugin for "agent-driven external process execution." Could this same pattern be used for deployment verification — for example, having an agent generate a `docker compose up` script or a health-check script, execute it via Bash, then read its stdout/exit-code as a structured result — and what are the Bash tool constraints (timeout, no interactive TTY, no persistent shell state between calls) that would make or break this for long-running startup processes?

Q5: Adding a "Deploy" workflow (project creation → scaffold → feature loop → local deployment with DB/FE/BE) would require new Automation Config keys such as `Compose file`, `Health check URL`, `Services`, and `Deploy command`. Per the versioning policy in CLAUDE.md, are these new keys optional (MINOR bump) or required (MAJOR bump), and does the existing `Hooks` section (Pre-fix, Post-fix, Pre-publish, Post-publish) already provide a hook point sufficient to inject a deployment step — or would a new `Deploy` hook category be needed, which would itself be a contract-level change?
