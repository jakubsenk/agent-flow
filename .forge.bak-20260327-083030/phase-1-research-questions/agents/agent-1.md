# Research Questions — Scaffold Gaps & Issue Tracker

## Domain 1: Existing Scaffold Pipeline Gaps

Q1: The scaffold pipeline (Step 9) can optionally create epics and sub-issues in the issue tracker after implementation — but it only does so when the tracker is already configured in the generated CLAUDE.md (no TODO markers). Since `Instance` and `Remote` are always generated with TODO placeholders, Step 9 is effectively unreachable in practice. What is the intended bootstrap sequence: should the user configure the tracker *before* running `/scaffold`, or should `/scaffold` support a `--tracker` flag to pre-fill those values so Step 9 can actually run?

Q2: The scaffolder generates a CLAUDE.md with all required Automation Config sections, but marks `Instance` (issue tracker URL) and `Remote` (source control owner/repo) as TODO. The `implement-feature` command requires both to be non-TODO before it can run (it performs an MCP pre-flight check and reads branch naming from `Remote`). Is there currently any command or guided flow that bridges this gap — taking the user from a freshly scaffolded project with TODO placeholders to a fully configured project ready for `/implement-feature`? Or does this require manual editing with no tooling support?

Q3: Scaffold Step 7 (feature implementation loop) deliberately skips all hooks (`Pre-fix`, `Post-fix`, `Pre-publish`, `Post-publish`) because "the project is being created from scratch." However, the fixer in Step 7a runs the Build command from the generated CLAUDE.md, and the test-engineer in Step 7c runs the Test command. If those commands depend on infrastructure (e.g., a running database, a `.env` file with secrets) that doesn't exist yet during scaffold, what is the expected failure path? Does the scaffolder agent generate a `.env.example`/`.env.test` that is sufficient for the build and test commands to pass in isolation, or is there an unhandled gap where build/test commands silently require external services?

Q4: The scaffold pipeline ends with `git init` and a single commit at Step 4, then individual subtask commits at Step 7d — all on the default branch (no remote branch, no PR). The `implement-feature` pipeline, by contrast, always creates a feature branch and a PR via publisher. After scaffold completes, the working tree is on the default branch with no remote configured. What is the intended handoff: does the user manually push the scaffold result to a remote repo and then configure `Remote` in CLAUDE.md, or is there a missing `scaffold-push` / `scaffold-finalize` step that should set up the remote and push?

## Domain 2: Issue Tracker Project Creation

Q5: The MCP servers configured via `/init` cover issue CRUD (create/read/update/close issues, add labels, set states). The `forgejo-mcp`, `@vitalyostanin/youtrack-mcp`, `@modelcontextprotocol/server-github`, and other configured packages are referenced only by package name in `docs/reference/trackers.md` — there is no capability matrix documenting which MCP tools each server exposes. Do any of the six supported MCP servers expose tools for *project-level operations* (create repository, create project/board, create label set, create milestone/sprint) — or are they all scoped to issue-level operations only? Answering this determines whether project creation can be automated via existing MCP calls or requires a new integration layer.
