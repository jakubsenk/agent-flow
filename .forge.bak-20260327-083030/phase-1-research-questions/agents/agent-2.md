# Research Questions — Feature Loop Orchestration

Q1: `implement-feature` requires an Issue ID as mandatory input (line 8 of implement-feature.md) and calls spec-analyst to read feature details directly from the issue tracker via MCP. For a scaffold-first workflow where features come from `spec/epics/*.md` rather than an issue tracker, what is the minimum viable bridge? Specifically: does Step 9 of scaffold (optional issue tracker card creation) produce issue IDs that implement-feature can consume, and if the tracker is not yet configured (TODO markers present), what alternative input path exists — given that `--spec` is only a flag on `/scaffold`, not on `/implement-feature`?

Q2: Scaffold writes a single `state.json` under `.ceos-agents/scaffold-{timestamp}/` with `mode: "code-project"`. Each `implement-feature` run creates its own `state.json` under `.ceos-agents/{ISSUE-ID}/` with `mode: "code-feature"`. There is no cross-run linking field in the schema (state/schema.md). If a feature loop runs N implement-feature calls after scaffold, how should the relationship between the scaffold run and the N feature runs be tracked — is a new `parent_run_id` field needed in the schema, or is a naming convention (e.g., feature branch names derived from scaffold run_id) sufficient for resume and metrics purposes?

Q3: Scaffold's Step 7 (Feature Implementation Loop) already runs fixer + reviewer + test-engineer inline for every subtask — it does NOT call `/implement-feature` as a sub-command. The `spec/epics/*.md` files are consumed directly by scaffold's architect step (Step 5) and decomposed into subtasks. What would need to change for scaffold to delegate each epic to a separate `implement-feature` invocation instead? Specifically: scaffold's Step 7 has no spec-analyst (epics are pre-formatted into the Feature Specification format inline), no publisher (no PRs per subtask), and no acceptance-gate — which of these pipeline stages would need to be conditionally skipped via `--profile` or a new flag when implement-feature is called in a scaffold-chained context?

Q4: The `implement-feature` decomposition decision (Step 5) runs `core/decomposition-heuristics.md` against code-analyst output — but the feature pipeline does not include a code-analyst step (line 52: "code-analyst = N/A — feature pipeline does not have code-analyst"). Instead, the architect directly determines whether decomposition is needed. For a feature loop scenario where epics from `spec/epics/*.md` are already pre-decomposed by scaffold's architect into subtasks with `depends_on` DAGs, would passing `--decompose` to implement-feature force a second decomposition (architect re-running on already-decomposed work), and is there a way to pass the existing task tree from `.claude/decomposition/{scaffold-run-id}.yaml` to skip re-decomposition?

Q5: The scaffold pipeline has a hard rule that hooks (Pre-fix, Post-fix, Pre-publish) are NOT executed during scaffold because "there is no existing CI, linter config, or external integration to hook into" (scaffold.md line 340). When the feature loop starts after scaffold completes — meaning CLAUDE.md now has TODO markers for Issue Tracker and Source Control that may still be unfilled — what is the earliest point at which implement-feature's MCP pre-flight check (Step 0, mcp-preflight.md) would pass, and should the feature loop design include a setup-validation gate (e.g., calling `/ceos-agents:check-setup`) before the first implement-feature invocation to detect incomplete CLAUDE.md configuration before any agent work begins?
