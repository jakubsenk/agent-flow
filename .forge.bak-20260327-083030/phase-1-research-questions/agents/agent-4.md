# Research Questions — Cross-Plugin Bridge (Forge Integration)

## Domain: Cross-Plugin Skill Calls, Interface Contract, and Epic-to-Card Decomposition

Q1: Does Claude Code support calling a skill from one installed plugin (`filip-superpowers:forge-plan`) inside a command or skill from a different installed plugin (`ceos-agents:implement-feature`)? The bug-workflow skill already calls ceos-agents commands via `Skill(skill='ceos-agents:<command>', args='...')` within the same plugin — is this `Skill()` invocation mechanism available across plugin namespaces, or is it scoped to intra-plugin calls only? Answer should cite Claude Code plugin system documentation or empirical evidence from the forge SKILL.md (path: `C:/Users/FSABACKY/.claude/plugins/cache/filip-superpowers-marketplace/filip-superpowers/0.1.0/skills/forge/SKILL.md`) where it references sub-skills like `/filip-superpowers:forge-resume`.

Q2: What is the exact output format produced by `filip-superpowers:forge-plan` at the end of its pipeline — specifically the task tree schema it writes to `.forge/`? The `forge-plan` skill (path: `C:/Users/FSABACKY/.claude/plugins/cache/filip-superpowers-marketplace/filip-superpowers/0.1.0/skills/forge-plan/SKILL.md`) must produce some structured decomposition artifact. Does its task tree schema (field names, nesting, dependency representation, acceptance criteria references) match or conflict with the format that ceos-agents `architect` agent produces (YAML with `id`, `title`, `scope`, `files`, `estimated_lines`, `depends_on`, `acceptance_criteria`, `maps_to`)? A field-by-field comparison determines whether a format translation layer is needed or whether ceos-agents can consume forge-plan output directly.

Q3: The `implement-feature` command (step 3) runs `spec-analyst` to extract acceptance criteria from an issue tracker card, then passes those AC to `architect` (step 4) for decomposition. If `filip-superpowers:forge-plan` is used instead as the decomposition source, it would run *before* the issue tracker card exists — it operates on an epic description, not a tracker issue ID. What is the intended creation sequence: (a) user writes epic → forge-plan decomposes → ceos-agents creates cards in tracker → ceos-agents runs implement-feature on each card, or (b) user creates epic card in tracker → `implement-feature --expert` calls forge-plan with the issue description → forge-plan returns a task tree → implement-feature uses it instead of architect output? Which sequence avoids the MCP pre-flight check becoming a blocking dependency at the wrong pipeline stage?

Q4: The `architect` agent in ceos-agents produces a task tree with `maps_to` fields linking each subtask back to a parent acceptance criterion (format: `AC-{N}: {text}`). The `implement-feature` command performs an AC coverage check that fails if any parent AC index is unmapped. If forge-plan produces its own decomposition tree, does it include equivalent traceability fields — or would the AC coverage check in `implement-feature` step 5 always fail because forge-plan subtasks lack `maps_to` entries in the `AC-{N}:` format? This determines whether the AC coverage check must be skipped, adapted, or whether forge-plan's output must be post-processed to inject `maps_to` annotations before being handed to the fixer loop.

Q5: The scaffold pipeline (Step 9) can already create epic cards and sub-issues in the issue tracker after implementation, but this is a post-hoc card-creation step for a project that was already fully built. The cross-plugin bridge inverts this: forge-plan would create cards *before* implementation so that ceos-agents can run `implement-feature` on each one sequentially. Are the MCP tool capabilities of the six supported issue trackers (youtrack, github, jira, linear, gitea, redmine) sufficient to support the required pre-implementation card creation operations — specifically: create epic/parent issue, create sub-issue with description and acceptance criteria, link sub-issue to parent, set initial state — or do any of the supported MCP servers lack `create-issue` or `link-issue` capabilities that would make automated card creation impossible without manual tracker setup?
