# Devil's Advocate Review Report

## Score: 0.75/1.0

## Failure Scenario 1: Batch 7 Conditional Logic Excludes Valid Playwright Use Cases on Non-JS Web Stacks

- **Trigger:** A user scaffolds a Django+templates or Rails+views project. Batch 6 correctly fires (web project detected). Batch 7 then checks for `@playwright/test` in `package.json` `devDependencies` or `dependencies`. But Django/Rails projects have no `package.json` at all -- their dependencies live in `requirements.txt` / `Gemfile`. Playwright can be used with these stacks (via `playwright` Python package or via a separate `package.json` created solely for E2E), but Batch 7 has no detection path for non-JS Playwright installations. Conversely, if a user does add a `package.json` with `@playwright/test` for E2E testing alongside Django, Batch 7 fires but generates `playwright.config.ts` (TypeScript) and `e2e/smoke.spec.ts` -- TypeScript files in a Python project with no `tsconfig.json` or `ts-node` setup.
- **Impact:** (a) Non-JS web projects silently skip E2E test scaffolding even when the scaffold v2 spec explicitly calls for Playwright-based E2E. The downstream Step 8 in `scaffold/SKILL.md` then finds no `E2E Test` section in CLAUDE.md and skips e2e-test-engineer entirely, leaving a gap. (b) If detection is forced via manual `package.json`, the generated `.ts` files won't run without additional TypeScript tooling that Batch 7 does not set up for non-JS stacks. The scaffolder's own verification step 4 ("run test command") does not run `test:e2e` -- it only runs the primary test command -- so this breakage would not be caught during scaffold.
- **Likelihood:** Medium -- Django/Rails+Playwright is a real pattern (Playwright's official docs recommend it for Python via `pytest-playwright`), and scaffold v2 mode unconditionally requires E2E Test config generation (line 94-95 of scaffolder.md: "when running in scaffold v2 mode, MUST generate with framework auto-detected from tech stack"). This creates a contradiction: the scaffolder MUST generate E2E config but has no path to detect or scaffold Playwright for non-JS stacks.
- **Mitigation:** Extend Batch 7 detection to also check `requirements.txt` / `Pipfile` for `pytest-playwright` (Python) and `Gemfile` for Playwright-related gems (Ruby). When detected on non-JS stacks, generate `pytest`-based E2E config (`conftest.py` with Playwright fixtures, `e2e/test_smoke.py`) instead of `.ts` files. The `MUST generate E2E Test config` requirement in scaffold v2 mode should clarify what happens when Playwright is not present -- either auto-add it as a dependency or explicitly document the skip-and-warn behavior.

## Failure Scenario 2: docs/ARCHITECTURE.md Becomes Stale After Feature Implementation, Misleading Downstream Agents

- **Trigger:** Batch 8 generates `docs/ARCHITECTURE.md` during scaffold (Step 3). The scaffolder then auto-populates `Module Docs` with `Path: docs/` in the generated CLAUDE.md (line 98). The scaffold pipeline continues through Steps 5-7 (architect, fixer, reviewer, test-engineer) which implement features from the spec. These feature implementations add new directories, new dependencies, new patterns (e.g., a WebSocket service layer, a caching module). However, no step in the scaffold pipeline updates `docs/ARCHITECTURE.md` after feature implementation. The file retains its skeleton-era content: the original directory tree (missing new dirs), the original patterns (missing new patterns), and the original dependencies (missing newly added packages).
- **Impact:** After scaffold completes, the project enters its operational lifecycle. When `/fix-bugs` or `/fix-ticket` runs, code-analyst (step 2) reads Module Docs at `docs/` and finds `ARCHITECTURE.md`. The agent trusts this as authoritative architecture documentation but it describes a skeleton that no longer matches reality. Code-analyst may narrow its root cause search to the wrong module, or miss that a newly introduced service layer is the actual source of the bug. Architect (step 2 in `/implement-feature`) similarly reads stale architecture docs when planning new features, potentially designing solutions that conflict with patterns added during scaffold implementation. The CLAUDE.md contract says Module Docs is consumed by code-analyst and architect -- both read it before analysis, giving stale content high influence on their reasoning.
- **Likelihood:** High -- every scaffold v2 run generates docs, then implements features that change the project structure. The staleness is guaranteed by design since no update step exists. The severity depends on how much the feature implementation diverges from the skeleton, but even moderate divergence (2-3 new directories, 1 new pattern) is enough to mislead analysis.
- **Mitigation:** Add a post-implementation documentation refresh step in `scaffold/SKILL.md` after Step 7 (test-engineer) or Step 9 (spec compliance check). This step would re-invoke the scaffolder (or a dedicated doc-refresh task) with the instruction: "Update docs/ARCHITECTURE.md to reflect the current project state -- new directories, dependencies, and patterns added during feature implementation." Alternatively, add a constraint to `docs/ARCHITECTURE.md` header: "Auto-generated during scaffold. May be outdated after feature implementation. Regenerate with [command]." -- but this only mitigates if downstream agents are instructed to treat it as advisory rather than authoritative.

## Failure Scenario 3: Test Coverage Is Structurally Shallow -- Cannot Detect Semantic Regressions or File Count Drift

- **Trigger:** The test file `tests/scenarios/scaffolder-e2e-batch.sh` uses `grep -q` to verify presence of specific strings in `scaffolder.md`. All 15 assertions are pure string-existence checks. Three categories of regressions pass undetected: (1) Someone changes Batch 7 from conditional to unconditional (removes the skip logic but keeps the "Skip this batch entirely" text elsewhere, or rephrases to "This batch is always generated"). The test checks `grep -q "Skip this batch entirely"` which matches Batch 6's identical phrasing on line 58, not Batch 7's. Since grep matches ANY occurrence in the file, the test passes even if Batch 7's conditional is deleted. (2) Someone adds Batch 9 but the file count ceiling stays at 27 -- the test only checks `grep -q "27"` which would still pass. (3) The test checks ordering (Batch 7 before Batch 8) but uses `head -1` on the grep output -- if Batch 7 heading text appears elsewhere earlier in the file (e.g., in a comment), the ordering check gives a false positive.
- **Impact:** The test suite provides false confidence. A contributor could make Batch 7 unconditional (generating Playwright files for CLI tools and API-only projects, bloating them with unused E2E infrastructure), and the test would not catch it. The file count ceiling of 27 could become inaccurate after adding Batch 9 (e.g., OpenAPI spec generation adding 2-3 files), leading scaffolder to either (a) generate too many files and violate its own constraint, creating reviewer confusion, or (b) self-limit by dropping files to stay under 27, losing needed content. Both scenarios are invisible to the test harness.
- **Likelihood:** High for the grep-aliasing issue (the "Skip this batch entirely" string appears identically in Batch 6 line 58 and Batch 7 line 67-68, so the grep will match Batch 6 even if Batch 7's skip logic is removed). Medium for file count drift (depends on future development pace). The fast-track skip of phases 1-5 means these structural weaknesses in the test were never caught by spec review or plan validation.
- **Mitigation:** (1) Replace broad `grep -q "Skip this batch entirely"` with a targeted grep that includes Batch 7 context: `grep -A2 "Batch 7" "$SCAFFOLDER" | grep -q "Skip this batch entirely"`. (2) Add a semantic check: count the number of `**Batch N**` headings and verify it matches expected count (currently 8). (3) For file count validation, extract the actual number from the constraint line and verify it arithmetically against batch file counts. (4) Add a negative test: verify that Batch 7's skip conditions reference both "NOT a web project" AND "Playwright is NOT in the project's dependencies" -- checking the AND condition, not just one leg.

## Overall Assessment

The changes are well-structured and follow established patterns (conditional batch with detection logic, unconditional batch with scorecard integration, Module Docs wiring). The scaffolder agent definition is internally consistent -- Batch 8 feeds Module Docs, the scorecard checks both new items, the constraint section has updated file counts.

However, three structural weaknesses emerge:

1. **Cross-stack blindness in Batch 7:** The Playwright detection is JS-ecosystem-only (`package.json` check), but Batch 6's web project detection includes non-JS stacks (Django, Rails, Flask). This creates a detection gap where web projects on non-JS stacks cannot get E2E scaffolding, conflicting with the scaffold v2 MUST-generate requirement.

2. **Documentation decay by design:** Batch 8 generates architecture docs at skeleton time, but the scaffold pipeline implements features afterward. No refresh step exists, and Module Docs consumers (code-analyst, architect) will read stale content in all subsequent pipeline runs.

3. **Test fragility:** The grep-based test strategy cannot distinguish which batch a matched string belongs to, creating false-positive risk. The fast-track skip of phases 1-5 means this test is the primary quality gate, yet it lacks the semantic depth to catch conditional-logic or file-count regressions.

The 0.75 score reflects that the core design is sound and the changes are backward-compatible (no contract breaks), but the edge cases above represent real failure paths that would likely surface in production use -- particularly the stale-docs scenario (guaranteed on every scaffold v2 run) and the non-JS Playwright gap (affects Django/Rails users).

## Recommendation

- **PASS** -- with the understanding that all three failure scenarios should be tracked as follow-up items. None represent blocking regressions (the plugin still works for its primary JS web project use case), but Scenario 2 (stale docs) is a design-level gap that will affect every scaffold v2 project over time, and Scenario 1 (non-JS Playwright) will confuse users of Django/Rails web stacks.
