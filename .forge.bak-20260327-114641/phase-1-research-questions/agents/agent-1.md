## Question 3: Step 4b TODO marker scanning logic

**Exact scanning logic (lines 267–272):**

```
Check the generated CLAUDE.md for TODO markers (`<!-- TODO:`):

1. Scan all `| Key | Value |` rows in `## Automation Config` for values containing `<!-- TODO:`.
2. Collect incomplete keys into a list: `incomplete_keys[]`.

If `incomplete_keys` is empty → skip to Step 4c (no TODOs to fill).
```

**Key observations:**

1. **Scope of scan:** Only `## Automation Config` table rows (`| Key | Value |` format) are scanned — not the entire CLAUDE.md.
2. **Marker format:** The exact HTML comment prefix `<!-- TODO:` is the detection token. The closing `-->` is implied but not explicitly defined in the scan rule.
3. **Data structure:** All incomplete keys are collected into `incomplete_keys[]` before prompting begins.
4. **Early exit:** If `incomplete_keys` is empty, the step jumps immediately to Step 4c.
5. **After collection (lines 295–298):**
   - Uses the Edit tool to replace `<!-- TODO: ... -->` markers with user-provided values in CLAUDE.md.
   - Re-validates by scanning for remaining TODOs.
   - Commits: `git add CLAUDE.md && git commit -m "chore: configure Automation Config"`.

**Per-key behavior (lines 282–293):**
- `Type`, `Instance`, `Project`: interactive prompts with format hints.
- `Bug query`, `State transitions`, `On start set`: auto-filled from tracker type defaults, shown to user for confirmation (press Enter to keep).
- `Remote`, `Base branch`, `Branch naming`: interactive with defaults shown.

**What is NOT handled:**
- The scan is purely textual — it does not parse the CLAUDE.md table structure semantically.
- There is no explicit handling of partially-formed TODO markers (e.g., `<!-- TODO` without colon).
- No distinction between optional and required keys during the scan — all `<!-- TODO:` markers are treated as requiring resolution.

## Question 4: MCP Pre-flight Check section and its relation to Step 0-MCP

**Full MCP Pre-flight Check section (lines 540–551):**

```
## MCP Pre-flight Check

MCP pre-flight check is only required when:
- `--issue` flag is used (Step 1 — reading issue description from tracker)
- Step 9 — creating cards in issue tracker (only when tracker is configured and user opts in)

For `--no-implement`, keep the same behavior as v3.x (MCP check before stack-selector).

Before any MCP operation, verify MCP tool availability:
- Read Type from Automation Config (Issue Tracker section)
- Check that at least one `mcp__*` tool matching the tracker type is accessible
- If not accessible → STOP with: "MCP server for {Type} is not available. Run `/ceos-agents:check-setup` for diagnostics or `/ceos-agents:init` to configure."
```

**Structural placement:** This section appears after Step 10 and before the `## Rules` section. It is a **policy section**, not an inline step — meaning the command must apply this check contextually at the two named trigger points, not as a discrete pipeline step.

**Relationship to "Step 0-MCP" (as proposed in the infrastructure design plan):**

The current file has **no Step 0-MCP**. The MCP Pre-flight Check section defines a policy:
- Triggered lazily at Steps 1 (`--issue`) and 9 (card creation).
- For `--no-implement`: triggered before L1 (stack-selector).
- The check is `mcp__*` tool availability — not connectivity or authentication.

**Proposed Step 0-MCP** (from the design plan context) would consolidate this into an upfront infrastructure validation step that runs before any work begins, rather than being scattered across Step 1 and Step 9 invocations. The current architecture defers MCP validation until it is needed, which means a scaffolding session can proceed through specification and skeleton generation before failing on MCP at Step 9.

**Gap identified:** The current `--no-implement` MCP note ("keep the same behavior as v3.x — MCP check before stack-selector") implies MCP is checked at L0 (before L1) in that flow, but there is no explicit L0 step defined. This is an implicit pre-condition with no corresponding step label.

## Summary of Key Findings

1. **Step 0** initializes state and branches immediately on `--no-implement`; mode selection is shown only for the full v2 flow.
2. **Steps L1–L6** constitute a complete self-contained mini-pipeline for `--no-implement` that bypasses all of Steps 4–10.
3. **Step 4b** uses pure text scanning (`<!-- TODO:`) on `## Automation Config` table rows; it has no semantic parser and no distinction between optional/required keys.
4. **Step 4c** is informational-only and has no explicit Full YOLO guard — it relies on Step 4b's outcome to determine whether to display anything.
5. **The MCP Pre-flight Check** is a policy section (not a pipeline step), applied lazily at Steps 1 and 9. There is no Step 0-MCP today; introducing one would require refactoring the policy from a trailing section into a first-class pipeline step near Step 0.
6. **Full YOLO** skips all human interaction steps (2, 4b, 4c implicitly, 6, 9) and escalates to Block on data-integrity failures (Steps 5, 7b).
