# Robustness / Devil's Advocate Review

## Failure Scenarios

| # | Scenario | Verdict | Detail |
|---|----------|---------|--------|
| 1 | Zero stories in an epic | HANDLED | Step 4e, line 536: "If epic has zero stories (no `### Story` headings found): skip story iteration -- epic-only issue is sufficient." The guard is explicit and correctly scoped: the epic-level issue is still created (step 1.b), only the inner story loop is skipped. The display message formula `Created {N}/{M} tracker issues ({S} stories, {F} story failures)` also works correctly -- S=0 and F=0, producing `Created 1/1 tracker issues (0 stories, 0 story failures).` which is accurate if slightly awkward in phrasing. No logic gap. |
| 2 | Resume after Step 4e crash (partial epic creation) | HANDLED | The idempotency guard (line 522) checks for `<!-- {TrackerType}: ... -->` back-reference comments per epic and per story (line 532). On crash mid-iteration: (a) already-created epics have back-references written to the file, so they are skipped on resume; (b) the crash interrupts *before* the `git commit` at step 2, so partially-written back-references for the current epic's stories may be in the working tree but not committed. On resume, the current epic's back-reference IS in the file (step 1.b writes it immediately after creation), so the epic-level guard fires, jumps to step 1.c, and re-processes stories -- stories with back-references are skipped, stories without are re-created. **One subtle gap:** if the crash happens after the epic is created (back-reference written to file) but before any stories are created, and the file is NOT saved to disk (OS buffer), the back-reference is lost. The epic would be re-created as a duplicate. However, this is an OS-level edge case, not a protocol-level one. The idempotency design is sound at the application level. |
| 3 | Mixed tracker results (5 stories succeed, 2 fail) | HANDLED | Step 4e accumulator (lines 540-549) explicitly handles partial success. Per-story failure: `WARN: Could not create story sub-issue for {story title} in {epic filename}: {error}`, continue to next story. The epic is considered succeeded if the epic-level issue was created (line 541). After iteration, any successful back-references are committed (`git add spec/ && git commit`). Display: `Created {N}/{M} tracker issues ({S} stories, {F} story failures).` However, **Step 8b has a subtle gap here**: line 742 says "For GitHub/Gitea (standalone story issues): also close each story issue individually. Read story IDs from back-reference comments within the epic file." If 2 stories failed creation, they have NO back-reference comments. Step 8b will simply not find them and will not attempt to close them -- which is correct (they don't exist). But the display at step 8b line 746 counts by *epic*, not by *story*. The epic is marked fully completed only if none of its *subtasks* (from architect decomposition) are blocked -- story creation failures at Step 4e do not flag the epic as "blocked" in the Step 7 sense. So an epic where 2 stories failed at 4e but all subtasks passed at Step 7 would be closed at 8b, leaving orphaned (never-created) tracker stories. This is arguably correct behavior (the code works, only the tracker bookkeeping is incomplete), but the user is not warned about this mismatch. **Verdict: PARTIAL** -- the accumulator works, but the 4e/8b interaction has an information gap where failed-to-create stories are silently forgotten at closure time. |
| 4 | GitHub/Gitea standalone issues -- closing at Step 8b | PARTIAL | Step 8b line 742 says: "For GitHub/Gitea (standalone story issues): also close each story issue individually. Read story IDs from back-reference comments within the epic file." The logic relies on back-reference comments (`<!-- {TrackerType}: {STORY-ISSUE-ID} -->`) to identify story issues. **How does Step 8b distinguish standalone vs native sub-issues?** It does NOT use a flag or marker. Instead, Step 8b branches on the tracker type: line 742 targets "GitHub/Gitea" and line 743 targets "YouTrack, Jira, Linear, Redmine". The tracker type is known from `tracker_type` in-memory variable. This is correct and deterministic. **However, there is no explicit instruction for how Step 8b obtains the tracker type.** The guard clause (lines 728-734) references `tracker_effective_status` and `tracker_write_available`, but the transition logic at line 741-743 silently assumes `tracker_type` is available. Since Step 8b runs long after Step 0-INFRA, this relies on the in-memory variable surviving across all intermediate steps -- which is stated as the contract at line 100 and 831. The logic is correct but **fragile on resume**: if the pipeline crashes after Step 7 and resumes at Step 8b, `tracker_type` must be restored from `state.json` (infrastructure.tracker_type). The resume logic at lines 132-147 does restore infrastructure state. So this works. **One gap:** the standalone issue closure uses `State transitions -> Done` syntax. For GitHub/Gitea, the Done mapping is typically `close` (as seen in example configs). But the instruction says "Transition the epic issue to Done using `State transitions -> Done` syntax" (line 741) and separately "also close each story issue individually" (line 742). The word "close" for stories is ambiguous -- does it mean use the same `State transitions -> Done` mapping, or literally close the issue? If the user's Done mapping is `closed` with a label (e.g., `Done -> closed label:done`), the "close each story issue individually" instruction may not apply the label. This is an underspecification. |
| 5 | Missing Done mapping in State transitions | HANDLED | Step 8b guard clause (lines 733-734): "`State transitions` value from Automation Config does not contain a `Done` mapping" triggers skip. Line 734: "If guard triggers for missing Done mapping: display `WARN: State transitions does not include a 'Done' mapping. Skipping issue closure.`" This is explicit and correct. The pipeline continues to Step 9 without attempting any transitions. Older CLAUDE.md files without Done mapping will get a clear warning and graceful skip. |
| 6 | Tracker API failure at Step 8b (partial Done transitions) | HANDLED | Step 8b line 744: "On failure: WARN (`Could not transition {issue_id} to Done: {error}`), continue to next." This is the same WARN-and-continue pattern as Step 4e's accumulator. Line 746 displays: `Transitioned {N}/{M} tracker issues to Done. {skipped} skipped (blocked subtasks).` **However, there is a counting discrepancy.** The display formula has three variables: N (succeeded transitions), M (total eligible epics), and {skipped} (blocked subtasks). But API failures are a fourth category -- epics that are eligible and not blocked, but whose transition failed. The formula N + skipped = M only holds if there are zero API failures. With API failures, N + skipped + failed < M is possible, and the display would show e.g., `Transitioned 3/6 tracker issues to Done. 1 skipped (blocked subtasks).` -- the missing 2 are not accounted for in the summary. The per-issue WARN messages explain them individually, but the summary line is misleading. **Verdict: PARTIAL** -- individual errors are handled, but the summary display formula does not account for API failures as a distinct category. |

## Additional Observations

**A. "Blocked features list" is not formally defined.** Step 7 block handler (line 675-689) calls rollback-agent and reports the block to stdout. The block handler updates `state.json` with `status: "blocked"` and a `block` object. But `state.json` only has a single `block` field (not a list). In a `continue` fail strategy, multiple subtasks can block. The state schema (line 121, `"block": null`) stores only ONE block at a time. Step 8b (line 739) references "the blocked features list (computed by Step 7 block handler)" -- but no such list is explicitly accumulated. The Step 9 final report (line 791) also references "Blocked features (if any):" with a list format. **This is an implicit data structure.** The LLM is expected to remember which subtasks blocked during execution. On resume, this information would be lost because state.json only stores the last block. For `fail-fast` strategy this is fine (only one block before stop). For `continue` strategy, multiple blocks can occur and the only record is stdout output, not state.json.

**B. Step 4e back-reference write timing.** Back-references are written to spec files immediately after each issue creation (lines 524, 535). But the `git commit` only happens after the full iteration completes (line 546). This means a crash mid-iteration leaves uncommitted changes in the working tree. On resume, if the user runs `git checkout .` or similar cleanup before re-running, all back-references are lost and all issues would be re-created as duplicates. The skill does not warn about this or protect against it.

**C. Step 8b story closure for GitHub/Gitea is underspecified.** Line 742 says "also close each story issue individually" but does not specify the mechanism. Should it use the same `State transitions -> Done` syntax? Should it use the GitHub/Gitea `close` API directly? If the Done mapping includes label changes (e.g., `Done -> closed label:done`), should stories also get those labels? This is left to LLM interpretation.

## Score: 0.75 / 1.0

## Summary

The core robustness mechanisms (idempotency guards, accumulator pattern, guard clauses, WARN-and-continue) are well-designed and consistently applied across Steps 4e and 8b. The zero-stories edge case and missing-Done-mapping scenarios are explicitly handled. The main weaknesses are: (1) the "blocked features list" referenced by Step 8b is never formally accumulated in state.json, creating a resume gap for `continue` fail strategy; (2) the Step 8b display message formula does not account for API failures as a distinct category separate from blocked-subtask skips; (3) story-level failures at Step 4e are not tracked through to Step 8b, meaning closure logic can silently ignore never-created stories; and (4) the Step 8b story closure mechanism for GitHub/Gitea standalone issues is underspecified regarding Done mapping application. None of these are critical -- the pipeline will not crash or produce incorrect code -- but they represent information loss and ambiguous behavior at the tracker integration boundary.
